# Defining the System Architecture

---

## 1. Overview

Choosing appropriate technologies is one of the most important parts of new system development. System architecture bridges the gap between:
- **Systems Analysis** (what the system should do)
- **Detailed Design** (how individual components are built)

This lecture covers two key design activities:
1. **Describe the Environment** — technical architecture and external interfaces
2. **Design the Application Components** — allocating functions to software components

---

## 2. Anatomy of a Modern System — Amazon.com as Example

Modern systems like Amazon.com involve many interconnected layers of hardware, software, and network components working together. Understanding each layer is essential for system architecture decisions.

---

## 3. Core Architectural Concepts

### 3.1 Servers and Clients

**Server:** A computer or group of computers that:
- Manages shared resources (file systems, databases, websites)
- Enables users and other computers to access those resources over a network

**Client:** Personal computing devices that access server resources
- Desktops, laptops, tablets, smartphones

```
Client devices          Server
┌──────────┐           ┌──────────────────┐
│ Desktop  │ ────────→ │ File Server      │
│ Laptop   │ ────────→ │ Database Server  │
│ Tablet   │ ────────→ │ Web Server       │
│ Phone    │ ────────→ │ Application Srv  │
└──────────┘           └──────────────────┘
```

---

### 3.2 Computer Networks

| Term | Definition |
|---|---|
| **Computer network** | A collection of hardware, software, and transmission media that enables computing devices to communicate and share resources |
| **Internet** | Networks of all sizes interconnected with one another globally |
| **LAN (Local Area Network)** | A small network typically spanning a single home, small office, or one floor of a building |
| **WWW (World Wide Web)** | An interconnected set of resources accessed via the Internet |
| **URL (Uniform Resource Locator)** | Identifier of a Web resource containing a protocol header, server name/address, and resource name |
| **Hyperlink** | The URL of one Web resource embedded within another Web resource |

---

### 3.3 Software Types

| Type | Description | Examples |
|---|---|---|
| **Application software** | Performs user- or business-specific tasks | CRM system, accounting app, email client |
| **App** | Application software installed on the storage device of a computer or phone | Mobile banking app, VS Code |
| **System software** | Works behind the scenes to support application software and control hardware | Operating systems, web server software |
| **Web-based application** | Application software deployed on the Web | Gmail, Salesforce, online banking |
| **Embedded software** | Apps or functions embedded within another app (browser or OS) | Browser toolbars, plug-ins, widgets |

#### Web-Based Application — Key Characteristics
- Uses a **Web browser** as the user interface
- Has a **URL** for application access
- Uses a **Web server** and server-side software components
- Uses **Web standards** for communication between browser and server

---

### 3.4 Protocols

**Protocol:** A set of languages and rules to ensure communication and data exchange between hardware and software.

#### Network Protocols
Enable accurate and secure message transmission among various computers and software components. Implemented within **all computers and network hardware devices**.

#### Web Protocol Suite

| Protocol | Full Name | Purpose |
|---|---|---|
| **URL** | Uniform Resource Locator | Defines a resource name, location, and encoding protocol |
| **HTML** | HyperText Markup Language | Defines structure and content of a Web page (text, tables, images, animation) |
| **XML** | Extensible Markup Language | Extends HTML; enables groups of users to define additional encoding within Web documents |
| **HTTP** | HyperText Transfer Protocol | Defines format and content for transfer of Web documents |
| **HTTPS** | HTTP Secure | Encrypted and secure HTTP transfers |

#### Application Protocols

| Protocol | Type | Purpose |
|---|---|---|
| **RPC** | Remote Procedure Call | Program-to-Program communication |
| **ODBC** | Open Database Connectivity | Program-to-Database communication |

---

### 3.5 Architecture Types

| Architecture | Definition |
|---|---|
| **Technology architecture** | The set of **computers, network hardware, and system software** in an organisation |
| **Application architecture** | The set of **software applications and their configuration** in an organisation |

```
Application Architecture  sits on top of  Technology Architecture
(software apps, configs)                  (computers, network, OS)
```

---

### 3.6 Services

#### SaaS — Software as a Service
A software delivery model similar to a utility, in which application software is accessed via the Internet **without locally installed programs**.

Key characteristics:
- Little or no software installed on users' devices
- Application service is accessed **remotely**
- User data is isolated and stored on **common servers**

**Examples:** Microsoft 365, Salesforce, Google Workspace, Dropbox

#### Web Service
A software function that is executed using Web standards:
- Accessed via a **URL**
- Inputs sent via the URL
- Executes **remotely**
- Data returned within a Web page

---

## 4. Distributed Architectures

### 4.1 Client/Server Architecture

Software design where **part of the application runs on a server** and **part runs on the client**.

> Note: In Client/Server Architecture, "server" refers to **software** (not just a physical device).

- **Server:** Manages system resources and provides access through a well-defined communication interface
- **Client:** Uses the communication interface to request resources and receive responses

```
Client                    Server
┌───────────┐  Request   ┌───────────────┐
│ UI Layer  │ ─────────→ │ Business      │
│ (browser/ │            │ Logic         │
│  app)     │ ←───────── │ + Data        │
└───────────┘  Response  └───────────────┘
```

### 4.2 Three-Layer Architecture

A more refined Client/Server architecture dividing the application into **three distinct layers**:

| Layer | Name | Responsibility |
|---|---|---|
| **Layer 1** | View Layer | The **user interface** — what users see and interact with |
| **Layer 2** | Business Logic Layer | **Program logic** that implements functional requirements — the "brain" |
| **Layer 3** | Data Access Layer | **Functions to access data** — reads/writes to the database |

```
┌─────────────────────────────────┐
│         VIEW LAYER              │  ← Browser, Mobile App, Desktop UI
│    (User Interface)             │
└─────────────────┬───────────────┘
                  │ calls
┌─────────────────▼───────────────┐
│     BUSINESS LOGIC LAYER        │  ← Application Server
│  (Functional Requirements)      │     Rules, calculations, workflows
└─────────────────┬───────────────┘
                  │ queries
┌─────────────────▼───────────────┐
│      DATA ACCESS LAYER          │  ← Database Server
│   (Data Storage & Retrieval)    │     SQL, ORM, stored procedures
└─────────────────────────────────┘
```

**Why three layers?**
- **Separation of concerns** — each layer has one job
- **Maintainability** — change the UI without touching business logic
- **Scalability** — scale each layer independently
- **Security** — data layer is isolated from direct user access

---

## 5. Architectural Diagrams

Three types of diagrams are used to document system architecture:

### 5.1 Location Diagram
**Purpose:** Identifies the **geographical placement** of hardware, software, and users.

Shows:
- Where servers are physically located (data centers, offices, cloud regions)
- Where users are located
- Which locations connect to which

**When to use:** Large enterprise systems spanning multiple cities, countries, or cloud providers.

```
Example:
┌─────────────┐         ┌─────────────┐
│  Sydney HQ  │ ──WAN── │  Melbourne  │
│  App Server │         │  Branch     │
│  DB Server  │         │  Office     │
└─────────────┘         └─────────────┘
        │
        │ Internet
        ▼
┌─────────────┐
│ Cloud (AWS) │
│ Backup      │
└─────────────┘
```

### 5.2 Network Diagram
**Purpose:** Shows **how the components of a network are interconnected**.

Shows:
- Network devices (routers, switches, firewalls)
- Connection types (LAN, WAN, Internet)
- IP addressing and topology
- Security boundaries (DMZ, firewalls)

**When to use:** Planning network infrastructure, security design, troubleshooting connectivity.

```
Internet
    │
[Firewall]
    │
[Load Balancer]
    │
┌───┴────────┐
[Web Server] [Web Server]   ← DMZ
    │
[App Server]
    │
[Database]                  ← Internal network
```

### 5.3 Deployment Diagram
**Purpose:** Shows **how application software is deployed** across the hardware and system software.

Shows:
- Which software components run on which physical/virtual machines
- Communication paths between deployed components
- Software dependencies

**When to use:** Implementation planning, DevOps setup, understanding what goes where.

```
«device» Web Server        «device» App Server
┌──────────────────┐       ┌──────────────────┐
│ «software»       │ HTTP  │ «software»        │
│ Apache/Nginx     │──────→│ Node.js / Java    │
│ Static Assets    │       │ Business Logic    │
└──────────────────┘       └────────┬─────────┘
                                    │ JDBC/ODBC
                           ┌────────▼─────────┐
                           │ «device» DB Srv  │
                           │ PostgreSQL/MySQL  │
                           └──────────────────┘
```

### Comparison of Three Diagram Types

| Diagram | Focus | Key Question Answered |
|---|---|---|
| **Location** | Geography | *Where are things physically located?* |
| **Network** | Connectivity | *How are things connected to each other?* |
| **Deployment** | Software placement | *What software runs on what hardware?* |

---

## 6. Describing the Environment (Design Activity 1)

**Two key elements:**
1. External systems and interfaces
2. Technology architecture

### Key Questions to Answer

| Question | Purpose |
|---|---|
| What are the key features of the existing or new environment? | Understand current state and constraints |
| What are the external systems or DBMSs? | Identify integration points |
| What devices will be required? | Define hardware requirements |
| What user-interface technology will be used? | Decide browser/mobile/desktop |

---

## 7. Designing Application Components (Design Activity 2)

### Application Component Boundaries

**An application component** = a well-defined unit of software that performs specific functions.

**Core question:** Which components perform which functions?

### Two Approaches to Defining Boundaries

| Approach | Description |
|---|---|
| **Top-down (Factoring)** | Start with the whole system → break down into smaller components |
| **Bottom-up (Grouping)** | Start with individual functions → group similar ones into components |

### Three Criteria for Measuring Similarity (Grouping Functions)

| Criterion | Question | Example |
|---|---|---|
| **Actors** | What functions do particular actors use? | All functions used by "Customer" → Customer Component |
| **Shared data** | What functions use the same data? | Functions that all access "Order" data → Order Management Component |
| **Events** | What functions occur in the same business event? | All functions triggered by "Customer places order" → Sales Component |

### RMO CSMS Application Components (Partial Example)

The RMO CSMS system is divided into components based on the subsystems identified during analysis:
- Customer Account Component
- Sales Component
- Order Fulfillment Component
- Marketing Component
- Reporting Component

Each component maps to use cases and actors identified during the analysis phase — demonstrating the **continuity from analysis to design**.

---

"What the system does"  bridges  "How the system is built"
         ↑                              ↑

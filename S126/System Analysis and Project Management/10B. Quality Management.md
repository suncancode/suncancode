# Software Quality Management

## 1. Why Software Quality Matters

### "Software Eats the World"
Marc Andreessen (Wall Street Journal) argued that software companies are taking over large swathes of the economy — more and more major businesses and industries are being run on software and delivered as online services.

**With great power comes great responsibility.** Low-quality software has real consequences:

| Consequence | Real Example |
|---|---|
| **Costs jobs** | Software failures lead to layoffs and business collapse |
| **Costs money** | Ariane 5 rocket explosion (1996): 10 years + $7 billion → destroyed in <60 seconds due to a 64-bit number being stuffed into a 16-bit space |
| **Costs lives** | Safety-critical systems failures in medical, aviation, and military domains |

### The Ariane 5 Case — A Classic Example
- **Date:** 4 June 1996, 12:33 GMT
- **What happened:** European Space Agency launched Ariane 5 on its maiden unmanned flight
- **Cause of failure:** A small computer program tried to convert a 64-bit floating point number into a 16-bit signed integer — causing an **overflow exception** that crashed the inertial reference system
- **Result:** Rocket self-destructed less than 1 minute into flight
- **Cost:** $7 billion development + the rocket and payload destroyed
- **Lesson:** A single software defect in a non-critical function (the backup inertial system was already switched off) brought down the entire mission

## 2. Defining Software Quality

Two foundational frameworks for defining software quality:

### 2.1 McCall's Quality Factor Tree
An early model that organizes software quality into factors grouped around three perspectives:
- **Product operation** — how well it runs (correctness, reliability, efficiency, integrity, usability)
- **Product revision** — how easily it can be changed (maintainability, testability, flexibility)
- **Product transition** — how easily it can move to new environments (portability, reusability, interoperability)

### 2.2 ISO 9126 — International Standard for Software Quality

ISO 9126 *"Software Engineering — Product Quality"* is the international standard for evaluating software quality. It defines two dimensions:

- **External qualities** — reflect the **user's view** of the software (what they experience)
- **Internal qualities** — reflect the **developer's view** of the software (how it is built)

> Internal quality influences external quality: well-structured code (internal) leads to reliable, maintainable software (external).

## 3. ISO 9126 — Six External Quality Characteristics

### 3.1 Functionality
*Does it satisfy user needs?*

| Sub-characteristic | Description |
|---|---|
| **Suitability** | Provides an appropriate set of functions for specified tasks and user objectives |
| **Accuracy** | Provides correct or agreed results |
| **Interoperability** | Ability to interact with other software components |
| **Security** | Protects information and data; controls access to the system |

### 3.2 Reliability
*Can the software maintain its level of performance?*

| Sub-characteristic | Description |
|---|---|
| **Maturity** | Frequency of failure due to faults — fewer failures = higher maturity |
| **Fault tolerance** | Maintains a specified level of performance in the case of faults or violations of specified interfaces |
| **Recoverability** | Re-establishes specified performance level and recovers data directly affected after a failure |

**Key Metrics for Reliability:**
- **Availability** = percentage of time the system is usable
- **Mean Time Between Failures (MTBF)** = total service time ÷ number of failures

### 3.3 Usability
*How easy is it to use?*

| Sub-characteristic | Description |
|---|---|
| **Understandability** | Easy to understand what the software does and how to use it |
| **Learnability** | Easy to learn how to use the software |
| **Operability** | Easy to operate and control |
| **Attractiveness** | Capability of the software to be attractive to users |

### 3.4 Efficiency
*What physical resources are used during execution?*

| Sub-characteristic | Description |
|---|---|
| **Time behaviour** | Response and processing times (e.g., response time < 2 seconds) |
| **Resource utilization** | Amount of resources used (e.g., CPU, memory usage) |

### 3.5 Maintainability
*How much effort is needed to make changes?*

| Sub-characteristic | Description |
|---|---|
| **Analysability** | Ease of identifying the cause of a failure or the parts to be modified |
| **Changeability** | How easily the software can be modified |
| **Stability** | Avoids unexpected effects from modifications |
| **Testability** | Enables modified software to be validated |

**Key Metrics for Maintainability:**
- **Mean effort to repair** — average hours needed to fix a fault (benchmark: 4 hours for high maintainability)
- **Mean elapsed time to repair** — average calendar time from fault reported to fault corrected (benchmark: 7 days for high maintainability)

### 3.6 Portability
*How easily can it be moved to a new environment?*

| Sub-characteristic | Description |
|---|---|
| **Adaptability** | Can be adapted for different specified environments without additional actions |
| **Installability** | Can be installed in a specified environment |
| **Co-existence** | Can co-exist with other independent software sharing common resources |
| **Replaceability** | Can be used in place of another specified software product for the same purpose |

## 4. Relationships Between Internal and External Quality

```
Internal Quality  →  External Quality  →  Quality in Use
(Code structure,      (Functionality,       (User satisfaction,
 architecture,         Reliability,          Productivity,
 modularity)           Usability,            Safety,
                       Efficiency,           Effectiveness)
                       Maintainability,
                       Portability)
```

- A well-structured codebase (internal) → fewer bugs in production (reliability)
- Clean modular design (internal) → easier to change (maintainability)
- Efficient algorithms (internal) → fast response times (efficiency)

## 5. Applying ISO 9126 in Practice

### Step 1: Prioritize Quality Characteristics for Your System
Different systems have different quality priorities:

| System Type | Most Critical Quality |
|---|---|
| Safety-critical (medical, aviation) | **Reliability** — must not fail |
| Real-time systems (trading, gaming) | **Efficiency** — must be fast |
| End-user consumer apps | **Usability** — must be easy to use |
| Enterprise systems | **Maintainability** — must be easy to change |

### Step 2: Select Relevant Measurements
For each prioritized quality, select specific measurable attributes:

**Reliability → Availability:**
```
Availability = (Total available time - Downtime) / Total available time × 100%
```

**Reliability → Mean Time Between Failures (MTBF):**
```
MTBF = Total service time / Number of failures
```

**Efficiency → Response time** (typical scale):

| Response Time | Rating |
|---|---|
| < 2 seconds | Exceeds requirement |
| 2–5 seconds | Target range |
| 6–10 seconds | Minimally acceptable |
| > 10 seconds | Unacceptable |

## 6. Worked Example — Availability and MTBF

### The Scenario
A tutor management system is normally available **8:00am – 6:00pm, Monday to Friday** (10 hours/day).

Over a **4-week period** (20 working days = 200 available hours):
- **1 whole day unavailable** due to disk drive problems → 10 hours lost
- **2 days unavailable until 10:00am** due to overnight batch processing → 2 hours lost each = 4 hours lost

**Total downtime = 10 + 4 = 14 hours**
**Total available time = 200 hours**

**Availability:**
```
Availability = (200 - 14) / 200 × 100%
             = 186 / 200 × 100%
             = 93%
```

**MTBF:**
```
Number of failures = 3 (1 full day + 2 partial days)
MTBF = 186 / 3 = 62 hours between failures
```

## 7. Worked Example — Maintainability Assessment

### The Scenario
Module AA247 fault log:

| Fault | Date Reported | Date Corrected | Effort (hours) | Elapsed Time |
|---|---|---|---|---|
| 1 | 1 Apr 2004 | 2 Apr 2004 | 5 | 1 day |
| 2 | 10 Apr 2004 | 5 May 2004 | 4 | 25 days |
| 3 | 12 Apr 2004 | 5 May 2004 | 3 | 23 days |
| 4 | 6 May 2004 | 7 May 2004 | 2 | 1 day |

**Benchmark for high maintainability:**
- Mean effort to repair: **4 hours**
- Mean elapsed time to repair: **7 days**

**Calculated values:**
```
Mean effort = (5 + 4 + 3 + 2) / 4 = 14 / 4 = 3.5 hours
Mean elapsed time = (1 + 25 + 23 + 1) / 4 = 50 / 4 = 12.5 days
```

**Assessment:**
- Mean effort (3.5 hrs) < benchmark (4 hrs) → ✅ **Good from developer perspective** — faults don't take long to fix
- Mean elapsed time (12.5 days) > benchmark (7 days) → ❌ **Poor from user perspective** — users wait too long for fixes

**Why the discrepancy?**
Faults 2 and 3 were reported close together (10 Apr and 12 Apr) but both fixed on 5 May — suggesting they were **batched together**, which is efficient for developers but creates long waiting times for users. Management should consider whether quicker individual responses are needed.

## 8. Product Quality vs Process Quality Management

### The Testing Limitation
Testing alone cannot guarantee quality because:
- Testing can only find **known unknowns** — you can only test for what you know to look for
- Software is subject to **unknown unknowns** — defects that no one anticipated
- Software constantly changes — functionalities, behaviour, and environment evolve

### Process Quality Management

> *"Quality of a product is heavily dependent on the quality of the process which produces it."*

Rather than measuring a completed product, **process quality management** embeds quality checks at **every stage of development**.

**Key principle:** Errors not removed at early stages become **exponentially more expensive** to correct at later stages.

```
Cost to fix a defect:
Requirements stage:  $1
Design stage:        $10
Coding stage:        $100
Testing stage:       $1,000
Production:          $10,000+
```

### Entry, Implementation, and Exit Requirements

For every activity in the development process, define three sets of requirements:

| Requirement Type | Definition | Example |
|---|---|---|
| **Entry requirements** | Must be in place **before** an activity can start | *"A comprehensive set of test data and expected results must be prepared and independently reviewed against system requirements before testing can commence"* |
| **Implementation requirements** | Define **how** the activity is to be conducted | *"Whenever an error is found and corrected, all test runs must be completed again, including those previously successfully passed"* |
| **Exit requirements** | An activity will not be completed until these are met | *"The testing phase is finished only when all tests have been run in succession with no outstanding errors"* |

## 9. Tools and Techniques for Quality Control

### The Seven Basic Tools of Quality

| Tool | Purpose |
|---|---|
| **Cause-and-effect diagram** (Fishbone/Ishikawa) | Identify root causes of a problem |
| **Control chart** | Monitor process stability over time — detect when a process goes out of control |
| **Checksheet** | Structured form for collecting and analyzing data |
| **Scatter diagram** | Show relationship (correlation) between two variables |
| **Histogram** | Show frequency distribution of a variable |
| **Pareto chart** | Identify the most significant causes (80/20 rule — 80% of problems come from 20% of causes) |
| **Flowchart / Run chart** | Visualize process steps or data over time |

### Pareto Chart — Special Focus
The Pareto chart combines a bar chart and a cumulative line graph:
- Bars are arranged in **descending order** of frequency
- The cumulative line shows the **running total percentage**
- Helps teams focus on the **vital few** causes that produce the **majority** of defects

**Example:** If 5 types of bugs are found in testing, a Pareto chart might show that 2 bug types account for 80% of all failures — fix those 2 first.

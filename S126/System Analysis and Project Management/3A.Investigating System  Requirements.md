## 1. Ridgeline Mountain Outfitters (RMO) — Background

**RMO** is an outdoor equipment retailer with an elaborate set of information systems supporting its operations and management.

### Why a New System?

Three pressures drove the need for change:
- **Customer expectations** — customers expect seamless, modern shopping experiences
- **Modern technological capabilities** — new technology enables better integration and features
- **Competitive pressures** — competitors are offering superior digital experiences

→ RMO proposed a new **Consolidated Sales and Marketing System (CSMS)** — a major project that grew directly out of RMO's **strategic planning process** (recall SWOT analysis and IT Planning Pyramid from Lecture 2B).

## 2. RMO's IT Architecture

Two layers of architecture define how an organisation uses technology:

| Architecture Type | Definition |
|------------------|-----------|
| **Technology Architecture** | The set of computing hardware, network hardware and topology, and system software employed by the organisation |
| **Application Architecture** | The information systems (software applications) that support the organisation |

### RMO's Existing Application Architecture (The Problem)

| System | Age / Technology | Issues |
|--------|-----------------|--------|
| **Supply Chain Management (SCM)** | 5 years old; Java/Oracle | Tradeshow system interfaces with it |
| **Phone/Mail Order System** | 12 years old; Visual Studio/MS SQL | Reached capacity; minimal integration |
| **Retail Store System (RSS)** | Older package solution | Minimal integration with other systems |
| **Customer Support System (CSS)** | Web-based; evolved over years | Minimal integration with other systems |

**The core problem:** Each system operates in isolation. There is **minimal integration** between them, leading to data silos, redundant processes, and a fragmented customer experience.

### Proposed Application Architecture — The Solution

Integrate SCM with a new **CSMS** that unifies all customer-facing operations:

```

│                    New CSMS                         │

│  ┌─────────────┐  ┌──────────────────────────────┐  │

│  └─────────────┘  └──────────────────────────────┘  │

│  ┌──────────────────┐  ┌────────────────────────┐   │
│  │ Customer Account │  │   Marketing Subsystem  │   │
│  │    Subsystem     │  │                        │   │
│  └──────────────────┘  └────────────────────────┘   │

                               │ integrates with

                        │     SCM     │
                        └─────────────┘
```

### The Four CSMS Subsystems

| Subsystem | Key Features |
|-----------|-------------|
| **Sales Subsystem** | Integrates online, phone, and retail store sales into one unified system |
| **Order Fulfillment Subsystem** | Track shipments, rate products and services |
| **Customer Account Subsystem** | Shopping history, account linkups, "Mountain Bucks" rewards program |
| **Marketing Subsystem** | Promotional packages, partner relationships, complete merchandise information and reporting |

## 3. Core Process 3 — Systems Analysis Activities

> **Core Process 3:** *Discover and understand the details of the problem or need.*

This is the heart of **systems analysis** — the goal is to understand in great detail **what** the information system needs to accomplish.

> Note: Analysis activities do not happen in isolation — they integrate with other SDLC activities across iterations.

### The Five Systems Analysis Activities

```
1. Gather Detailed Information
        │
        ▼
2. Define Requirements
        │
        ▼
3. Prioritise Requirements
        │
        ▼
4. Develop User-Interface Dialogs
        │
        ▼
5. Evaluate Requirements with Users
```

#### Activity 1: Gather Detailed Information
Collect information about the current situation and future needs using:
- **Interviews** with stakeholders
- **Questionnaires** distributed to users
- **Documents** — existing forms, reports, procedure manuals
- **Observing business processes** — watch how work actually gets done
- **Researching vendors** — explore available solutions and technologies
- **Comments and suggestions** from staff and customers

#### Activity 2: Define Requirements
Translate gathered information into formal requirements:
- Model **functional requirements** — what the system must DO
- Model **non-functional requirements** — how the system must PERFORM

#### Activity 3: Prioritise Requirements
Not all requirements are equal. Classify each as:
- **Essential** — must be in the system for it to work at all
- **Important** — significantly adds value; should be included if possible
- **Nice to have** — desirable but can be deferred to a later iteration

#### Activity 4: Develop User-Interface Dialogs
Model the **flow of interaction** between users and the system:
- Sketch screen layouts and navigation flows
- Validate that the system will be usable
- Identify interaction sequences for each user task

#### Activity 5: Evaluate Requirements with Users
- Involve users continuously throughout analysis
- Gather feedback on models, prototypes, and dialogs
- Adapt requirements as understanding improves

## 4. What Are Requirements?

> **System requirements** define what the system must do and how well it must do it.

Requirements are divided into two major categories:

### Functional Requirements
> Activities the system **must perform** — the functions users carry out.

- Business uses and processes the system supports
- Rules the system must enforce (business rules)
- Calculations, data storage, and retrieval the system must handle

**Examples for RMO CSMS:**
- The system must allow customers to place orders online
- The system must apply "Mountain Bucks" rewards automatically at checkout
- The system must track the status of all shipments

### Non-Functional Requirements
> **Other system characteristics** — constraints and performance goals that define HOW WELL the system performs its functions.

Non-functional requirements do not describe a specific behaviour but define the quality of the system overall.

## 5. FURPS — Requirements Classification

The **FURPS** acronym provides a systematic way to categorise all types of requirements:

| Letter | Category | Description |
|--------|----------|-------------|
| **F** | **Functional** | What the system must do — features and capabilities |
| **U** | **Usability** | How easy the system is to learn and use |
| **R** | **Reliability** | System availability, accuracy, failure recovery |
| **P** | **Performance** | Response time, throughput, capacity, scalability |
| **S** | **Security** | Access control, data protection, authentication |

### FURPS+ — Extended Categories

Beyond the core FURPS categories, additional constraints are captured under **FURPS+**:

| Category | Description | Example |
|----------|-------------|---------|
| **Design constraints** | Specific restrictions on hardware and software | Must run on existing Oracle database |
| **Implementation requirements** | Specific languages, tools, protocols | Must be developed in Java |
| **Interface requirements** | Links to other systems the new system must integrate with | Must interface with SCM system |
| **Physical requirements** | Physical facility and equipment constraints | Kiosks must be touch-screen compatible |
| **Supportability requirements** | Ability to update and maintain the system | Must support automatic software updates |

## 6. Stakeholders

> **Stakeholders** are persons who have an interest in the successful implementation of the system.

Identifying the right stakeholders is critical — missed stakeholders mean missed requirements.

### Classification 1: Internal vs. External

| Type | Definition | Examples |
|------|-----------|---------|
| **Internal** | Persons within the organisation | Employees, managers, IT staff |
| **External** | Persons outside the organisation | Customers, partner companies, regulators |

### Classification 2: Operational vs. Executive

| Type | Definition | Examples |
|------|-----------|---------|
| **Operational** | Persons who regularly interact directly with the system | Order clerks, warehouse staff |
| **Executive** | Persons who don't interact directly but use outputs or have financial interest | Senior managers, CFO, CEO |

### Stakeholder Classification Matrix

```
                    INTERNAL          EXTERNAL
               ┌───────────────┬─────────────────┐
  OPERATIONAL  │  Order clerks │   Customers     │
               │  Warehouse    │   Partner firms │
               │  Marketing    │   Shippers      │
               │  staff        │                 │
               ├───────────────┼─────────────────┤
  EXECUTIVE    │  Sales mgrs   │   (Regulators,  │
               │  Accounting   │    Investors,   │
               │  Senior execs │    etc.)        │
               └───────────────┴─────────────────┘
```

### RMO CSMS Stakeholders — Classified

| Stakeholder | Internal/External | Operational/Executive |
|-------------|------------------|----------------------|
| Phone/mail sales order clerks | Internal | Operational |
| Warehouse and shipping personnel | Internal | Operational |
| Marketing personnel (online catalogue) | Internal | Operational |
| Marketing, sales, accounting managers | Internal | Executive |
| Senior executives | Internal | Executive |
| Customers | External | Operational |
| Partner companies (e.g., shippers) | External | Operational |

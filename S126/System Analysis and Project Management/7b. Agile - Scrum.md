# Agile Project Management Using Scrum

---

## 1. What is Scrum?

Scrum is an **agile process** that allows teams to focus on delivering the highest business value in the shortest time. It allows rapid and repeated inspection of actual working software (every 2 weeks to 1 month).

**Key characteristics:**
- Self-organizing teams
- Product progresses in a series of **sprints** (short iterations)
- Requirements captured as **user stories** in a **product backlog**
- No specific engineering practices prescribed
- One of the most widely adopted agile frameworks

### Who uses Scrum?
Scrum is used by organizations such as Microsoft, Google, Yahoo, Nokia, BBC, Siemens, Philips, Salesforce, and many others — for projects ranging from commercial software and websites to life-critical medical systems, satellite-control software, and video game development.

## 2. The Scrum Framework

Scrum is built around three pillars: **Roles**, **Ceremonies (Events)**, and **Artifacts**.

```

│                 SCRUM FRAMEWORK                 │

│ Product Owner   │ Sprint        │ Product       │

│                 │   spective    │   Charts      │

```

## 3. Roles

### 3.1 Product Owner
- Defines the **features** of the product
- Decides on **release date and content**
- Responsible for the **profitability (ROI)** of the product
- **Prioritizes features** according to market value
- Adjusts features and priority every iteration as needed
- **Accepts or rejects** work results from the team

### 3.2 ScrumMaster
- Represents **management** to the project
- Responsible for **enacting Scrum values and practices**
- **Removes impediments** (blockers) for the team
- Ensures the team is **fully functional and productive**
- Enables **close cooperation** across all roles and functions

### 3.3 The Team
- Typically **5–9 people**
- **Cross-functional**: programmers, testers, UX designers, etc.
- **Self-organizing** — decides how to do the work
- Ideally no fixed titles
- Membership should change **only between sprints**

## 4. Artifacts

### 4.1 Product Backlog

The product backlog is the **master list** of everything that needs to be done. It contains:

| Item Type | Example |
|---|---|
| **Features** | "Allow a guest to make a reservation" |
| **Bugs** | "Fix login crash on mobile" |
| **Technical work** | "Upgrade all developers' workstations to Windows 10" |
| **Knowledge acquisition** | "Research JavaScript libraries and make a selection" |

- **Prioritized** by the Product Owner
- **Reprioritized** at the start of each sprint
- Items are written as **User Stories**

### 4.2 User Stories

User stories are **short, simple descriptions of a feature** from the perspective of the user who desires the capability.

**Template:**
> As a **[type of user]**, I want **[some goal]** so that **[some reason]**.

**Examples:**
- *As a site visitor, I can read current news on the home page.*
- *As a trainer, I can create a new course or event.*
- *As a hotel employee, I can run RevPAR reports.*

### Sample Product Backlog

| Backlog Item | Estimate (Story Points) |
|---|---|
| Allow a guest to make a reservation | 3 |
| As a guest, I want to cancel a reservation | 5 |
| As a guest, I want to change the dates of a reservation | 3 |
| As a hotel employee, I can run RevPAR reports | 8 |
| Improve exception handling | 8 |

### 4.3 Sprint Backlog

The sprint backlog is the **list of tasks** the team commits to completing during a sprint. It is derived from selected product backlog items.

**Example Sprint Backlog:**

| Task | Mon | Tue | Wed | Thu | Fri |
|---|---|---|---|---|---|
| Code the user interface | 8 | 4 | - | - | - |
| Code the middle tier | 16 | 12 | - | 11 | - |
| Test the middle tier | 8 | 16 | - | 8 | - |
| Write online help | 12 | 8 | - | - | - |
| Write the foo class | 8 | - | 4 | - | 8 |
| Add error logging | 8 | 10 | 16 | 8 | 8 |

- Any team member can **add, delete, or change** sprint backlog items
- Work is updated daily to reflect remaining effort

### 4.4 Burndown Chart

A **visual chart** showing remaining work (hours) over time during a sprint.

```
Hours remaining
  50 |*
  40 | *
  30 |   *
  20 |     *  *
  10 |          *
   0 |_______________
     Mon Tue Wed Thu Fri
```

- Ideal: line goes steadily down toward zero
- Shows whether the team is on track to complete the sprint

## 5. Sprints

A **sprint** is a fixed-length iteration during which a potentially shippable product increment is created.

- Typical duration: **2–4 weeks** (calendar month at most)
- Product is **designed, coded, and tested** within the sprint
- **No changes** allowed during a sprint — the scope is locked
- Plan sprint duration around how long the team can commit to keeping change out

### Sequential vs. Overlapping Development

| Traditional (Waterfall) | Scrum |
|---|---|
| All requirements → all design → all code → all test | A little of everything all the time |
| One thing at a time | Parallel, overlapping activities |
| Long feedback cycle | Short feedback cycle (every sprint) |

## 6. Estimation

### 6.1 Ideal Days
Estimate how long something would take if:
- It's **all you worked on**
- You had **no interruptions**
- **Everything you need** is available

> Like a soccer game: *ideal time = 90 minutes*, but *elapsed time = ~2 hours* (due to breaks, halftime, etc.)

### 6.2 Story Points
Story points represent the **relative effort** of completing a user story. Effort is influenced by:
- How **hard** the story is
- How **much** there is to do

**Relative comparison:**
- Login screen = **2 points**
- Search feature = **8 points**

Story points are **team-specific** — what matters is internal consistency, not absolute values.

### 6.3 Fibonacci Series
Story point values commonly follow the Fibonacci sequence: **1, 2, 3, 5, 8, 13, 21...**

This avoids meaningless precision — the gap between numbers increases, reflecting the uncertainty in larger estimates.

### 6.4 Triangulation
Confirm estimates by **comparing a story to multiple other stories** — not just one reference point.

> "This story is similar to Story A (3 pts) and smaller than Story B (8 pts), so it's probably 5 pts."

### 6.5 Planning Poker

An **iterative, collaborative estimation technique**.

**Steps:**
1. Each estimator receives a deck of cards with valid estimate values
2. Product Owner reads a user story and it is discussed briefly
3. Each estimator **privately selects** a card representing their estimate
4. All cards are **revealed simultaneously**
5. **Discuss differences**, especially outliers
6. **Re-estimate** until estimates converge

**Why Planning Poker works:**
- Those who **do the work** estimate the work
- Estimators must **justify** their estimates
- Group discussion **improves accuracy**
- Emphasizes **relative** rather than absolute estimating
- Everyone's opinion is heard
- Quick and engaging

## 7. Ceremonies (Events)

### 7.1 Sprint Planning Meeting

**Purpose:** The team decides on the **goal for the sprint** and selects items from the product backlog they commit to completing.

**Participants:** Product Owner, ScrumMaster, Team

**Outputs:**
- Sprint Goal — a short statement of what the work will focus on
- Sprint Backlog — tasks identified and estimated (1–16 hours each)

**Activities:**
- Analyze and evaluate product backlog
- Select sprint goal based on business conditions and priorities
- Decide how to achieve the sprint goal (high-level design)
- Create the sprint backlog from selected user stories
- Estimate sprint backlog items in hours

**Example Sprint Goals:**

| Domain | Sprint Goal |
|---|---|
| Financial services | Support more technical indicators than Company ABC with real-time, streaming data |
| Life sciences | Support features necessary for population genetics studies |
| Database application | Make the application run on SQL Server in addition to Oracle |

### 7.2 Daily Scrum Meeting

**Purpose:** Daily synchronization and planning for the next 24 hours.

**Parameters:**
- **Daily**, **15 minutes**, **stand-up** format
- Not for problem solving
- Only team members, ScrumMaster, and Product Owner can talk
- Helps avoid other unnecessary meetings

**Three questions every team member answers:**

| # | Question |
|---|---|
| 1 | What did you do **yesterday**? |
| 2 | What will you do **today**? |
| 3 | Is anything **in your way** (blocking you)? |

> These are not status reports to the ScrumMaster — they are **commitments made in front of peers**.

### 7.3 Sprint Review Meeting

**Purpose:** Team presents what it accomplished during the sprint.

**Format:**
- Demo of new features or underlying architecture
- **Informal** — no slides required
- Maximum **2-hour prep time** rule
- **Whole team participates**

### 7.4 Sprint Retrospective Meeting

**Purpose:** Reflect on the process and improve for next sprint.

**Parameters:**
- **15–30 minutes**
- Done **after every sprint**
- Whole team participates (ScrumMaster, Product Owner, Team, possibly customers)

**Start / Stop / Continue format:**

| Category | Question |
|---|---|
| **Start** | What should we start doing? |
| **Stop** | What should we stop doing? |
| **Continue** | What is working well and should continue? |

## 8. The Scrum Flow — Putting It All Together

```
Product Backlog
(prioritized list of all features)
         ↓
   Sprint Planning
   (select items → Sprint Backlog)
         ↓
┌─────────────────────────────┐
│         SPRINT              │
│       (2–4 weeks)           │
│                             │
│   Daily Scrum (15 min/day)  │
│   Design → Code → Test      │
└─────────────────────────────┘
         ↓
  Potentially Shippable
  Product Increment
         ↓
  Sprint Review (demo)
         ↓
  Sprint Retrospective (improve)
         ↓
  Next Sprint Planning...
```

**Q1. What kind of software development projects can be executed by Scrum Project Management Framework?**

- Choice 1: Complete software packages
- Choice 2: Customer projects
- Choice 3: Sub-systems, components or parts of bigger systems

> **Answer: Choice 4** — Scrum can be applied to ALL kinds of software development projects.

---

**Q2. What does NOT belong to cornerstones of the agile manifesto?**

- Choice 1: Individuals and interactions over processes and tools
- Choice 2: Working software over comprehensive documentation
- Choice 3: Processes over people
- Choice 4: Customer collaboration over contract negotiation
- Choice 5: Responding to change over following a plan

> **Answer: Choice 3** — "Processes over people" directly contradicts the Agile Manifesto, which values individuals and interactions.

---

**Q3. What is defined by the Scrum Framework?**

- A) Rules & Roles
- B) Document guidelines
- C) Artifacts and events
- Choice 2: B
- Choice 3: C
- Choice 4: A, B, C
- Choice 5: A, C

> **Answer: Choice 5 (A, C)** — Scrum defines Rules & Roles and Artifacts & Events. Document guidelines are NOT part of the Scrum Framework.

---

**Q4. Where are the customer requirements stored?**

- Choice 1: In the Product Backlog
- Choice 2: In the Sprint Backlog
- Choice 3: In a database
- Choice 4: In a Scrum Product Requirement Specification
- Choice 5: Nowhere. The Scrum Product Owner knows them

> **Answer: Choice 1** — Customer requirements are stored in the **Product Backlog**.

---

> **Answer: Choice 4 (A, B, C, E)** — Scrum defines Sprint Planning, Sprint Retrospective, Sprint Review, and Daily Scrum. "Mid-Sprint Status Review" is NOT a Scrum event.

- Choice 1: Scrum Master

**Q8. What is important in all Scrum projects?**

## 10. Summary

```
SCRUM FRAMEWORK
│
├── ROLES
│     ├── Product Owner  → defines & prioritizes features, accepts/rejects work
│     ├── ScrumMaster    → removes blockers, enforces Scrum values
│     └── Team           → 5–9 people, cross-functional, self-organizing
│
├── ARTIFACTS
│     ├── Product Backlog  → master list (features, bugs, tech work) as user stories
│     ├── Sprint Backlog   → tasks committed for the current sprint
│     └── Burndown Chart   → visual of remaining work over time
│
├── CEREMONIES
│     ├── Sprint Planning     → select sprint goal + build sprint backlog
│     ├── Daily Scrum         → 15-min standup: yesterday / today / blockers
│     ├── Sprint Review       → demo of completed work
│     └── Sprint Retrospective → Start / Stop / Continue improvement
│
├── SPRINTS
│     → 2–4 week fixed iterations
│     → No changes during a sprint
│     → Output: potentially shippable product increment
│
└── ESTIMATION
      ├── Ideal Days      → effort without interruption
      ├── Story Points    → relative effort (Fibonacci: 1,2,3,5,8,13...)
      ├── Triangulation   → compare to multiple reference stories
      └── Planning Poker  → collaborative, simultaneous card reveal
```

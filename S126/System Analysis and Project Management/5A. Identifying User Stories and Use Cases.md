# Functional Requirements — User Stories, Use Cases & Event Decomposition

---

## 1. Two Ways to Define Functional Requirements

The two main approaches are **User Stories** and **Use Cases**. They share the same goal but differ in approach and detail level.

### Comparison: User Stories vs Use Cases

| Aspect | User Stories | Use Cases |
|--------|-------------|-----------|
| **Definition** | One-sentence description of a task a user does to achieve a goal | An activity the system performs, usually in response to a user request |
| **Format** | "As a [user], I want to [action] so that [goal]" | Verb-Noun phrase (e.g., "Look Up Product") |
| **Detail level** | Minimal — details filled in later during design/coding | More complete — includes detailed steps for user and system |
| **Formality** | Less formal | More formal |
| **Best suited for** | **Agile** development methodologies | Traditional / iterative methodologies |
| **Focus** | User collaboration, simplicity, value added | Comprehensive system behaviour documentation |

> 💡 **Key insight:** Both techniques focus on **user goals** and show functions at an appropriate level of detail. The difference is *when* and *how much* detail is captured.

## 2. User Stories

### Definition
> A **User Story** is a one-sentence description of a work-related task done by a user to achieve some goal or result.

### Template

```
"As a [type of user],
 I want to [perform some task/action]
 so that [I can achieve some goal/benefit]."
```

### Examples

| User Story | User | Action | Goal |z
|-----------|------|--------|------|
| "As a **teller**, I want to **make a deposit** so that I can **quickly serve more customers**." | Bank teller | Make a deposit | Serve customers faster |
| "As a **bank customer**, I want to **withdraw cash** so that I feel **confident the amount is correct**." | Bank customer | Withdraw cash | Confidence in accuracy |

### Why User Stories Work Well in Agile

- **Simple** to write and understand
- Focus on **value added** to the user
- Encourage **user collaboration** throughout development
- Less formal than UML diagrams — details emerge during sprints

### Acceptance Criteria

Every User Story needs **Acceptance Criteria** — a list of features that must be present when the story is implemented.

**Purpose:**
- Used for **testing** once user stories are implemented
- Focused on **functionality** (functional requirements only)
- Helps verify whether users are writing stories at the **right level of detail**

**Example:**
```
User Story:
"As a customer, I want to add items to a cart so that I can purchase multiple items at once."

Acceptance Criteria:
- Customer can add any available product to cart
- Cart displays item name, quantity, and price
- Cart total updates automatically when items are added or removed
- Customer cannot add out-of-stock items
```

## 3. Use Cases

### Definition
> A **Use Case** is an activity that the system performs, usually in response to a request by a user.

### Format
Use cases are specified as **Verb-Noun phrases**:
- ✅ "Look Up Supplier"
- ✅ "Enter/Update Product Information"
- ✅ "Process Customer Order"
- ❌ "Customer" (no verb — not a use case)
- ❌ "Typing customer name" (too low-level)

### Two Techniques for Identifying Use Cases

```
Use Case Identification
        │
        ├── User Goal Technique
        │   (simple, user-focused, interview-based)
        │
        └── Event Decomposition Technique
            (comprehensive, event-driven, more complete)
```

---

## 4. User Goal Technique

### Definition
> A technique to identify use cases by determining what **specific goals or objectives** must be completed by the system for each type of user.

### Why Use It?
- Simple and effective
- Naturally aligns with stakeholder thinking
- Produces use cases that map directly to user needs

### 8-Step Process

| Step | Action |
|------|--------|
| **1** | Identify all potential users of the new system |
| **2** | Classify users by **functional role** (e.g., shipping, marketing, sales) |
| **3** | Further classify users by **organisational level** (operational, management, executive) |
| **4** | Interview each type of user to find specific goals they have when using the system (include both current goals and innovative new functions) |
| **5** | Create a preliminary list of use cases, organised by type of user |
| **6** | Look for **duplicates** with similar names; resolve inconsistencies |
| **7** | Identify where **different user types need the same use cases** |
| **8** | **Review** the completed list with each type of user and with stakeholders |

### Example: RMO CSMS — User Goal Technique

| User Type | Role | Level | Sample Use Cases |
|-----------|------|-------|-----------------|
| Order clerk | Sales | Operational | Look Up Product, Enter New Order, Check Order Status |
| Warehouse staff | Fulfillment | Operational | Update Shipment Status, Process Return |
| Marketing manager | Marketing | Management | Generate Sales Report, Create Promotion |
| Senior executive | All | Executive | View Sales Dashboard, Approve Budget |
| Customer | External | Operational | Browse Catalogue, Place Order, Track Shipment |

## 5. Event Decomposition Technique

### Definition
> A more comprehensive technique that identifies use cases by finding **events** that occur in the system environment and determining what the system must do in response to each event.

### The Right Level of Detail — Critical Rule

Use cases must be at the right granularity — not too small, not too large:

| Example | Correct? | Why |
|---------|----------|-----|
| "Typing a customer name on a form" | ❌ Too low-level | This is a single keystroke, not a business process |
| "Add New Customer Account" | ✅ Correct | One complete business process, one use case |
| "All activities with customers (add, update, contact late payers)" | ❌ Too high-level | Combines multiple distinct processes |

### Elementary Business Process (EBP)

The correct level of detail for a use case is defined by the **EBP**:

> **Elementary Business Process (EBP)** = a fundamental task in a business that is:
> - Performed by **one person**
> - In **one place**
> - In response to a **single business event**
> - Adds **measurable value**
> - Leaves the system and its data in a **stable, consistent state**

## 6. Events and Event Types

### What is an Event?

> An **event** is something that occurs at a specific time and place, can be described, and **should be remembered by the system**.

**Key question when identifying system requirements:**
> *"What business events occur that will require the system to respond?"*

### Three Types of Events

```
Events
├── External Events    (triggered by someone/something outside)
├── Temporal Events    (triggered by time)
└── State Events       (triggered by internal system state change)
```

---

### Type 1: External Events

> An event that occurs **outside the system**, initiated by an **external agent or actor** (a person or organisational unit that supplies or receives data from the system).

**Four categories of external events:**

| Category | Description | Example |
|----------|-------------|---------|
| **External agent wants a transaction** | Actor wants to do something that changes data | Customer orders a product |
| **External agent wants information** | Actor wants to retrieve/view data | Customer wants to know product details |
| **External data changes** | Data about an actor has changed and needs updating | Customer has a new address |
| **Management wants information** | Manager needs a report or status | Sales manager wants to check order status |

---

### Type 2: Temporal Events

> An event that occurs as a result of **reaching a point in time**.

**Key difference from external events:** The system automatically produces the required output **without any actor initiating it**.

| Example | Trigger |
|---------|---------|
| Monthly management reports | End of each month |
| Email reminders for outstanding payments | X days after invoice due date |
| Year-end financial summary | January 1st |

---

### Type 3: State Events

> An event that occurs when something **inside the system** reaches a certain state that triggers a process.

**Key characteristic:** The trigger point in time cannot be defined in advance — it depends on what happens in the system.

| Example | State Change |
|---------|-------------|
| Inventory reorder triggered | Stock drops below reorder threshold |
| Account flagged for review | Customer's credit score drops below minimum |
| Automatic escalation | Support ticket open for more than 48 hours |

> 💡 **State vs Temporal Events:**
> - **Temporal** = triggered at a *known, scheduled time* (e.g., end of month)
> - **State** = triggered when a *condition is met* (e.g., inventory < 10 units) — the exact time is unknown in advance

---

## 7. Events vs Prior Conditions and Responses

A common mistake is confusing the **actual event** with what happened before it (prior condition) or what happens after (response).

**How to find the true event:**

```
Sequence of transactions:
Customer places order (EVENT 1 → Use Case: Process Customer Order)
        │
        ▼
Order is processed and stock is reduced
        │
        ▼
Stock drops below threshold (EVENT 2 → Use Case: Trigger Reorder)
        │
        ▼
Supplier is contacted
        │
        ▼
End of month arrives (EVENT 3 → Use Case: Generate Monthly Report)
```

Each distinct event maps to a **separate use case**.

---

## 8. Event Decomposition: Three-Step Process

### Step 1: External Events
- Consider all external events in the system environment
- For each: what does the system need to DO in response?
- Name each use case as a Verb-Noun phrase

### Step 2: Temporal Events
- Consider all time-based triggers
- For each: what does the system automatically produce at that time?
- Name each use case

### Step 3: State Events
- Consider all internal state changes that should trigger a system response
- For each: define the state change and name the use case

## 11. Key Concepts Summary

| Concept | One-Line Definition |
|---------|-------------------|
| **User Story** | "As a [user], I want to [action] so that [goal]" — one sentence, Agile-friendly |
| **Acceptance Criteria** | Testable conditions that define when a user story is "done" |
| **Use Case** | A system activity in response to a user request — Verb-Noun format |
| **User Goal Technique** | Find use cases by interviewing users about their goals — 8-step process |
| **Event Decomposition** | Find use cases by identifying all events the system must respond to |
| **EBP** | Elementary Business Process — one person, one place, one event, adds value, leaves system stable |
| **External Event** | Triggered by an actor outside the system |
| **Temporal Event** | Triggered by reaching a point in time — system responds automatically |
| **State Event** | Triggered when an internal system condition is met — time is unpredictable |

## 12. Quick Quiz Prep

**Q: What is the format of a User Story?**
→ "As a [type of user], I want to [action] so that [goal/benefit]."

**Q: What makes Event Decomposition more comprehensive than User Goal Technique?**
→ It finds use cases for ALL types of events (external, temporal, state) — not just the ones users consciously think of. Temporal and state events are easily missed in interviews.

**Q: What is an EBP and why does it matter?**
→ EBP defines the correct granularity for a use case — one person, one place, one business event. It prevents use cases from being too small (a keypress) or too large (an entire business domain).

**Q: What is the difference between a temporal and a state event?**
→ Temporal = triggered at a known scheduled time (end of month). State = triggered when an internal condition is met — the exact time cannot be predicted in advance.

**Q: Use Cases are specified as what type of phrase?**
→ Verb-Noun (e.g., "Process Customer Order", "Generate Sales Report")

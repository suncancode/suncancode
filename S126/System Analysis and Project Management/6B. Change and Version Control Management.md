# Change and Version Control Management

---

## 1. Configuration Management (CM)

Software systems are **constantly changing** during development and use — new features, bug fixes, environmental changes. Without proper management, it becomes impossible to track what changed, who changed it, and which version is currently running.

**Configuration Management (CM)** is concerned with the policies, processes, and tools for managing changing software systems.

### Why do we need CM?
- It is easy to lose track of what changes and component versions have been incorporated into each system version.
- CM is essential for team projects to control changes made by different developers.

---

## 2. Four CM Activities

### 2.1 Change Management
Keeping track of requests for changes to the software from customers and developers, working out the costs and impact of changes, and deciding which changes should be implemented.

### 2.2 Version Management
Keeping track of the multiple versions of system components and ensuring that changes made by different developers do not interfere with each other.

### 2.3 System Building
The process of assembling program components, data, and libraries, then compiling these to create an executable system.

### 2.4 Release Management
Preparing software for external release and keeping track of the system versions that have been released for customer use.

---

## 3. Change Management — Detail

Organizational needs and requirements change during the lifetime of a system; bugs must be repaired and systems must adapt to changes in their environment.

**Goal**: Ensure that system evolution is a managed process and that priority is given to the most urgent and cost-effective changes.

The change management process is concerned with:
- **Analyzing** the costs and benefits of proposed changes
- **Approving** those changes that are worthwhile
- **Tracking** which components in the system have been changed

---

## 4. Version Management

**Version Management (VM)** is the process of keeping track of different versions of software components (or configuration items) and the systems in which these components are used.

It also involves ensuring that changes made by different developers to these versions do not interfere with each other.

Version management can be thought of as the process of managing **codelines** and **baselines**.

---

## 5. Codelines and Baselines

### Codeline
A **sequence of versions** of source code, where later versions are derived from earlier versions. Codelines apply to individual components (e.g., a class or a file).

```
Codeline (A):   A  →  A1.1  →  A1.2  →  A1.3
Codeline (B):   B  →  B1.1  →  B1.2  →  B1.3
Codeline (C):   C  →  C1.1  →  C1.2  →  C1.3

Libraries and external components:   L1 | L2 | Ex1 | Ex2
```

### Baseline
A **definition of a specific system** — specifying which component versions are included in the system, plus libraries used, configuration files, etc. Think of it as a "snapshot" of the entire system at a point in time.

```
Baseline V1:   A    + B1.2 + C1.1 + L1 + L2 + Ex1
Baseline V2:   A1.3 + B1.2 + C1.2 + L1 + L2 + Ex2
```

> 💡 **Codeline** = history of one component. **Baseline** = snapshot of the whole system.

## 6. Version Control (VC) Systems

Version control systems **identify, store, and control access** to different versions of components.

There are two types of modern version control systems:

---

### 6.1 Centralized Version Control
*(Examples: Subversion / SVN, CVS)*

A **single master repository** maintains all versions of software components. All developers work against this one central repository.

**Workflow:**
1. Developer **checks out** (downloads) a component from the repository into their private workspace.
2. Developer works on the copy locally.
3. Developer **checks in** (uploads) the modified component back to the repository.
4. If another developer has already checked out the same component, the system **warns** them.

```
         [Master Repository]
              ↑         ↑
         check_in    check_in
              ↓         ↓
    check_out ↑    check_out ↑
   ┌──────────────┐   ┌──────────────┐
   │ Alice's      │   │ Bob's        │
   │ Workspace    │   │ Workspace    │
   │ A1.0  B1.0   │   │ X1.0  Y1.0  │
   │ A1.1  B1.1   │   │ X1.1  Y1.1  │
   └──────────────┘   └──────────────┘
```

---

### 6.2 Distributed Version Control
*(Example: Git)*

**Multiple full copies** of the repository exist simultaneously — each developer has a complete clone of the entire project on their own machine.

**Workflow:**
1. Developer **clones** the master repository — downloads a complete copy to their local machine.
2. Developer works on files and **commits** changes to their local (private) repository.
3. Developer **pushes** committed changes to the master (shared) repository.

```
         [Master Repository — Server]
              ↓ clone         ↓ clone
   ┌──────────────────┐   ┌──────────────────┐
   │ Alice's          │   │ Bob's            │
   │ Local Repository │   │ Local Repository │
   │ A1.0  B1.0  C1.0 │   │ A1.0  B1.0  C1.0 │
   │ A1.1  B1.1  C1.1 │   │ C1.1  X1.1  Y1.1 │
   └──────────────────┘   └──────────────────┘
         ↑ push                  ↑ push
         └──────── Master ───────┘
```

### Benefits of Distributed VC

| Benefit | Description |
|---|---|
| **Backup** | If the server repository is corrupted, work can continue and be restored from local copies |
| **Offline working** | Developers can commit changes without a network connection |
| **Independent testing** | Developers can compile and test the entire system on their local machines |
| **Default workflow** | Project support is naturally built into the distributed model |

---

### 6.3 Centralized vs. Distributed — Comparison

| Feature | Centralized (SVN) | Distributed (Git) |
|---|---|---|
| Repository location | Single central server | Every developer has a full copy |
| Working offline | ❌ Not supported | ✅ Fully supported |
| Backup | Single point of failure | Naturally backed up on every machine |
| Operation | Check-out / Check-in | Clone / Commit / Push |
| Speed | Slower (requires network) | Faster (local operations) |
| Examples | Subversion, CVS | Git |

## 7. Branching and Merging

Rather than a linear sequence of versions, there may be **several independent sequences** (branches) — this is normal when different developers work on different features simultaneously.

### Branching
Creating an **independent copy** of a codeline to develop separately without affecting the main line.

### Merging
**Combining** two branches back into one version that includes all changes made in both.

- If the changes involve **different parts** of the code → merge can happen **automatically**
- If both branches changed **the same lines** → a **conflict** occurs and must be resolved manually

```
Codeline 1:    V1.0 ──→ V1.1 ──→ V1.2
                                    │
                              <branch>
                                    ↓
Codeline 2:          V2.0 ──→ V2.1 ──────────────────────────→ V2.4
                                    │                    ↑
                              <branch>              <merge>
                                    ↓                    │
Codeline 2.1:               V2.1.1 ──→ V2.1.2 ──────────┘
                                    │
                                  V2.2 ──→ V2.3 ──────────────┘
```

### Key Concepts

| Term | Meaning |
|---|---|
| **Branch** | An independent sequence of development split from the main codeline |
| **Merge** | Combining changes from two branches into one unified version |
| **Conflict** | When two branches modified the same lines — must be resolved manually |
| **Main / Mainline** | The primary branch that represents the official state of the system |

## 8. Key Terminology Summary

| Term | Definition |
|---|---|
| **Configuration Management (CM)** | Policies, processes, and tools for managing changing software systems |
| **Change Management** | Tracking, analyzing, and approving change requests |
| **Version Management** | Tracking multiple versions of components and preventing interference |
| **System Building** | Assembling and compiling components into an executable system |
| **Release Management** | Preparing and tracking software versions released to customers |
| **Codeline** | Sequence of versions of a single source code component |
| **Baseline** | A defined snapshot of a complete system (specific versions of all components) |
| **Check-out** | Downloading a component from the repository to a private workspace |
| **Check-in** | Uploading modified components back to the repository |
| **Clone** | Downloading a complete copy of the repository (distributed VC) |
| **Commit** | Saving changes to the local repository |
| **Push** | Uploading committed changes from local to the master repository |
| **Branch** | An independent line of development derived from an existing codeline |
| **Merge** | Combining two branches into one unified version |
| **Conflict** | When two branches have incompatible changes to the same code |

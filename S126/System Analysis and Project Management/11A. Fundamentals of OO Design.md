# Fundamentals of Object-Oriented Design

---

## 1. What is Object-Oriented Design?

**Object-Oriented Design (OOD)** is the process by which a set of detailed OO design models are built to be used for coding.

### Key Characteristics
- **Extends** requirements models from systems analysis into design models
- **Use case driven** — design is carried out use case by use case
- Design models are created **in parallel** with actual coding/implementation in iterative SDLC
- Designs classes in **three layers**: user interface, problem domain, and database access

### Strength of OOD
The biggest strength is **continuity** — the domain model class diagram (DMCD) from analysis becomes the primary source for the Design Class Diagram (DCD). Nothing is thrown away; everything is extended.

```
Analysis Phase                  Design Phase
──────────────                  ────────────
Domain Model (DMCD)   ──────→  Design Class Diagram (DCD)
Use Cases             ──────→  Sequence / Communication Diagrams
System Sequence Diag. ──────→  Detailed Method Signatures
Actors                ──────→  Boundary + Controller Classes
```

## 2. Core OO Concepts

### 2.1 Objects and Classes

| Concept | Definition | Analogy |
|---|---|---|
| **Class** | A template/blueprint defining the structure of data fields and program logic | A cookie cutter |
| **Object** | A specific instance of a class created in memory | A cookie made from the cutter |
| **Instantiation** | The creation of an object in memory based on the class template | Baking a cookie |

```
Class: Customer
──────────────────
- customerID: String
- name: String
- email: String
+ placeOrder(): void
+ getBalance(): float

Object (instance):
customerID = "C001"
name = "John Smith"
email = "john@email.com"
```

### 2.2 Methods

**Method** = a fragment of program logic (i.e., a function) within a class.

Methods are:
- Called/invoked through **messages** between objects
- Also invoked **when a class is instantiated** (constructor methods)

### 2.3 Encapsulation

Each object has **data and program logic encapsulated within itself** — the internal workings are hidden from outside objects. Other objects interact through defined interfaces (methods).

## 3. Three Tools for Documenting OO Design

The flow of execution of a use case can be documented using three tools. The choice depends on complexity:

| Complexity | Tool | When to Use |
|---|---|---|
| **Simple** use case | CRC Cards | Quick brainstorming; manual process |
| **Medium** use case | Communication Diagram | Shows object relationships and messages |
| **Complex** use case | Sequence Diagram | Shows time-ordered message flow in detail |

### 3.1 Sequence Diagram
Shows the **time-ordered sequence of messages** between objects to accomplish a use case.

```
Example: "Update student name"

:UI Form     :StudentController    :Student        :StudentDB
    |               |                  |                |
    |--updateName()->|                  |                |
    |               |--find(studentID)->|                |
    |               |<--student--------|                |
    |               |--setName(name)--->|                |
    |               |--save(student)------------------->|
    |               |<--success-------------------------|
    |<--confirmation-|                  |                |
```

- Read **top to bottom** = time flows downward
- Vertical lines = **lifelines** of each object
- Arrows = **messages** sent between objects

### 3.2 Communication Diagram
Shows the **same information as a sequence diagram** but organized around object relationships rather than time order.

```
Example: "Update student name"

:UI Form ──1: updateName()──→ :StudentController
                                      │
                          2: find()   │  3: setName()
                                      ▼
                                   :Student ──4: save()──→ :StudentDB
```

- Numbers indicate **message order**
- Emphasizes **which objects collaborate** with which

### 3.3 CRC Cards (Class-Responsibility-Collaboration)

A manual brainstorming tool — one card per class.

```
┌─────────────────────────────────────────────────┐
│  Class Name: Student                            │
├──────────────────────┬──────────────────────────┤
│  Responsibilities    │  Collaborators           │
├──────────────────────┼──────────────────────────┤
│  Know student ID     │  StudentController       │
│  Know student name   │  EnrollmentRecord        │
│  Know major          │  StudentDB               │
│  Update own name     │                          │
│  Return own data     │                          │
└──────────────────────┴──────────────────────────┘
Back of card: attributes (studentID, name, major...)
```

**How CRC Cards work:**
- Each card = one class
- Left column = what the class is **responsible for** (knowing and doing)
- Right column = which other classes it **collaborates with**
- Back of card = attributes needed

## 4. Design Class Diagram (DCD)

### 4.1 From DMCD to DCD

The **primary source** for the DCD is the Domain Model Class Diagram (DMCD) from analysis.

What gets **added** in the DCD:
1. **Elaborated attributes** — data types, initial values, visibility, properties
2. **Method signatures** — complete method definitions
3. **Stereotypes** — categorization of class types
4. **Navigation visibility arrows** — showing which objects can access which
5. **Additional objects** — window objects, database access objects, controller classes

### 4.2 Stereotypes

A **stereotype** categorizes a model element by its characteristics. Written using guillemets: `«stereotype»`

| Stereotype | Class Type | Description | Layer |
|---|---|---|---|
| *(no stereotype)* | **Entity class** | Problem domain class; usually **persistent** (data survives shutdown) | Problem domain |
| `«boundary»` | **Boundary class** | Exists on the automation boundary — input windows, web pages, forms | View layer |
| `«controller»` | **Controller class** | Mediates between boundary classes and entity classes; acts as a switchboard | Between view and domain |
| `«data access»` | **Data access class** | Retrieves data from and sends data to a database | Database access layer |

```
Three-Layer Mapping:
┌─────────────────────────────────────────┐
│  VIEW LAYER        «boundary» classes   │  ← Windows, Web pages, Forms
├─────────────────────────────────────────┤
│  DOMAIN LAYER      entity classes       │  ← Problem domain objects
│                    «controller» classes │  ← Mediators/switchboards
├─────────────────────────────────────────┤
│  DATA ACCESS LAYER «data access» class  │  ← DB read/write operations
└─────────────────────────────────────────┘
```

### 4.3 Persistent vs Non-Persistent Classes

| Type | Definition | Example |
|---|---|---|
| **Persistent class** | Objects exist after the system is shut down — data is remembered | Customer, Order, Product |
| **Non-persistent class** | Objects exist only while the system is running | UI window objects, session objects |

> Persistent classes are NOT marked with a stereotype — they are the default entity classes.

### 4.4 Abstract vs Concrete Classes

| Type | Definition | UML Notation |
|---|---|---|
| **Abstract class** | Cannot be instantiated; only provides a template for subclasses | Class name in *italics* |
| **Concrete class** | Can be instantiated — actual objects can be created | Normal class name |

> Static attributes and methods are shown with **underline**.

## 5. Elaborated Attributes

In the DCD, attributes are **fully elaborated** (unlike the DMCD which may only list names):

```
Format:
visibility  attributeName : DataType = initialValue {property}

Examples:
- customerID : String {key}
+ name : String
- balance : Float = 0.0
- isActive : Boolean = true
```

### Visibility Symbols

| Symbol | Meaning |
|---|---|
| `+` | **Public** — visible and accessible to other objects |
| `-` | **Private** — not visible; accessible only within the class itself |
| `#` | **Protected** — accessible within the class and its subclasses |

---

## 6. Method Signatures

A **method signature** contains all information needed to invoke (call) the method:

```
Format:
visibility  methodName(parameterList) : returnType

Examples:
+ placeOrder(items: List, customerID: String) : Order
+ getName() : String
- validateCard(cardNum: String) : Boolean
+ calculateTotal(discount: Float) : Float
```

| Component | Description |
|---|---|
| **Visibility** | + public, - private, # protected |
| **Method name** | Descriptive verb-noun |
| **Parameter list** | Input arguments with their data types |
| **Return type** | Data type returned (or void if nothing returned) |

## 7. Navigation Visibility

**Navigation visibility** = the ability of one object to **find and interact** with another object.

- Achieved by adding an **object reference variable** to a class
- Shown as an **arrowhead on the association line** in the DCD
- Answers: *"Which classes need to have references to which other classes?"*

### Navigation Visibility Guidelines

| Guideline | Rule |
|---|---|
| **Superior/Subordinate** | One-to-many associations: navigate from **superior → subordinate** |
| **Mandatory associations** | Navigate from the **more independent** class to the **dependent** class |
| **Information need** | When object A needs data from object B → add navigation arrow A → B |
| **Bidirectional** | Navigation arrows **may** go both ways if needed |

```
Example: Customer → Sale

Customer has a reference variable "mySale"
Therefore Customer can find and interact with Sale

Customer ──────────────→ Sale
          mySale: Sale
```

## 8. Building a First-Cut Design Class Diagram

### Three Steps to Build a First-Cut DCD

**Starting point:** Domain Model Class Diagram (DMCD)

#### Step 1: Add a Controller Class
Add a `«controller»` class that handles the use case.

```
For use case "Create Telephone Sale":
Add: «controller» TelephoneSaleHandler
This class will coordinate all other classes needed for this use case.
```

#### Step 2: Elaborate the Attributes
For each class involved in the use case:
- Add **data types** to all attributes
- Add **visibility** symbols (+ or -)
- Add **initial values** where applicable
- Mark **key attributes** with {key}

#### Step 3: Add Navigation Visibility Arrows
Determine which classes need references to which other classes and add arrowheads accordingly.

### Full Example — Create Customer Account

**Step 1: Identify primary problem domain class**
```
Primary class: Customer (the main entity being created)
Add controller: «controller» CustomerAccountHandler
```

**Step 2: Add boundary class (UI)**
```
Add: «boundary» CustomerAccountForm (the input window)
```

**Step 3: Add data access class**
```
Add: «data access» CustomerDB (to save the new customer)
```

**Resulting DCD structure:**
```
«boundary»                «controller»              entity
CustomerAccountForm ──→  CustomerAccountHandler ──→ Customer
                                  │                     │
                                  └──────────────────→  «data access»
                                                         CustomerDB
```

**Navigation flow:**
```
Form gets input → sends to Controller
Controller creates Customer object
Controller tells CustomerDB to save Customer
CustomerDB confirms save
Controller tells Form to show confirmation
```

## 9. The Full OOD Process — Summary Flow

```
USE CASE
    │
    ▼
Choose design tool:
    ├── Simple    → CRC Cards (brainstorm responsibilities)
    ├── Medium    → Communication Diagram
    └── Complex   → Sequence Diagram
    │
    ▼
Build First-Cut DCD:
    1. Start with DMCD
    2. Add «controller» class for use case
    3. Add «boundary» classes (UI windows/forms)
    4. Add «data access» classes
    5. Elaborate attributes (type, visibility, initial value)
    6. Add method signatures
    7. Add navigation visibility arrows
    │
    ▼
Refine DCD:
    → Sequence diagrams reveal additional methods needed
    → Add those methods back to the DCD
    → Iterate until complete
```

## 1. Introduction to Ridgeline Mountain Outfitters (RMO)

**RMO** is a fictional outdoor equipment retailer used throughout this course as a running case study. It provides a concrete, consistent example to illustrate how systems analysis and design concepts are applied in practice.

The specific system we will follow is the **RMO Tradeshow System** — an information system designed to support RMO's sales operations at trade shows and outdoor expos.

## 2. Systems and Subsystems

Before diving into the case study, two foundational concepts must be understood:

### System
> A **system** is a complete solution that functions by itself.

It is the whole — a set of interrelated components working together to achieve a defined purpose.

### Subsystem
> A **subsystem** is an identifiable and partitioned portion of a larger system. It does **not** function independently — it operates within the context of the full system.

**Example:** In the RMO Tradeshow System:
- The **overall Tradeshow System** is the system
- **Order processing**, **inventory lookup**, and **customer management** could each be subsystems

## 3. The Six Core Processes of the SDLC (Review)

Recall from Lecture 1A — the **System Development Life Cycle (SDLC)** consists of 6 core processes applied across multiple iterations:

| # | Core Process | Question Answered |
|---|-------------|-------------------|
| 1 | Identify the problem or need and obtain approval | *Should we do this?* |
| 2 | Plan and monitor the project | *How will we organise and track the work?* |
| 3 | Discover and understand the details | *What exactly does the system need to do?* |
| 4 | Design the system components | *How will the system be built?* |
| 5 | Build, test, and integrate components | *Does it work?* |
| 6 | Complete system tests and deploy | *Is it ready for real users?* |

In **iterative development**, all six processes are revisited in each iteration, with effort shifting across processes as the project progresses.

## 4. Key Documents in the SDLC

### 4.1 System Vision Document

> A **System Vision Document** is a brief statement describing the **overall objective** of the new system.

**Purpose:**
- Provides basic information about the *need* for the system and the proposed *solution approach*
- Helps the client/sponsor decide whether to **approve** a development project
- Produced during **Core Process 1** (Identify the problem/need)

**Typical contents:**
- Problem statement or business need
- Vision of the proposed solution
- Expected benefits
- High-level scope and constraints

> Think of it as the "elevator pitch" for the entire system — concise enough to communicate to decision-makers, specific enough to establish direction.

---

### 4.2 Work Breakdown Structure (WBS)

> A **Work Breakdown Structure (WBS)** is a hierarchical decomposition of **all the tasks** that must be accomplished for a project.

**Purpose:**
- Identify every piece of work required to complete the project
- Provide the foundation for building a **project schedule**
- Ensure nothing is forgotten

**Contents:**
- A comprehensive list of all tasks and sub-tasks
- An estimate of the **effort** (time/resources) required for each task
- Tasks are typically organised hierarchically (project → phases → tasks → sub-tasks)

**Example WBS structure for RMO Tradeshow System:**

```
RMO Tradeshow System Project
├── 1. Project Management
│   ├── 1.1 Project Planning
│   ├── 1.2 Progress Monitoring
│   └── 1.3 Stakeholder Communication
├── 2. Requirements Analysis
│   ├── 2.1 Stakeholder Interviews
│   ├── 2.2 Use Case Development
│   └── 2.3 Requirements Documentation
├── 3. System Design
│   ├── 3.1 Architecture Design
│   ├── 3.2 Database Design
│   └── 3.3 UI Design
├── 4. Development
│   ├── 4.1 Core Module Coding
│   ├── 4.2 Integration
│   └── 4.3 Unit Testing
└── 5. Deployment
    ├── 5.1 System Testing
    ├── 5.2 User Training
    └── 5.3 Go-Live
```

---

### 4.3 Iteration Schedule

> An **Iteration Schedule** is a project planning document that organises WBS tasks into a specific **iteration (time-boxed sprint)**, showing task dependencies, resource assignments, and timeline.

**How it differs from a WBS:**

| Aspect | WBS | Iteration Schedule |
|--------|-----|--------------------|
| **Purpose** | Identify *what* work needs to be done | Plan *when* and *by whom* each task is done |
| **Content** | Tasks + effort estimates | Tasks + dependencies + resources + timeline |
| **Dependencies** | Not shown | Explicitly shown (what must finish before what can start) |
| **Resource allocation** | Not included | Included — assigns people to tasks |
| **Usage** | Input to planning | Guides day-to-day project monitoring |

> In short: the **WBS tells you what to do**; the **Iteration Schedule tells you how to get it done within an iteration**.

## 5. Key Diagrams in Systems Analysis and Design

As we walk through the SDLC processes using RMO's case, two categories of diagrams are used:

### Analysis Diagrams (Core Process 3 — Discover & Understand)
Used to model **what** the system must do:
- **Use Case Diagrams** — show interactions between users (actors) and the system's functions
- **Activity Diagrams** — show the flow of activities within a process
- **Domain Class Diagrams** — show the key "things" in the business domain and their relationships

### Design Diagrams (Core Process 4 — Design)
Used to model **how** the system will be built:
- **Design Class Diagrams** — show the software classes and their attributes/methods
- **Sequence Diagrams** — show how objects interact over time to carry out a use case
- **Package Diagrams** — show the high-level architecture and grouping of components

## 6. Walking Through the SDLC with the RMO Tradeshow System

### Core Process 1 — Identify the Problem and Obtain Approval

**Goal:** Understand the business need and decide if a project is worth pursuing.

**RMO context:** RMO participates in trade shows and currently handles sales manually or with ad hoc tools. This creates problems with order accuracy, inventory visibility, and customer data management.

**Key output:** **System Vision Document**
- Problem: Inefficient, error-prone manual tradeshow sales process
- Proposed solution: A dedicated Tradeshow System to manage orders, inventory, and customers
- Expected benefits: Faster order processing, fewer errors, better data for follow-up

---

### Core Process 2 — Plan and Monitor the Project

**Goal:** Define how the project will be organised and tracked.

**Key outputs:**
- **WBS** — identifies all tasks needed to build the Tradeshow System
- **Iteration Schedule** — assigns tasks to iterations, showing dependencies and resource allocation
- **Project Charter** (from Integration Management) — formally authorises the project

**RMO context:** The PM creates a WBS covering analysis, design, development, and deployment phases, then schedules work into several iterations, each delivering a portion of the system.

---

### Core Process 3 — Discover and Understand the Details

**Goal:** Gather detailed requirements — what the system must do.

**Key techniques:** Interviews with stakeholders, observation, workshops
**Key diagrams:**
- **Use Case Diagrams** — e.g., "Record Sale", "Look Up Inventory", "Register Customer"
- **Activity Diagrams** — e.g., the flow of processing a tradeshow order
- **Domain Class Diagrams** — e.g., relationships between Customer, Order, Product, and Payment

---

### Core Process 4 — Design the System Components

**Goal:** Specify how the system will be built to satisfy the requirements.

**Key diagrams:**
- **Design Class Diagrams** — classes with attributes and methods (e.g., `Order`, `Product`, `Customer`)
- **Sequence Diagrams** — how the UI, controller, and database objects collaborate to process an order
- **Package Diagrams** — overall architecture (e.g., presentation layer, business logic layer, data layer)

---

### Core Process 5 — Build, Test, and Integrate

**Goal:** Code the designed components, unit test them, and integrate them into a working system.

**Activities:**
- Developers implement classes and methods from the design diagrams
- Unit tests verify individual components
- Integration tests verify that subsystems work together correctly

---

### Core Process 6 — Complete System Tests and Deploy

**Goal:** Validate the complete system against requirements, then go live.

**Activities:**
- System (end-to-end) testing
- User acceptance testing (UAT) with RMO staff
- Training end users
- Deploying the system to the tradeshow environment

## 9. Quick Reference: Q&A

**Q1. What is the difference between a system and a subsystem?**
A system is a complete, independently functioning solution. A subsystem is an identifiable portion of a system that does not function independently — it operates within the context of the whole system.

**Q2. What is the purpose of a System Vision Document?**
It provides a brief description of the overall system objective and proposed solution approach, giving the client/sponsor enough information to decide whether to approve the development project.

**Q3. What is the purpose of a WBS?**
To identify all the tasks (the work) that must be accomplished for a project, providing a comprehensive basis for building a project schedule.

**Q4. What are the components of a WBS and what does it show?**
A WBS contains a hierarchical list of all tasks and sub-tasks, along with an estimate of the effort each requires. It provides a complete picture of the work to be done and serves as the foundation for scheduling.

**Q5. What is the difference between a WBS and an Iteration Schedule?**
A WBS identifies *what* tasks need to be done. An Iteration Schedule takes those tasks and adds task *dependencies*, *resource assignments*, and a *timeline* for a specific iteration. The Iteration Schedule is used by the project manager to allocate resources and monitor day-to-day progress.

# Systems Design Fundamentals

---

## 1. Systems Analysis vs Systems Design

Before diving in, it is critical to understand where **design** sits in the overall development lifecycle.

| | Systems Analysis | Systems Design |
|---|---|---|
| **Focus** | Understanding **WHAT** the system should do | Describing **HOW** the system will be built |
| **Question asked** | What are the requirements? | How will we implement them? |
| **Outputs** | Requirements models (use cases, domain model, SSD) | Design models (class diagrams, DB schema, UI mockups) |
| **Who leads** | Systems Analyst | System Designer / Architect |

> 💡 **Key insight:** Analysis and design are not always sequential. In iterative/agile projects, they often happen **concurrently** — analysis informs design and design reveals gaps in analysis.

---

## 2. From Analysis Models to Design Models

During analysis, models are built to **represent the real world** and understand business processes:
- Use Case Diagrams
- Domain Model Class Diagrams
- System Sequence Diagrams (SSD)
- Activity Diagrams

During design, these are **converted into solution-oriented models**:
- Design Class Diagrams (with methods added)
- Database Schema (from domain model)
- UI wireframes and prototypes
- Component diagrams
- State-machine diagrams

The **formality** of a project affects how much design detail is explicitly modeled — a small project may skip some formal models, while a large enterprise system requires comprehensive documentation.

## 3. Five Key Systems Design Activities

Systems design is organized around five major activities. Each activity answers a specific design question.

```
┌─────────────────────────────────────────────────────────┐
│              FIVE DESIGN ACTIVITIES                     │
├─────────────────────────────────────────────────────────┤
│ 1. Describe the Environment                             │
│ 2. Design the Application Components                    │
│ 3. Design the User Interface                            │
│ 4. Design the Database                                  │
│ 5. Design Software Classes and Methods                  │
└─────────────────────────────────────────────────────────┘
```

### 3.1 Describe the Environment

The environment defines the **external context** in which the system will operate. Two key elements:

#### Communications with External Systems
When the system needs to communicate with other systems, designers must specify:

| Element | Description |
|---|---|
| **Message formats** | Structure and format of data exchanged (e.g., JSON, XML) |
| **Web and networks** | How the system connects (HTTP, HTTPS, REST APIs, etc.) |
| **Communication protocols** | Rules governing data transmission (TCP/IP, SOAP, REST) |
| **Security methods** | How communication is secured (SSL/TLS, tokens, certificates) |
| **Error detection and recovery** | How the system handles failed or corrupted transmissions |

#### Conforming to an Existing Technology Architecture
Most organizations have an existing **technology architecture** — the set of:
- Computing hardware (servers, workstations, mobile devices)
- Network hardware and topology (routers, switches, LAN/WAN)
- System software (OS, middleware, databases, frameworks)

New systems must **fit within** this existing architecture unless a deliberate architectural change is approved.

---

### 3.2 Design the Application Components

An **application component** is a well-defined unit of software that performs one or more specific functions.

#### Key Packaging Decisions

| Decision | Questions to Answer |
|---|---|
| **Scope and size** | What functions does this component perform? Where are its boundaries? What are its interfaces? |
| **Programming language** | What languages are acceptable in this environment? |
| **Build or buy** | Is an acceptable version available to purchase/license? Is it worth building from scratch? |

#### Grouping and Interaction
- How will system functions be **grouped or packaged** into components?
- How will components **interact** with one another once assembled?
- What is the **impact of the technology architecture** on component design?

This activity is followed by **software class design** (Activity 5).

#### Typical Models Used
- **Package diagrams** — showing how components are grouped
- **Component diagrams** — showing component interfaces and dependencies
- **Deployment diagrams** — showing how components map to hardware

---

### 3.3 Design the User Interface

> *"To the user, the UI is the system."*

The user interface is the most visible part of any system. Poor UI design directly reduces user productivity and system adoption.

#### Characteristics of UI Design
- Involves both **analysis tasks** (understanding user needs) and **design tasks** (creating the interface)
- Requires **heavy user involvement** throughout
- Must support **multiple devices and environments** — desktop browsers, mobile apps, tablets, voice interfaces

#### UI Design Process
1. Understand user tasks and workflows (from use cases and activity diagrams)
2. Define navigation structures and screen flows
3. Create wireframes and prototypes
4. Validate with users through usability testing
5. Refine iteratively based on feedback

#### Key UI Design Principles
- **Consistency** — similar actions behave similarly across the system
- **Feedback** — system communicates what is happening
- **Error prevention** — design to prevent mistakes before they occur
- **Efficiency** — minimize steps for frequent tasks
- **Accessibility** — usable by people with different abilities

---

### 3.4 Design the Database

All information systems require persistent data storage — typically in a **Relational Database Management System (RDBMS)**.

#### Core Task
Convert the **Domain Model Class Diagram** (from analysis) into a **relational database schema**.

#### Mapping Rules (Domain Model → Database)
| Domain Model Element | Database Element |
|---|---|
| Domain Class | Table |
| Attribute | Column |
| Key/Identifier | Primary Key |
| Association (1-to-many) | Foreign Key |
| Many-to-many association | Junction/Bridge Table |
| Generalization/Specialization | Multiple strategies (single table, table per class, table per concrete class) |

#### Additional Technical Considerations
- **Throughput and response time** — how many transactions per second? How fast must queries return?
- **Indexing strategy** — which columns need indexes for performance?
- **Normalization** — eliminate data redundancy
- **Security** — who can access which tables? Row-level vs column-level security?
- **Backup and recovery** — how is data protected?

---

### 3.5 Design Software Classes and Methods

This activity **extends the analysis models** to incorporate software-specific elements that were not present in the domain model.

#### What Gets Added
- **Methods** (operations) on each class — the domain model has only attributes; software classes need behavior
- **Visibility** modifiers (public, private, protected)
- **Data types** for each attribute
- **Constructors and destructors**
- **Navigation arrows** showing which class "knows about" the other

#### Resulting Models

| Model | Purpose |
|---|---|
| **Design Class Diagram** | Full class diagram with attributes, methods, visibility, and types |
| **Sequence Diagrams** | Shows how objects interact over time to carry out a use case |
| **State-Machine Diagram** | Shows the different states an object can be in and transitions between states |

#### Design Class Diagram vs Domain Model Class Diagram

| | Domain Model Class Diagram | Design Class Diagram |
|---|---|---|
| Methods? | ❌ No | ✅ Yes |
| Data types? | Optional | Required |
| Visibility? | No | Yes (+/-/#) |
| Purpose | Understand the problem domain | Blueprint for implementation |

## 5. Integrity Controls

**Controls** are mechanisms and procedures built into a system to **safeguard the system and the information within it**.

**Integrity controls** specifically:
- Reject **invalid data inputs**
- Prevent **unauthorized data outputs**
- Protect data and programs against **accidental or malicious tampering**

### Primary Objectives of Integrity Controls
1. Ensure that only **appropriate and correct business transactions** occur
2. Ensure that transactions are **recorded and processed correctly**
3. **Protect and safeguard** the assets of the organization (hardware, software, and information)

### Four Key Aspects of Integrity Controls

#### 5.1 Input Controls
Controls that **prevent invalid or erroneous data** from entering the system.

Examples:
- **Field validation** — checking data type, format, range (e.g., date must be in YYYY-MM-DD format)
- **Completeness checks** — required fields must not be empty
- **Consistency checks** — end date must be after start date
- **Referential integrity** — foreign key values must exist in the referenced table
- **Check digits** — e.g., the last digit of a credit card number is a calculated check

#### 5.2 Output Controls
Controls that ensure output **arrives at the proper destination** and is **accurate, current, and complete**.

Examples:
- Totals and record counts on reports
- Confirmation messages sent to correct recipients
- Audit trails showing who received what output
- Timestamping of output

#### 5.3 Redundancy, Backup, and Recovery
Procedures designed to **protect software and data** from hardware failure and malicious destruction.

Examples:
- **RAID storage** for disk redundancy
- **Regular backups** (daily, weekly, off-site)
- **Disaster recovery plans** — documented procedures for restoring systems
- **Failover systems** — automatic switchover to backup systems
- **Transaction logs** — enable rollback to a known good state

#### 5.4 Fraud Prevention
Means to prevent users — including **rightful users** — from committing fraud.

**The Fraud Triangle** — three conditions that enable fraud:

```
         Opportunity
            /\
           /  \
          /    \
         /      \
        ──────────
     Motive   Rationalization
```

| Element | Description | System Design Response |
|---|---|---|
| **Opportunity** | The ability to commit fraud | Separate duties, limit access, audit trails |
| **Motive** | A reason to commit fraud | Outside scope of system design |
| **Rationalization** | Justifying the act | Visible monitoring, clear policies |

Design strategies to reduce opportunity:
- **Separation of duties** — no single person can both approve and process a transaction
- **Mandatory dual authorization** for large transactions
- **Audit logging** — every transaction is recorded with user ID, timestamp
- **Alerts** for unusual patterns

## 6. Security Controls

**Security controls** protect the assets of an organization from **all threats**, with a **primary focus on external threats**.

In addition to integrity control objectives, security controls must:
- Maintain a **stable, functioning operating environment** for users and applications
- Protect information and transactions **during transmission** across insecure environments (public Wi-Fi, the Internet)

### 6.1 Access Control
A security control that **limits a user's ability to access resources** such as servers, files, web pages, application programs, and database tables.

Key concepts:
- **Access Control List (ACL)** — a list attached to a specific resource describing users/groups and the nature of their permitted access
- **Role-Based Access Control (RBAC)** — access is granted based on the user's role (e.g., Admin, Manager, Viewer)
- **Principle of least privilege** — users should have only the minimum access needed to do their job

### 6.2 Authentication
The process of **identifying users** who request access to sensitive resources.

| Method | Description | Example |
|---|---|---|
| **Something you know** | Password, PIN, security questions | Username + password |
| **Something you have** | Physical token, smart card, phone | OTP via SMS |
| **Something you are** | Biometrics | Fingerprint, face recognition |
| **Multi-factor (MFA)** | Combines two or more methods | Password + OTP |

### 6.3 Authorization
The process of **allowing or restricting** a specific authenticated user's access to a specific resource based on an access control list.

```
User requests resource
        ↓
Authentication: WHO are you? → verify identity
        ↓
Authorization: WHAT can you do? → check ACL
        ↓
Access granted or denied
```

### 6.4 Data Encryption

**Encryption** is the process of **altering data** so that unauthorized users cannot read them.
**Decryption** is the process of **converting encrypted data back** to their original state.
An **encryption algorithm** is a complex mathematical transformation that encrypts or decrypts binary data.

#### Types of Encryption

| Type | Description | Key Characteristic | Use Case |
|---|---|---|---|
| **Symmetric key encryption** | Same key used to encrypt and decrypt | Fast, but key must be shared securely | Encrypting data at rest (files, databases) |
| **Asymmetric key encryption** | Public key encrypts; private key decrypts (or vice versa) | Secure key exchange; slower | HTTPS, digital signatures, email encryption |

#### Encryption Key
An **encryption key** is a piece of data used by the algorithm to transform plaintext into ciphertext.
- Longer keys = stronger encryption
- AES-256 is a common symmetric standard; RSA-2048 is common for asymmetric

#### Where Encryption Is Applied
- **Data at rest** — databases, files, backups (symmetric encryption)
- **Data in transit** — SSL/TLS for HTTPS connections (asymmetric + symmetric hybrid)
- **End-to-end encryption** — only sender and recipient can read (messaging apps)

## 7. Integrity Controls vs Security Controls — Comparison

| | Integrity Controls | Security Controls |
|---|---|---|
| **Primary focus** | Internal correctness and validity of data | Protection from external and internal threats |
| **Scope** | Within a specific system | Across multiple systems |
| **Threats addressed** | Invalid input, unauthorized output, accidental damage | External attacks, unauthorized access, data interception |
| **Examples** | Input validation, backup/recovery, fraud prevention | Authentication, encryption, access control |
| **Goal** | Ensure transactions are correct and complete | Ensure assets are protected from all threats |

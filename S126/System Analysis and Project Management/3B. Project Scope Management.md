## 1. What is Project Scope Mangement?

### Scope 
>**Scope** refers to all the work involved in creating the products of a project, and the processes used to create them.

A **deliverable** isa product produced as part of a project - this can be hardware, software, planning documents, or even meeting minutes.

### Project Scope Mangement
> **Project Scope Mangement** includes all the processes involved in **defining and controlling what is - and is not - included** in a project.

Its core goal is to ensure that the project team and all stakeholders share the same understanding of:
- What products the project will produce
- What processes the team will use to produce them

Without this shared understanding, teams waste effort on work that was never needed, or miss work that was critical.

---

## 2. The Six Processes of Project Scope Management

```
Planning -> Collecting -> Defining -> Creating WBS -> Validating -> Controlling
```

### Purposes of the 6 Processes
1. **Planning Scope Mangement:** Determine how scope and requirements will be manged
2. **Collecting Requirements:** Define and document features, functions, and processes needed
3. **Defining Scope:** Create a detailed project scope statement
4. **Creating the WBS:** Subdivide major deliverables into smaller, manageable components
5. **Validating Scope:** Formally accept completed project deliverables
6. **Controlling Scope:** Manage changes to project scope throughout the project life cycle

---

## 3. Planning Scope Mangement

The project team uses expert judgment, data analysis, and meetings to produce two important planning documents.

### 3.1 Scope Management Plan
A subsidiary part of the overall Project Management Plan. It documents **how scope will be managed**, covering:
- How to prepare a detailed project scope statement
- How to create, maintain, and approve the WBS
- How to obtain formal acceptance of completed deliverables
- How to control requests for changes to project scope

### 3.2 Requirements Management Plan
Documents **how project requirements will be analysed, documented, and managed**, including:
- How to plan, track, and report requirements activities
- How to perform configuration management activities
- How to prioritise requirements
- How to use product metrics
- How to trace and capture attributes of requirements

> **Requirement (PMBOK Guide, 6th Ed.):** "A condition or capability that is necessary to be present in a product, service, or result to satisfy a business need."

---

## 4. Collecting Requirements

Requirements must be gathered from stakeholders before scope can be properly defined. Common techniques include:

1. **Interviewing stakeholders**: One-to-one or group discussions with key stakeholders
2. **Focus group & facilitated workshop:** Structured group sessions to elicit requirements collaboratively
3. **Group creativity techniques:** Brainstorming, nominal group technique, affinity diagrams
4. **Questionnaires & surveys:** Written questions distributed to a large group of stakeholders
5. **Observation studies:** Watching users perform their current work to identify needs
6. **Benchmarking:** Comparing with similar projects or products to identify best practices

---

## 5. Defining Scope - The Scope Statement

The **scope statement** is the primary output of the Defining Scope process. It establishes a clear, shared understanding of the project boundaries. 

### Important Elements

1. **Product scope description:** Describes the characteristics of the product or service to be created
2. **Product user acceptance criteria:** The conditions that must be met for deliverables to be formally accepted
3. **Detailed deliverables information:** Describes all outputs the project will produce
4. **Project boundaries:** Explicitly states what is OUT of scope
5. **Constraints:** Limitations on the project (budget cap, technology restrictions, etc.)
6. **Assumptions:** Things assumed to be true for planning purposes
7. **Supporting document references:** References to product specs, regulations, etc.

> As a project progresses, the scope statement becomes increasingly detailed and specific. Initial statements are hight-level; later versions are precise.

## 6. Creating the Work Breakdown Structure (WBS)

### What is a WBS?
> A **Work Breakdown Structure (WBS)** is a **deliverable-oriented grouping** of all the work involved in a project that defines the **total scope of the project**.

It is the foundation document that provides the basis for planning and managing project schedules, costs, resources, and changes.

### Key Concepts

1. **Decomposition:** The main technique - subdividing deliverables into progressively smaller components
2. **Work package:** The lowest-level task in the WBS - small enough to estimate, assign, and track
3. **Scope baseline:** The approved scope statement + WBS + WBS dictionary; used as a reference point for controlling changes
 
### Example WBS (Software Product Release 5.0)
 
```
1.0 Software Product Release 5.0
├── 1.1 Project Management
│   ├── 1.1.1 Planning
│   ├── 1.1.2 Meetings
│   └── 1.1.3 Administration
├── 1.2 Product Requirements
│   ├── 1.2.1 Software
│   ├── 1.2.2 User Documentation
│   └── 1.2.3 Training Program Materials
├── 1.3 Detail Design
│   ├── 1.3.1 Software
│   └── 1.3.2 User Documentation
├── 1.4 Construct
│   ├── 1.4.1 Software
│   ├── 1.4.2 User Documentation
│   └── 1.4.3 Training Program Materials
└── 1.5 Integration and Test
    ├── 1.5.1 Software
    ├── 1.5.2 User Documentation
    └── 1.5.3 Training Program Materials
```
 
### Five Approaches to Developing a WBS
 
| Approach | Description | Best Used When |
|----------|-------------|----------------|
| **Guidelines** | Follow templates from standards bodies (e.g., DOD) | Organisation has established standards |
| **Analogy** | Adapt WBS from a similar completed project | Similar projects exist as reference |
| **Top-down** | Start from the whole project and break it down | Scope is well understood upfront |
| **Bottom-up** | Start from specific tasks and group upward | Team has deep task-level knowledge |
| **Mind mapping** | Use branching diagrams to brainstorm tasks visually | Early-stage creative planning |

### Golden Rules for Creating a WBS
 
- Each unit of work should appear at **only one place** in the WBS
- The work content of a WBS item = the **sum of all items below it**
- Each WBS item should be the responsibility of **only one individual** (even if many people work on it)
- The WBS must reflect **how work will actually be performed** — it serves the project team first
- **Project team members should be involved** in developing the WBS to ensure accuracy and buy-in
- Each WBS item must be documented in the WBS Dictionary for accurate understanding
- The WBS must be **flexible enough** to accommodate inevitable changes while maintaining scope control

## 7. The WBS Dictionary
 
Many WBS task names are too brief to be unambiguous. The **WBS Dictionary** provides detailed information about each WBS item.
 
### Typical WBS Dictionary Entry Contents
- WBS item number and name
- Detailed description of the work involved
- Responsible person/team
- Dependencies (what must precede or follow this task)
- Resource requirements
- Estimated effort and duration
 
### Example Entry
 
| Field | Content |
|-------|---------|
| Project Title | IT Upgrade Project |
| WBS Item Number | 2.2 |
| WBS Item Name | Database Update |
| Description | Review current hardware/software database, produce departmental reports, update data after physical inventory. Dependent on WBS Item 2.1 (Physical Inventory). Must precede WBS Item 3.0 (Hardware and Software Acquisition). |

## 8. Scope Creep — The Silent Project Killer
 
> **Scope creep** is the tendency for project scope to gradually expand beyond its original boundaries, often without corresponding adjustments to time, cost, or resources.
 
Scope creep is one of the most common causes of IT project failure. It happens when:
- New features are added informally without review
- Stakeholders make verbal change requests that bypass formal processes
- Requirements were vague from the start and expand as the project progresses
 
### Strategies to Reduce Scope Creep
 
**Improving user input:**
- Place users directly on the project team
- Conduct regular meetings with defined agendas
- Deliver working software to users frequently (so they can react early)
- Never promise to deliver what the team cannot achieve in the given timeframe
- Locate users physically with developers where possible
 
**Managing changing requirements:**
- Develop and follow a formal requirements management process
- Use techniques such as prototyping, use case modelling, and JAD (Joint Application Development)
- Put all requirements in writing and keep them current
- Use a requirements management database
- Conduct testing throughout the entire project life cycle
- Review all change requests from a systems perspective
- Emphasise completion dates to keep focus on priorities
- Allocate specific resources for handling change requests

## 9. Validating Scope
 
> **Validating Scope** involves the **formal acceptance** of completed project deliverables by the customer or sponsor.
 
Acceptance is typically achieved through:
1. Customer inspection of the deliverable
2. Formal sign-off confirming the deliverable meets the agreed acceptance criteria
 
This process ensures there is no ambiguity about whether a deliverable is "done."
 
---
 
## 10. Controlling Scope
 
> **Controlling Scope** involves managing changes to the project scope throughout the entire project life cycle.
 
### Goals of Scope Control
- Influence the factors that cause scope changes (prevent unnecessary change requests)
- Ensure all changes are processed through the **integrated change control** process
- Manage and incorporate approved changes when they do occur
 
### Key Concept: Variance
> **Variance** = the difference between planned performance and actual performance.
 
Scope variance is tracked by comparing actual work completed against the scope baseline. When variance is detected, corrective action must be taken and trade-offs in the triple constraint must be evaluated.

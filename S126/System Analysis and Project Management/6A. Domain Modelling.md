# Domain Modelling

---

## 1. Problem Domain and Things

- **Problem domain**: The specific area (or domain) of the users' business need that is within the scope of an information system.
- **Things**: Items users work with when accomplishing tasks. They need to be remembered by the system.
  - Examples: products, sales, shippers, customers, invoices, payments
- Things are modelled as **domain classes** (in system analysis) or **data entities** (in database design).

---

## 2. Identifying Things in the Problem Domain

There are two main techniques for identifying things:

1. **Brainstorming Technique**
2. **Noun Technique**

---

## 3. Brainstorming Technique

Uses a checklist of typical types of things to systematically identify domain classes.

### Categories of Things

| Category | Examples |
|---|---|
| Tangible things | airplane, book, vehicle, document, worksheet |
| Roles played | employee, customer, doctor, patient, end user, system administrator |
| Organizational units | division, department, section, task force, workgroup |
| Devices | sensor, timer, controller, assembly line, production machine, sorter, printer, inventory bin |
| Sites / Locations | warehouse, branch office, factory, retail store, desktop |
| Incidents, events, or interactions | flight, service call, logon, logoff, contract, purchase, order, payment |

### Steps

1. Identify a user and a set of use cases.
2. Brainstorm with the user to identify things involved when carrying out the use case — things about which information should be captured by the system.
3. Use the categories above to systematically ask questions, e.g.:
   - Are there any tangible things you store information about?
   - Are there any locations involved?
   - Are there roles played by people that you need to remember?
4. Continue to work with all types of users and stakeholders to expand the brainstorming list.
5. Merge the results, eliminate any duplicates, and compile an initial list.

---

## 4. Noun Technique

Identifies problem domain classes by finding, classifying, and refining a list of nouns that come up in discussions or documents.

**Characteristics:**
- Popular and systematic
- Produces long lists; many nouns will not need to be stored
- Difficulty in identifying synonyms and things that are really attributes
- Good starting point when no users are available to help brainstorm

### Steps

1. Using use cases, actors, and other system information (including inputs and outputs), identify all nouns.
2. Add items or categories from existing systems, current procedures, and current reports or forms.
3. Refine the list by asking these **inclusion** questions:
   - Is it a unique thing the system needs to know about?
   - Is it inside the scope of the system?
   - Does the system need to remember more than one of these items?
4. Ask these **exclusion** questions:
   - Is it really a synonym for something already identified?
   - Is it just an output produced from other information?
   - Is it just an input that results in recording some other information?
5. Ask these **research** questions:
   - Is it likely to be a specific piece of information (attribute) about another thing identified?
   - Is it something that might be needed if assumptions change?
6. Create a master list of all nouns identified, noting whether each should be **included**, **excluded**, or **researched further**.
7. Review the list with users, stakeholders, and team members, then define the final list of things in the problem domain.

## 5. Details about Domain Classes

### Attribute
Describes one piece of information about each **instance** of the class.
- Example: first name, last name, and phone number of a customer.

### Identifier / Key
An attribute that **uniquely identifies** an instance of the class.
- Example: customer ID.

### Compound Attribute
Two or more attributes combined into one structure to simplify the model.
- Example: an address includes house number, street, city, state, and zip.
- Sometimes an identifier or key is itself a compound attribute.

### Class vs. Object
- **Class**: A type of thing (e.g., Customer).
- **Object**: A specific instance of that class (e.g., John Smith, ID 101).
- Each object has its own values for each attribute.

**Example:**

| Attribute | Customer 101 | Customer 102 | Customer 103 |
|---|---|---|---|
| Customer ID | 101 | 102 | 103 |
| First name | John | Mary | Bill |
| Last name | Smith | Jones | Casper |
| Home phone | 555-9182 | 423-1298 | 874-1297 |
| Work phone | 555-3425 | 423-3419 | 874-8546 |

## 6. Associations Among Things

**Association**: A naturally occurring relationship between classes.

- Associations apply in **two directions** and should be read each way:
  - "A customer places an order"
  - "An order is placed by a customer"

### Multiplicity (Cardinality)
The number of instances involved in an association.

| Scenario | Multiplicity |
|---|---|
| A customer has placed no order yet, but might place many over time | Zero or more (optional) |
| An order must be placed by exactly one customer | One and only one (mandatory) |
| An order contains at least one item, but could contain many | One or more (mandatory) |

### UML Notation for Multiplicity

| Notation | Meaning |
|---|---|
| `0..1` | Zero or one (optional) |
| `1` | One and only one (mandatory) |
| `1..1` | One and only one — alternate notation |
| `0..*` or `*` | Zero or more (optional) |
| `1..*` | One or more (mandatory) |

### Types of Associations

- **Binary Association**: Between exactly two different classes.
  - Example: Course Section includes Students; Members join Club.
- **Unary Association (Recursive)**: Between two instances of the same class.
  - Example: Person married to Person; Part is made using Parts.
- **Ternary Association**: Among three classes.
- **N-ary Association**: Among n classes.

## 7. The Domain Model Class Diagram (DMCD)

### Key Definitions

| Term | Definition |
|---|---|
| **Class** | A classification used to describe a collection of objects |
| **Domain Class** | Classes that describe objects in the problem domain |
| **Class Diagram** | A UML diagram showing classes with attributes and associations (and methods for software classes) |
| **Domain Model Class Diagram** | A class diagram that only includes classes from the problem domain — no methods |

### Notation Rules

- **Class name**: Always capitalized (e.g., `Customer`, `OrderItem`).
- **Attribute names**: Lowercase, using **camelBack notation** (e.g., `custNumber`, `homePhone`, `billAddress`).
- **Compound class names**: Also use camelBack notation (e.g., `OrderItem`).
- **Key attributes**: Marked with `{key}` (e.g., `custNumber {key}`).
- **No methods** in a Domain Model Class Diagram.

### Simple DMCD Example

```
Customer                 Order                    OrderItem
──────────────────       ──────────────────       ──────────────────
custNumber               orderID                  itemID
name                     orderDate                quantity
billAddress      1  0..* amount           1  1..* price
homePhone
officePhone
          [places]                  [consists of]
```

Readings:
- A customer **places** zero or more orders.
- An order **is placed by** exactly one customer.
- An order **consists of** one or more order items.
- An order item **is part of** exactly one order.

### Bank DMCD Example

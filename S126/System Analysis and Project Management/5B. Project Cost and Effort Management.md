# Project Cost & Effort Management

---

## 1. Why Cost Management Matters

IT projects have a **poor track record** of meeting budget goals:

- **Cost overrun** = the additional percentage or dollar amount by which actual costs exceed estimates
- A 2011 Harvard Business Review study found an average cost overrun of **27%**
- The most dangerous finding: **"black swans"** — rare, high-impact, unpredictable cost explosions

### Real-World Example: UK NHS IT Disaster
- The UK National Health Service IT modernisation program was called **"the greatest IT disaster in history"**
- Estimated overrun: **$26 billion**
- Causes: incompatible systems, resistance from physicians, contractor disputes
- **Scrapped in 2011** after years of failure

> The lesson: poor cost estimation and management is not just an inconvenience — it can destroy entire programs.

---

## 2. What is Cost?

> **Cost** = a resource sacrificed or foregone to achieve a specific objective; usually measured in monetary units (dollars) paid to acquire goods and services.

### Project Cost Management — 4 Processes

| Process | Description |
|---------|-------------|
| **Planning cost management** | Determining the policies, procedures, and documentation for planning, executing, and controlling costs |
| **Estimating costs** | Developing an approximation of the costs of resources needed to complete the project |
| **Determining the budget** | Allocating the overall cost estimate to individual work items → establishes a **cost baseline** |
| **Controlling costs** | Controlling changes to the project budget |

---

## 3. Basis for Successful Estimating

Good estimates require two things:

**1. Information about past projects:**
- How big were they?
- How much effort/time did they need?
- What was the team's productivity?

**2. Ability to measure the amount of work involved:**
- Traditional size measurement for software = **Lines of Code (LOC)**
- More modern approach = **Function Points**

---

## 4. A Taxonomy of Estimating Methods

Four main categories of estimation approaches:

```
Estimating Methods
├── Bottom-up (activity based, analytical)
├── Top-down / Parametric (algorithmic models)
│   ├── Function Points (Albrecht/IFPUG)
│   └── COCOMO / COCOMO II
├── Expert Judgment
└── Analogy (case-based, comparative)
```

---

### 4.1 Bottom-Up Estimating

**How it works:**
1. Break the project into smaller and smaller components
2. Stop when you reach what **one person can do in one or two weeks**
3. Estimate costs for the **lowest-level activities** (work packages)
4. At each higher level, calculate the estimate by **summing** the estimates below

```
Project Total
├── Phase A ──► sum of tasks below
│   ├── Task A1 ──► estimate directly
│   ├── Task A2 ──► estimate directly
│   └── Task A3 ──► estimate directly
└── Phase B ──► sum of tasks below
    ├── Task B1 ──► estimate directly
    └── Task B2 ──► estimate directly
```

**Advantage:** Very accurate for well-understood projects
**Disadvantage:** Time-consuming; requires detailed task breakdown first (WBS)

> 💡 **Link to Scope Management:** Bottom-up estimating **requires a WBS** — this is why the WBS is called the "foundation document" for planning costs, schedules, and resources.

---

### 4.2 Top-Down Estimating

**How it works:**
1. Produce an **overall estimate** for the whole project using effort drivers
2. **Distribute proportions** of the overall estimate to components

```
Overall Project = 100 days
├── Design   30% = 30 days
├── Code      30% = 30 days
└── Test      40% = 40 days
```

**Advantage:** Fast; useful early when details are not yet known
**Disadvantage:** Less accurate; relies on correct proportional distribution

---

### 4.3 Expert Judgment

- Ask someone **familiar with the application area and technologies** to provide an estimate
- Particularly useful when **existing code is being modified** (change impact analysis)
- Research shows expert judgment in practice tends to be based on **analogy** (past experience)

---

### 4.4 Estimating by Analogy (Case-Based Reasoning)

**How it works:**
1. Collect attribute values of the **new (target) project**
2. Search a database of **completed (source) projects**
3. Select the source project with the **closest attribute values**
4. Use the effort from that source project as the estimate for the target

```
Source Cases (completed projects):
  Project A: [attributes...] → effort = 120 days
  Project B: [attributes...] → effort = 85 days
  Project C: [attributes...] → effort = 200 days
                                          ↑
Target Case: [new attributes] → most similar to Project A
→ Estimated effort ≈ 120 days
```

## 5. Albrecht/IFPUG Function Points

### Background
- Developed by **Allan Albrecht** at IBM in **1979**
- Maintained by **International Function Point Users Group (IFPUG)**
- Now an **ISO standard**
- Measures system size based on **functionality delivered to the user**, not lines of code

### The 5 Function Types

Function points are counted across **5 types**, grouped into 2 categories:

#### Data Functions (what data the system stores/references)

| Type | Abbreviation | Description | Example |
|------|-------------|-------------|---------|
| **Internal Logical File** | ILF | Data stored and maintained **within** your application | Database tables, flat files |
| **External Interface File** | EIF | Data your application **uses but does not maintain** (maintained by another application) | ILFs of other systems |

#### Transaction Functions (what the system does with data)

| Type | Abbreviation | Description | Example |
|------|-------------|-------------|---------|
| **External Input** | EI | Input transaction that **updates ILFs** | Data entry by users, file feeds |
| **External Output** | EO | Transaction that extracts and displays data from ILFs, **with derived/computed data** | Reports with calculations |
| **External Query** | EQ | User-initiated transaction that provides information **without updating ILFs and without derived data** | Simple data retrieval reports |

> 💡 **Key distinction:** EO vs EQ — if the report involves **computation/derived data** → EO; if it just retrieves and displays raw data → EQ.

---

### Step 1: Assess Complexity

#### For Data Functions (ILF and EIF)

Complexity is based on:
- **RET** = Record Element Types (subgroups of data elements within the file)
- **DET** = Data Element Types (unique, non-repeated fields)

| RETs | 1–19 DETs | 20–50 DETs | 51+ DETs |
|------|-----------|------------|----------|
| 1 | Low | Low | Medium |
| 2–5 | Low | Medium | High |
| 6+ | Medium | High | High |

**Example:** A Customer File contains: Name, Address fields + a separate subgroup for Credit Cards.
- RETs = 2 (main customer data + credit card subgroup)
- Count the unique fields → determines DETs
- Cross-reference table → complexity rating

#### For Transaction Functions (EI, EO, EQ)

Complexity is based on:
- **FTR** = File Type References (number of ILFs or EIFs referenced)
- **DET** = Data Element Types in the transaction

**For EI:**

| FTRs | 1–4 DETs | 5–15 DETs | 16+ DETs |
|------|----------|-----------|---------|
| 0–1 | Low | Low | Medium |
| 2 | Low | Medium | High |
| 3+ | Medium | High | High |

**For EO and EQ:**

| FTRs | 1–5 DETs | 6–19 DETs | 20+ DETs |
|------|----------|-----------|---------|
| 0–1 | Low | Low | Medium |
| 2–3 | Low | Medium | High |
| 4+ | Medium | High | High |

---

### Step 2: Apply Complexity Multipliers (Unadjusted Function Points)

| Type | Low | Medium | High |
|------|-----|--------|------|
| **EI** | 3 | 4 | 6 |
| **EO** | 4 | 5 | 7 |
| **EQ** | 3 | 4 | 6 |
| **ILF** | 7 | 10 | 15 |
| **EIF** | 5 | 7 | 10 |

**UFP = Σ (count of each function type × complexity multiplier)**

---

### Step 3: Calculate Value Adjustment Factor (VAF)

The **VAF** accounts for the overall environmental and processing complexity of the application.

**14 General System Characteristics (GSCs)** are each rated on a scale of **0–5**:

| # | Characteristic | Description |
|---|---------------|-------------|
| 1 | Data communications | How many communication facilities? |
| 2 | Distributed data processing | How are distributed functions handled? |
| 3 | Performance | Response time/throughput requirements? |
| 4 | Heavily used configuration | How heavily used is the hardware? |
| 5 | Transaction rate | How frequently are transactions executed? |
| 6 | On-line data entry | What % of information is entered on-line? |
| 7 | End-user efficiency | Designed for end-user efficiency? |
| 8 | On-line update | How many ILFs updated on-line? |
| 9 | Complex processing | Extensive logical/mathematical processing? |
| 10 | Reusability | Developed for one or many users' needs? |
| 11 | Installation ease | How difficult is conversion/installation? |
| 12 | Operational ease | How effective are start-up/backup/recovery? |
| 13 | Multiple sites | Designed for multiple sites/organisations? |
| 14 | Facilitate change | Designed to facilitate change? |

**Formulas:**

$$TDI = \sum_{i=1}^{14} DI_i$$

$$VAF = 0.65 + (0.01 \times TDI)$$

$$FP = UFP \times VAF$$

> The VAF ranges from **0.65** (all GSCs = 0) to **1.35** (all GSCs = 5), meaning it can adjust UFP by ±35%.

---

### Complete Function Point Example

**Payroll Application:**

| Function | Type | Complexity | FP Count |
|----------|------|-----------|---------|
| Input/amend/delete employee details | EI | Medium | 4 |
| Calculate and update pay from timesheet | EI | High | 6 |
| Compute and print pay-to-date details | EO | Medium | 5 |
| Payroll details file | ILF | Medium | 10 |
| Personnel file (external, name/address only) | EIF | Low | 5 |
| **UFP Total** | | | **30** |

If previous projects delivered **5 FPs per day**:
$$\text{Estimated time} = \frac{30}{5} = 6 \text{ days}$$

## 6. COCOMO — Constructive Cost Model

### Background
- Developed by **Barry Boehm** in **1981** (COCOMO 81)
- Updated as **COCOMO II** in 2000
- Measures size in **KLOC** (thousands of lines of code)
- Estimates effort in **person-months**

---

### COCOMO 81 — Basic Model

$$\text{Effort} = c \times \text{size}^k$$

Where size is in **KLOC** and effort is in **person-months**.

#### Three System Types

| System Type | c | k | Description |
|------------|---|---|-------------|
| **Organic** | 2.4 | 1.05 | Small team, small system, familiar environment, flexible requirements |
| **Semi-detached** | 3.0 | 1.12 | Mix of organic and embedded characteristics |
| **Embedded** | 3.6 | 1.20 | Very tight constraints, changes are very costly (e.g. real-time systems) |

**Example:** Organic system with 50,000 lines of code (50 KLOC):
$$\text{Effort} = 2.4 \times 50^{1.05} = \text{approx. } 152 \text{ person-months}$$

---

### COCOMO II — Updated Model

$$pm = A \times (\text{size})^{sf} \times em_1 \times em_2 \times em_3 \times \ldots$$

Where:
- $pm$ = person months
- $A$ = **2.94** (constant)
- $\text{size}$ = thousands of lines of code (KLOC)
- $sf$ = **scale factor**
- $em$ = **effort multipliers**

---

### Scale Factor (sf)

Based on **5 drivers** that are sensitive to system size:

| Driver | Name | Meaning |
|--------|------|---------|
| **PREC** | Precedentedness | Degree to which past examples exist to consult |
| **FLEX** | Development flexibility | Degree of flexibility when implementing |
| **RESL** | Architecture/risk resolution | Degree of uncertainty about requirements |
| **TEAM** | Team cohesion | Small tightly-knit team vs large dispersed team |
| **PMAT** | Process maturity | How structured and organised the development process is |

#### Scale Factor Rating Values

| Driver | Very Low | Low | Nominal | High | Very High | Extra High |
|--------|----------|-----|---------|------|-----------|------------|
| PREC | 6.20 | 4.96 | 3.72 | 2.48 | 1.24 | 0.00 |
| FLEX | 5.07 | 4.05 | 3.04 | 2.03 | 1.01 | 0.00 |
| RESL | 7.07 | 5.65 | 4.24 | 2.83 | 1.41 | 0.00 |
| TEAM | 5.48 | 4.38 | 3.29 | 2.19 | 1.10 | 0.00 |
| PMAT | 7.80 | 6.24 | 4.68 | 3.12 | 1.56 | 0.00 |

> Note: **Higher rating value = more difficult/uncertain** → increases estimated effort.
> A "Very High" PREC score means the team has a lot of past experience → **low value (1.24)** → easier project.

#### Scale Factor Formula

$$sf = 0.91 + 0.01 \times \sum \text{scale factor values}$$

---

### Effort Multipliers (em)

Six effort multipliers adjust the estimate for project-specific characteristics:

| Code | Name | Meaning |
|------|------|---------|
| **RCPX** | Product reliability and complexity | How complex and reliable must the software be? |
| **RUSE** | Reuse required | How much reusable code must be produced? |
| **PDIF** | Platform difficulty | How difficult is the target platform? |
| **PERS** | Personnel capability | How capable are the team members? |
| **PREX** | Personnel experience | How experienced is the team on this type of software? |
| **FCIL** | Facilities available | What development tools and facilities are available? |
| **SCED** | Schedule pressure | How compressed is the schedule? |

#### Effort Multiplier Values (selected)

| Driver | Extra Low | Very Low | Low | Nominal | High | Very High | Extra High |
|--------|-----------|----------|-----|---------|------|-----------|------------|
| RCPX | 0.49 | 0.60 | 0.83 | 1.00 | 1.33 | 1.91 | 2.72 |
| PDIF | — | — | 0.87 | 1.00 | 1.29 | 1.81 | 2.61 |
| PERS | 2.12 | 1.62 | 1.26 | 1.00 | 0.83 | 0.63 | 0.50 |
| PREX | 1.59 | 1.33 | 1.12 | 1.00 | 0.87 | 0.74 | 0.62 |
| FCIL | 1.43 | 1.30 | 1.10 | 1.00 | 0.87 | 0.73 | 0.62 |

> Note: For **PERS** and **PREX** — better capability/experience → **lower multiplier** → less effort needed.
> For **RCPX** and **PDIF** — more complexity/difficulty → **higher multiplier** → more effort needed.

---

### Complete COCOMO II Example

**Setup:**
- PREC = Very High → 1.24
- FLEX = Very Low → 5.07
- RESL = High → 2.83
- TEAM = High → 2.19
- PMAT = Low → 6.24

**Scale Factor:**
$$sf = 0.91 + 0.01 \times (1.24 + 5.07 + 2.83 + 2.19 + 6.24) = 0.91 + 0.1757 = 1.0857$$

**Unadjusted Effort** (10 KLOC):
$$pm = 2.94 \times 10^{1.0857} = 35.8 \text{ person-months}$$

**Effort Multipliers:**
- RCPX = Very High → 1.91
- PDIF = Very High → 1.81
- PERS = Extra High → 0.50
- PREX = Nominal → 1.00
- All others = Nominal → 1.00

**Adjusted Effort:**
$$pm_{\text{adjusted}} = 35.8 \times 1.91 \times 1.81 \times 0.50 = 61.9 \text{ person-months}$$

## 7. Comparison of All Estimating Methods

| Method | How | Best When | Weakness |
|--------|-----|-----------|---------|
| **Bottom-up** | Estimate each task, sum up | Detailed WBS exists; late-stage planning | Time-consuming |
| **Top-down** | Estimate whole, distribute proportions | Early stage; rough planning | Less accurate |
| **Expert judgment** | Ask an expert | Modifying existing code | Subjective; based on analogy |
| **Analogy** | Match to similar past project | Past project database exists | Quality depends on database |
| **Function Points** | Count functional components | Language-independent sizing | Requires trained counters |
| **COCOMO/COCOMO II** | Formula-based using LOC | When code size can be estimated | Sensitive to LOC accuracy |

---

## 8. Key Formulas Summary

| Formula | Description |
|---------|-------------|
| $UFP = \sum (\text{count} \times \text{multiplier})$ | Unadjusted Function Points |
| $TDI = \sum_{i=1}^{14} DI_i$ | Total Degree of Influence |
| $VAF = 0.65 + (0.01 \times TDI)$ | Value Adjustment Factor (range: 0.65–1.35) |
| $FP = UFP \times VAF$ | Adjusted Function Points |
| $\text{Effort (COCOMO81)} = c \times \text{size}^k$ | Basic COCOMO effort |
| $sf = 0.91 + 0.01 \times \sum \text{scale factor values}$ | COCOMO II scale factor |
| $pm = 2.94 \times \text{size}^{sf} \times \prod em_i$ | COCOMO II adjusted effort |

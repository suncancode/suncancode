# Domain Modelling: Complex Issues

---

## 1. The Problem: Many-to-Many Associations

Consider a **Course Enrolment** system at a university:

```
┌──────────────┐           ┌──────────────────┐          ┌─────────────┐
│    Course    │           │  CourseSection   │          │   Student   │
├──────────────┤           ├──────────────────┤          ├─────────────┤
│ courseNumber │  1   0..* │ sectionNumber    │  0..*  0..* │ studentID   │
│ title        ├───────────┤ startTime        ├──────────┤ name        │
│ creditHours  │           │ roomNumber       │          │ major       │
└──────────────┘           └──────────────────┘          └─────────────┘
```

**Problem:** A CourseSection has many Students, and a Student is registered in many CourseSections (many-to-many). But **where do we store the student's grade?**

- Grade is not an attribute of CourseSection (the section doesn't have one grade)
- Grade is not an attribute of Student (a student has different grades per section)
- Grade belongs to the **relationship itself**

**Solution → Association Class**

## 2. Association Class

An **association class** is an association that is treated as a class in a many-to-many relationship because it has attributes that need to be stored.

- It sits on the association line between two classes
- Connected to the association with a **dashed line**
- Its **key (identifier)** is the concatenation of the keys of both connected classes

### Example — CourseEnrollment Association Class

```
┌──────────────┐                              ┌─────────────┐
│    Course    │                              │   Student   │
├──────────────┤                              ├─────────────┤
│ courseNumber │                              │ studentID   │
│ title        │                              │ name        │
│ creditHours  │                              │ major       │
└──────────────┘                              └─────────────┘
       | 1                                           |
       |                                             |
       | 0..*      ╔══════════════════╗        0..*  |
       └───────────╢ CourseEnrollment ╟──────────────┘
                   ╠══════════════════╣
                   ║ grade            ║
                   ╚══════════════════╝
```

### Key Properties of Association Class

- The association class IS the association — it is not a separate entity added on top
- Its **unique key** = concatenation of the endpoint keys
  - Example: if CourseSection key = `CourseNumber + SectionNumber`, then CourseEnrollment key = `CourseNumber + SectionNumber + StudentID`
- If you need additional attributes to uniquely identify instances beyond the concatenated key → **the model is incorrect**

## 3. Generalization / Specialization (Inheritance)

A **hierarchical relationship** where subordinate classes are special types of a superior class. Also called an **inheritance hierarchy**.

### Key Terms

| Term | Definition |
|---|---|
| **Superclass** | The superior, more general class at the top of the hierarchy |
| **Subclass** | The subordinate, more specialized class that inherits from the superclass |
| **Inheritance** | Subclasses automatically inherit all attributes of their superclass |
| **Abstract class** | A class that allows subclasses to inherit characteristics but is **never instantiated** itself — shown in *italics* |
| **Concrete class** | A class that **can** have instances (objects) |

### UML Notation

- An **open triangle (△)** on the line points **toward the superclass**
- Subclasses branch off below

### Example 1 — Motor Vehicle Hierarchy

```
              MotorVehicle
                   △
       ┌───────────┼───────────┐
     Truck        Car       Tractor
                   △
       ┌───────────┼───────────┐
   SportsCar     Sedan    SportUtility
```

- Truck, Car, and Tractor are **special types** of MotorVehicle
- SportsCar, Sedan, and SportUtility are **special types** of Car

### Example 2 — Sale Hierarchy (RMO CSMS)

```
                    Sale  ← (abstract class, in italics)
              ┌─────────────────────┐
              │ saleDateTime        │
              │ priorityCode        │
              │ S&H                 │
              │ tax                 │
              │ totalAmt            │
              │ mountainBucks       │
              └─────────────────────┘
                        △
          ┌─────────────┼─────────────┐
    InStoreSale    OnlineSale    TelephoneSale
    ┌───────────┐  ┌──────────┐  ┌─────────────┐
    │ storeID   │  │timeOnSite│  │ clerkID     │
    │ registerID│  │ chatUse  │  │ lengthOfCall│
    │ clerkID   │  └──────────┘  └─────────────┘
    └───────────┘
```

**Inherited attributes of OnlineSale** (from Sale + its own):
`timeOnSite`, `chatUse`, `saleDateTime`, `priorityCode`, `S&H`, `tax`, `totalAmt`, `mountainBucks`

## 4. Whole-Part Relationships

A **whole-part relationship** exists when one class is a component of another class. There are two types:

### 4.1 Aggregation
- The **part can exist independently** from the whole
- The part can be **removed and replaced**
- UML: **open (hollow) diamond ◇** on the "whole" side

**Examples:**
- Computer **has** disk storage devices (storage devices exist apart from the computer)
- Car **has** wheels (wheels can be removed and still be wheels)

### 4.2 Composition
- The **part cannot exist without the whole**
- If the whole is destroyed, the parts are also destroyed
- UML: **filled (solid) diamond ◆** on the "whole" side

**Examples:**
- Order **has** OrderItems (without the Order, OrderItems don't exist)
- Chip **has** circuits (without a chip, circuits don't exist)

### Comparison Table

| Feature | Aggregation | Composition |
|---|---|---|
| Part exists independently? | ✅ Yes | ❌ No |
| Part survives whole's deletion? | ✅ Yes | ❌ No |
| UML symbol | ◇ Open diamond | ◆ Filled diamond |
| Example | Car has wheels | Order has OrderItems |

### Example — Computer and Its Parts (Composition)

```
              ┌──────────────┐
              │   Computer   │
              └──────────────┘
         ◆1..1 /    |    \ 1..1◆
        /           |           \
  1..*↓          1..1↓         1..*↓
Processor      MainMemory      Monitor
               InputDevice
               Storage
```

All parts (Processor, MainMemory, InputDevice, Storage, Monitor) are **parts of a Computer**.
Whole-part relationships can also have **multiplicity** notation.

## 5. Domain Modelling for RMO CSMS

### Approach

- Typically there is **one Domain Model Class Diagram (DMCD)** for each system
  - Provides an overview of the whole system
  - Includes rich and detailed information about all domain classes
- The RMO CSMS domain model is **complex** — it includes **27 domain classes**
- In practice, analysts can build **one DMCD per subsystem** first, then combine them into a system-wide diagram

## 6. Summary of Complex UML Relationships

| Concept | Description | UML Notation |
|---|---|---|
| **Association Class** | A many-to-many association that has its own attributes and is modelled as a class | Dashed line connecting class to association line |
| **Generalization / Specialization** | Hierarchical relationship where subclasses inherit from superclass | Open triangle △ pointing to superclass |
| **Abstract Class** | Superclass that is never instantiated; only defines shared attributes | Class name written in *italics* |
| **Concrete Class** | A class that can have actual instances | Normal class notation |
| **Aggregation** | Whole-part where part can exist independently | Open/hollow diamond ◇ on whole side |
| **Composition** | Whole-part where part cannot exist without the whole | Filled diamond ◆ on whole side |

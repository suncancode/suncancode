## 3. Brief Use Case Description

A **brief use case description** is a one-sentence overview showing the main steps in a use case. It captures what the actor does and what the system responds with.

### Examples — Customer Account Subsystem

| Use Case | Brief Description |
|---|---|
| Create customer account | User/actor enters new customer account data, and the system assigns account number, creates a customer record, and creates an account record. |
| Look up customer | User/actor enters customer account number, and the system retrieves and displays customer and account data. |
| Process account adjustment | User/actor enters order number, and the system retrieves customer and order data; actor enters adjustment amount, and the system creates a transaction record for the adjustment. |

## 4. Use Case Diagrams

A **Use Case Diagram** is a UML model used to graphically show use cases and their relationships to actors.

### Key Elements

| Element | Description |
|---|---|
| **Actor** | The UML name for an end user — represented as a stick figure |
| **Use Case** | A system function — represented as an oval/ellipse |
| **Connecting line** | Shows which actors participate in which use cases |
| **Automation boundary** | A rectangle representing the boundary between the computerized system and its users |

### Diagram Structure

```
[Actor (stick figure)]  ——————  [ Automation Boundary ]
                                 |   (Use Case oval)   |
                                 |   (Use Case oval)   |
                                 |_____________________|
```

---

## 5. Different Perspectives of Use Case Diagrams

The same system can be shown from different viewpoints depending on the audience:

- **All actors** — shows every use case and every actor in one diagram
- **Single actor** — shows only the use cases relevant to one specific actor (e.g., Customer only)
- **Multiple actors** — shows use cases shared between a subset of actors (e.g., Customer Service Representative and Store Sales Representative)

### Example — Sales Subsystem: Customer Actor Only

Use cases involving only the Customer:
- Search for item
- View product comments and ratings
- View accessory combinations
- Fill shopping cart
- Empty shopping cart
- Fill reserve cart
- Convert reserve cart
- Check out shopping cart
- Empty reserve cart

### Example — Sales Subsystem: Service Representative & Store Representative

Shared use cases:
- Search for item
- View product comments and ratings
- View accessory combinations
- Create phone sale *(Customer service representative only)*
- Create store sale *(Store sales representative only)*

## 6. The `<<includes>>` Relationship

The **`<<includes>>`** relationship is used when one use case is always called as part of another — similar to a subroutine in programming.

- The **arrow points to the included use case** (the "subroutine")
- The included use case is reusable across multiple parent use cases
- Shown as a **dashed arrow** labelled `«includes»`

### Example — Fill Shopping Cart includes:

```
Customer ——→ [Fill shopping cart]
                    |
          «includes»↓
              [Search for item]
                    |
          «includes»↓
       [View product comments and ratings]
                    |
          «includes»↓
       [View accessory combinations]
```

> Reading: When a customer fills their shopping cart, the system always includes searching for items, viewing product comments/ratings, and viewing accessory combinations.

## 7. Steps in Developing Use Case Diagrams

1. **Identify all stakeholders and users** who would benefit from seeing a use case diagram.
2. **Determine what each stakeholder needs to review** — per subsystem, per user type, or per specific use cases of interest.
3. **For each communication need**, select the relevant use cases and actors, then draw the diagram.
4. **Name each diagram carefully** and note how and when it should be used to review use cases with stakeholders and users.

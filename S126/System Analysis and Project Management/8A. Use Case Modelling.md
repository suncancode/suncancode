# Use Case Modelling 

---

## 1. Overview — Why We Need More Detail

Requirements reflect users' needs and specify business processes. Two primary aspects of functional requirements are:
- **Use Cases** — what the system does
- **Problem Domain Classes** — what the system remembers

A list of use cases and use case diagrams provide an **overview**, but additional details must be documented using:
- Fully-developed use case description
- Activity diagram
- System sequence diagram (SSD)

---

## 2. Use Case Descriptions — Three Levels of Detail

| Level | Description | When to Use |
|---|---|---|
| **Brief** | One-sentence overview | Simple use cases, early stages |
| **Intermediate** | More detail, stepping stone | Building toward fully-developed |
| **Fully-developed** | Complete formal template | Complex use cases, main documentation |

### Brief Use Case Description — Example
> *"Create customer account: User/actor enters new customer account data, and the system assigns account number, creates a customer record, and creates an account record."*

---

## 3. Fully-Developed Use Case Description

The **most formal method** for documenting a use case. Uses a structured template with the following fields:

| Field | Description |
|---|---|
| **Use case name** | Written as a verb-noun (e.g., "Create Customer Account") |
| **Scenario** | A special case or more specific path within the use case (if any) |
| **Triggering event** | What initiates the use case (based on event decomposition) |
| **Brief description** | The one-sentence description written earlier |
| **Actors** | One or more users identified from use case diagrams |
| **Related use cases** | Use cases linked via `<<includes>>` relationships |
| **Stakeholders** | Interested parties other than the specific actors |
| **Preconditions** | Conditions that must be true **before** the use case begins |
| **Postconditions** | What must be true upon **successful completion** of the use case |
| **Flow of activities** | Two-column table: Actor actions on the left, System responses on the right |
| **Exception conditions** | Alternative activities that don't belong to the main flow |

### Flow of Activities — Two-Column Format

| Actor | System |
|---|---|
| Actor enters new customer data | System validates data |
| | System assigns account number |
| | System creates customer record |
| | System creates account record |
| | System displays confirmation |

## 4. UML Activity Diagram

An **activity diagram** is a UML diagram that visually shows the **flow of activities** within a use case — who does what, in what order, and what decisions are made.

### Key Notation Elements

| Symbol | Meaning |
|---|---|
| Rounded rectangle | An activity (action step) |
| Diamond ◇ | Decision point (branch) |
| Thick horizontal bar | Synchronization bar — for loops or parallel activities |
| Arrow | Flow direction |
| Filled circle ● | Start point |
| Filled circle with ring ⊙ | End point |

### Key Features
- Shows **flow of activities only** (not data details)
- Can show use cases with `<<includes>>` relationships
- **Synchronization bar** used for loops
- **Decision diamond** used for conditional branches

### Examples
- Activity diagram for **Create Customer Account** — shows simple sequential flow
- Activity diagram for **Ship Items** — uses synchronization bar (loop) and decision diamond
- Activity diagram for **Fill Shopping Cart** — shows `<<includes>>` relationships with other use cases

## 5. System Sequence Diagram (SSD)

A **System Sequence Diagram** is a diagram showing the **sequence of messages** between an actor and the automated part of the system during a use case or scenario.

### Key Concepts
- Shows **what messages** are exchanged between actor and system
- Shows **order** of messages (top to bottom = time flows downward)
- Used to identify **system inputs and outputs**
- Complements the activity diagram (activity = what happens; SSD = what is communicated)

### SSD Message Notation

**General format:**
```
[true/false condition]  return-value := message-name (parameter-list)
```

| Symbol | Meaning |
|---|---|
| `*` (asterisk) | Repeating or looping message |
| `[ ]` (brackets) | True/false condition — message sent only if condition is true |
| `message-name` | Description of requested service — written as a verb-noun |
| `(parameter-list)` | Data passed with the message (with parentheses on initiating messages) |
| `:=` | Assigns a return value from the destination back to the source |
| Return-value | Data returned from system to actor in response to the message |

### SSD Frame Types

| Frame | Symbol | Meaning |
|---|---|---|
| **Loop frame** | `loop` | A set of messages that repeat |
| **Opt frame** | `opt` | Optional messages — only executed if condition is true |
| **Alt frame** | `alt` | If-else — one of two alternative paths is taken |

### Example SSD Structure
```
Actor                    :System
  |                          |
  |--- enterCustomerData() -->|
  |                          |--- validate data
  |                          |--- assign account number
  |<-- accountNumber ---------|
  |                          |
  |--- confirmAccount() ----->|
  |<-- confirmation ----------|
```

## 6. CRUD Technique

**CRUD** stands for **Create, Read/Report, Update, Delete** — a cross-check technique to ensure all domain classes are fully covered by use cases.

### Purpose
- A **validation tool**, NOT a primary identification technique
- Ensures that every domain class has use cases to handle all four operations
- Commonly used in database contexts
- Catches missing use cases that might be overlooked

### CRUD Steps

1. **Identify all domain classes**
2. **For each class, verify use cases exist to:**
   - **C**reate a new instance
   - **R**ead or report on information in the class
   - **U**pdate existing instances
   - **D**elete or archive inactive instances
3. **Add new use cases** as required — identify responsible stakeholders
4. **Identify which application** has responsibility for each action (which subsystem creates, updates, reads, or deletes)

### Example — CRUD Check for Customer Class

| Operation | Use Case |
|---|---|
| **Create** | Create customer account |
| **Read** | Look up customer; View customer account |
| **Update** | Update customer account |
| **Delete** | Archive inactive customer account |

## 7. Comparison of Use Case Modelling Tools

| Tool | Purpose | Key Strength |
|---|---|---|
| **Use case diagram** | Visual overview of all use cases and actors | Quick communication with stakeholders |
| **Brief use case description** | One-sentence summary | Fast, simple cases |
| **Fully-developed use case description** | Complete formal documentation | Most comprehensive, formal |
| **Activity diagram** | Visual flow of activities | Intuitive, shows decision and loop logic |
| **System sequence diagram (SSD)** | Messages between actor and system | Reveals inputs/outputs and system interface |
| **CRUD technique** | Validation of use case completeness | Cross-check — ensures nothing is missed |

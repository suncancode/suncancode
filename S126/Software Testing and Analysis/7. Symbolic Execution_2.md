# Symbolic Execution ‚Äî Part 2: Concolic Execution

---

## 1. Limitations of Pure Symbolic Execution

Despite its power, symbolic execution faces several fundamental challenges in practice:

### 1.1 Loops and Recursion ‚Äî Infinite Execution Tree

A loop that reads symbolic input can iterate an unbounded number of times, producing an **infinite execution tree** that symbolic execution can never fully explore.

```c
void testme_inf() {
    int sum = 0;
    int N = sym_input();
    while (N > 0) {
        sum = sum + N;
        N = sym_input();   // N is symbolic again each iteration!
    }
}
```

This loop can run 0, 1, 2, 3, ... times ‚Üí **infinitely many paths** ‚Üí symbolic execution never terminates.

### 1.2 Path Explosion ‚Äî Exponential Number of Paths

Every `if` statement forks the execution into 2 branches. With `n` nested conditionals, there are up to **2‚Åø paths**. Real programs have thousands of branches ‚Äî the number of paths grows faster than any tool can explore.

### 1.3 Heap Modeling ‚Äî Symbolic Pointers and Data Structures

When symbolic variables are **pointers or dynamic data structures** (linked lists, trees, graphs), modeling their behavior as logical constraints becomes extremely difficult or impossible.

### 1.4 SMT Solver Limitations ‚Äî Unsolvable Constraints

Some constraints that arise naturally in code are **beyond the reach of current SMT solvers**. For example:

```c
int foo(int v) { return (v * v) % 50; }
```

The constraint `(y‚ÇÄ √ó y‚ÇÄ) % 50 == x‚ÇÄ` involves non-linear arithmetic with modulo ‚Äî most solvers **cannot solve this**.

### 1.5 Environment Modeling ‚Äî External Interactions

When a program calls **system calls, external libraries, file I/O, or network operations**, symbolic execution has no model for what those external components do, causing it to get stuck or produce inaccurate results.

### 1.6 Coverage Problem ‚Äî Cannot Reach Deep Paths

When loops or deep branch trees are encountered, symbolic execution may get stuck exploring **shallow paths** and never reach deep parts of the execution tree where bugs may hide.

## 2. Two Existing Approaches and Their Tradeoffs

### Approach 1 ‚Äî Random Testing

Generate random inputs and execute the program on them.

```c
void test_me(int x) {
    if (x == 94389) ERROR;
}
```

- Probability of hitting ERROR = **1 / 2¬≥¬≤** ‚âà 0.00000002%
- Simple and fast, but **astronomically unlikely** to find rare bugs.

**Weakness**: Cannot reliably reach specific deep paths without extremely lucky input values.

### Approach 2 ‚Äî Pure Symbolic Execution

Use symbolic values for all inputs and solve path constraints.

```c
void test_me(int x) {
    if (bbox(x) != 17) ERROR;
    // bbox = opaque/blackbox function
}
```

- If `bbox` is a blackbox or involves non-linear operations, the symbolic constraint **cannot be solved**.
- **Does not scale** to large, complex programs.

**Weakness**: Gets stuck when constraints are too complex for the SMT solver.

---

## 3. Concolic Execution ‚Äî The Solution

### What is Concolic Execution?

**Concolic = Concrete + Symbolic**

Also called **Dynamic Symbolic Execution (DSE)**.

The key idea: execute the program **simultaneously** with both:
- A **concrete execution** using real input values (like normal testing)
- A **symbolic execution** tracking constraints alongside

When the symbolic engine gets stuck on an unsolvable constraint, it **falls back to the concrete value** ‚Äî substituting the real number in place of the symbolic expression ‚Äî and continues.

### Goal
Visit **deep into the program execution tree** by combining the practicality of random testing with the precision of symbolic execution.

### Implementations
- **SAGE** ‚Äî Microsoft Research (used to find hundreds of security bugs in Windows)
- **DART**, **CUTE** ‚Äî for C programs
- **jCUTE** ‚Äî for Java programs
- **CREST**, **PEX**, **EXE** (Stanford)

---

## 4. Concolic Execution ‚Äî 4 Steps

```
Step 1: Generate a random seed input to start execution
        e.g., x = 22, y = 7

Step 2: Run the program with BOTH concrete and symbolic execution simultaneously
        Collect the full path constraint
        e.g., (2b ‚âÝ a) AND (...)

Step 3: Negate the LAST conjunct of the path constraint
        e.g., (2b ‚âÝ a)  ‚Üí  negate last  ‚Üí  (2b == a)

Step 4: Solve the new constraint to get input for the next iteration
        Repeat from Step 2 with the new input
```

This strategy **systematically explores new branches** by flipping one decision at a time, always moving toward unexplored territory.

## 5. Concolic Execution ‚Äî Detailed Step-by-Step Example

### Program

```c
void testme(int x, int y) {
    int z = 2 * y;
    if (z == x) {
        if (x > y + 10)
            ERROR;
    }
}
```

---

### Iteration 1 ‚Äî Random seed: `x = 22, y = 7`

| Step | Concrete State | Symbolic State | Path Condition |
|------|---------------|---------------|----------------|
| Start | x=22, y=7 | x=a, y=b | ‚Äî |
| `z = 2*y` | z=14 | z=2b | ‚Äî |
| `if (z==x)` | 14‚âÝ22 ‚Üí FALSE | ‚Äî | `2b ‚âÝ a` |
| End | (terminate normally) | ‚Äî | `2b ‚âÝ a` |

**Path constraint collected:** `2b ‚âÝ a`

**Negate last conjunct:** `2b ‚âÝ a` ‚Üí `2b == a`

**Solve:** `2b == a` ‚Üí pick `b=1, a=2`

**Next input: `x = 2, y = 1`**

---

### Iteration 2 ‚Äî Input: `x = 2, y = 1`

| Step | Concrete State | Symbolic State | Path Condition |
|------|---------------|---------------|----------------|
| Start | x=2, y=1 | x=a, y=b | ‚Äî |
| `z = 2*y` | z=2 | z=2b | ‚Äî |
| `if (z==x)` | 2==2 ‚Üí TRUE | ‚Äî | `2b == a` |
| `if (x>y+10)` | 2>11? ‚Üí FALSE | ‚Äî | `2b==a AND a‚â§b+10` |
| End | (terminate normally) | ‚Äî | `2b==a AND a‚â§b+10` |

**Path constraint collected:** `2b==a AND a‚â§b+10`

**Negate last conjunct:** `a‚â§b+10` ‚Üí `a>b+10`

**Solve:** `2b==a AND a>b+10` ‚Üí substitute `a=2b`: `2b>b+10` ‚Üí `b>10` ‚Üí pick `b=15, a=30`

**Next input: `x = 30, y = 15`**

---

### Iteration 3 ‚Äî Input: `x = 30, y = 15`

| Step | Concrete State | Symbolic State | Path Condition |
|------|---------------|---------------|----------------|
| Start | x=30, y=15 | x=a, y=b | ‚Äî |
| `z = 2*y` | z=30 | z=2b | ‚Äî |
| `if (z==x)` | 30==30 ‚Üí TRUE | ‚Äî | `2b == a` |
| `if (x>y+10)` | 30>25 ‚Üí TRUE | ‚Äî | `2b==a AND a>b+10` |
| **ERROR** | **BUG TRIGGERED üêõ** | ‚Äî | ‚Äî |

**Bug found!** Concrete input `x=30, y=15` reproduces the error.

## 6. Key Innovation ‚Äî Concrete Fallback When Symbolic Gets Stuck

This is the most important advantage of Concolic over pure Symbolic Execution.

### Problem Scenario

```c
int foo(int v) { return (v * v) % 50; }

void testme(int x, int y) {
    int z = foo(y);
    if (z == x) {
        if (x > y + 10) ERROR;
    }
}
```

**Pure symbolic execution** computes `z = (y‚ÇÄ √ó y‚ÇÄ) % 50`, then needs to solve:

```
(y‚ÇÄ √ó y‚ÇÄ) % 50 == x‚ÇÄ
```

This is **non-linear modular arithmetic** ‚Äî the SMT solver cannot solve it. **Stuck.**

### Concolic Solution ‚Äî Use the Concrete Value

While symbolic execution is stuck, **concrete execution has already computed `z`**:

```
y = 7 (concrete)  ‚Üí  foo(7) = (7√ó7) % 50 = 49 % 50 = 49
```

**Replace symbolic `y‚ÇÄ` with its concrete value `7`** in the constraint:

```
(y‚ÇÄ √ó y‚ÇÄ) % 50 == x‚ÇÄ
        ‚Üì  substitute y‚ÇÄ = 7
      49 == x‚ÇÄ
```

Now the solver trivially finds: `x‚ÇÄ = 49`

**Next input: `x = 49, y = 7`**

### Result

Running with `x=49, y=7`:
- `z = foo(7) = 49`
- `if (49 == 49)` ‚Üí TRUE
- `if (49 > 7+10 = 17)` ‚Üí TRUE ‚Üí **ERROR triggered! üêõ**

> **Key insight**: By substituting concrete values when symbolic reasoning fails, concolic execution trades **some precision** (the solution is now tied to `y=7`) for **progress** (it can keep exploring instead of being stuck).

## 8. Finding Security and Safety Bugs with Concolic Testing

### Core Technique ‚Äî Automatic Checker Insertion

The concolic engine automatically **injects error checks** around potentially dangerous operations, then tries to find inputs that trigger them.

### Example ‚Äî Divide by Zero

```c
// Original code:
x = 3 / i;

// Concolic automatically transforms to:
if (i != 0)
    x = 3 / i;
else
    ERROR;    // ‚Üê concolic tries to find input where i == 0
```

### Example ‚Äî Buffer Overflow

```c
// Original code:
a[i] = 4;

// Concolic automatically transforms to:
if (0 <= i && i < a.length)
    a[i] = 4;
else
    ERROR;    // ‚Üê concolic tries to find input where i is out of bounds
```

The concolic engine then **systematically generates inputs** that satisfy `i == 0` or `i < 0 OR i >= a.length`, finding the exact values that trigger the bugs.

### Types of Bugs Detectable

- Divide by zero
- Buffer overflow / array out-of-bounds
- Integer overflow / underflow
- Null pointer dereference
- Assertion violations
- Security vulnerabilities (e.g., format string bugs)

## 9. Pen-and-Paper Exercise ‚Äî Full Concolic Walkthrough

```c
void hello(int x, int y) {
    int t = 0;
    if (x > y)    t = x*x - 3;
    else          t = y;
    if (t < x)    print("Hello World");
}
```

**Start with random seed: `x = 0, y = 0`**

---

### Iteration 1 ‚Äî `x=0, y=0`

| Step | Concrete | Path Condition |
|------|---------|----------------|
| `if (x>y)` | 0>0? FALSE | `a ‚â§ b` |
| `t = y` | t=0 | ‚Äî |
| `if (t<x)` | 0<0? FALSE | `a‚â§b AND b‚â§a` ‚Üí i.e., `a==b` |
| End | (no print) | `a‚â§b AND a‚â•b` |

**Negate last:** `a‚â•b` (i.e., `t‚â•x`) ‚Üí `a<b` ... wait, we're in the `else` branch so `t=b`. We need `b<a`. But `a‚â§b` ‚Äî **UNSAT**. Skip.

**Negate first conjunct instead:** `a‚â§b` ‚Üí `a>b`

**Solve:** `a>b` ‚Üí pick `a=1, b=0`

**Next input: `x=1, y=0`**

---

### Iteration 2 ‚Äî `x=1, y=0`

| Step | Concrete | Path Condition |
|------|---------|----------------|
| `if (x>y)` | 1>0? TRUE | `a > b` |
| `t = x*x - 3` | t = 1-3 = -2 | t = a¬≤-3 |
| `if (t<x)` | -2<1? TRUE | `a>b AND a¬≤-3 < a` |
| **print("Hello World")** | ‚úÖ **Printed!** | ‚Äî |

**Input `x=1, y=0` triggers `print("Hello World")`.**

---

### Summary of Iterations

| Iteration | Input | Branch 1 | Branch 2 | Result |
|-----------|-------|----------|----------|--------|
| 1 | x=0, y=0 | FALSE (else) | FALSE | No print |
| 2 | x=1, y=0 | TRUE (if) | TRUE | **Print!** ‚úÖ |

## 11. Summary

| Concept | One-line Definition |
|---------|-------------------|
| **Concolic Execution** | Run program simultaneously with concrete + symbolic inputs |
| **Random Seed** | Initial concrete input to start concolic exploration |
| **Negate Last Conjunct** | Flip the last branch decision to explore a new path |
| **Concrete Fallback** | When solver is stuck, substitute concrete value for symbolic variable |
| **Dedicated Checker** | Automatically inserted error condition to detect specific bug types |
| **Dynamic Symbolic Execution** | Another name for concolic execution (execution-driven) |
| **False Positive** | A reported bug that is not a real bug ‚Äî concolic has none since it uses real inputs |
| **Path Explosion** | Exponential growth of paths; concolic mitigates but does not fully solve this |

### Three Core Takeaways

1. **Pure symbolic execution fails in practice** due to path explosion, infinite loops, unsolvable constraints, and environment complexity.

2. **Concolic execution solves the "stuck" problem**: when the SMT solver cannot solve a symbolic constraint, the concrete value (from real execution running in parallel) is substituted in ‚Äî sacrificing some generality but maintaining progress.

3. **Concolic = systematic + practical**: it retains the high coverage and bug-finding power of symbolic execution while being grounded in real program execution, making it applicable to real-world, complex programs.

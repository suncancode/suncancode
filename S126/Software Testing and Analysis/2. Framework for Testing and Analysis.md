# A Framework for Testing and Analysis

---

## 1. Learning Objectives

- Introduce dimensions and tradeoffs between test and analysis activities
- Distinguish **validation** from **verification** activities
- Understand the **limitations and possibilities** of test and analysis

---

## 2. Verification vs. Validation

| | Validation | Verification |
|---|---|---|
| **Question** | Are we building the **right** software? | Are we building the software **right**? |
| **Checks against** | User's real needs | Requirements specifications |
| **Activities** | Usability testing, user feedback | Testing, inspections, static analysis |

### Diagram
```
Actual Needs ──► [Validation] ◄──► SW Specs ──► [Verification] ◄──► System
```

---

## 3. Verifiable vs. Unverifiable Specifications

A specification must be **precise and measurable** to be verifiable.

| Type | Example |
|---|---|
| ❌ **Unverifiable** (but validatable) | "...an elevator must arrive at floor i *soon*..." |
| ✅ **Verifiable** | "...an elevator must arrive at floor i *within 30 seconds*..." |

--- 

## 4. The Undecidability Problem

> **"You can't always get what you want"**

Correctness properties are **undecidable** - the halting problem can be embedded in almost every property of interest.

```
Property ──┐
           ├──► Decision Procedure ──► Pass / Fail
Program ───┘
```

This means: **no automated tool can perfectly verify all properties of all program**.

---

## 5. Dealing with Undecidability: Tradeoffs

Since perfect verification is impossible, we make tradeoffs:

### The Tradeoffs Space

- **Perfect verification**: Infinite effort, not practical (Exhausting testing of all inputs).
- **Model checking**: Decidable but possibly intractable, checks simple temporal properies.
- **Theory proving**: Unbounded effort, verifies general properties.
- **Data flow analysis**: Precise analysis of simple syntactic properties.
- **Typical testing**: Practical but imcomplete.

### Types of Inaccuracy

| Type | Meaning | Risk | Used in |
|---|---|---|---|
| **Optimistic inaccuracy** | May accept programs that do NOT have the property | Misses bugs | Testing |
| **Pessimistic inaccuracy** | May reject programs that DO have the property | False alarms | Static analysis |
| **Simplified properties** | Reduce freedom to make the check tractable | Limited scope | Many tools |

---

## 6. Example: Simplified Property — Semaphore Operations

**Original (hard) problem:** `lock(S)` and `unlock(S)` can appear anywhere in control flow — static matching is necessarily inaccurate.

```c
if ( ... ) {
    lock(S);
}
...
if ( ... ) {
    unlock(S);
}
```

**Simplified property (Java `synchronized`):** Always locks on entry, always unlocks on exit — statically checkable.

```java
synchronized(S) {
    ...
}
```

> By restricting flexibility, we gain the ability to **statically verify** the property.

---

## 7. Key Terminology

- **Property**: A statement or condition about a program's behavior that we want to be true.
- **Formula**: A formal specification that a program processes a property under consideration.
- **Safe**: An analysis with no optimistic inaccuracy - accepts only correct program.
- **Sound**: Returns `true` only when the program actually satisfies the formula (no optimistic inaccuracy -> always safe).
- **Complete**: Returns `true` whenever the program satisfies the formula (only optimistic inaccuracy is admitted).

```
Sound   → No false negatives (won't miss real errors) → may have false positives
Complete → No false positives (won't raise false alarms) → may miss real errors
```

---

## 8. Summary

1. **Most interesting properties are undecidable** — we cannot rely on fully automated tools without human intervention.
2. **Two complementary activities:**
   - *Validation*: Does the software do what it's supposed to do?
   - *Verification*: Does the system behave as specified?
3. **No single technique fits all purposes** — test designers must select a suitable combination of techniques.

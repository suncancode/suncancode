## 3. Sensitivity
> If a fault exists, the system should **always** expose it - not just occasionally.

### Key Ideas
- A test criterion is reliable if every test suite under that criterion gives the **same result**
- A runtime analysis is better if it is **machine-independent**

### Example: `stringCopy`

```c
// BAD: strcpy — fault, but may NOT fail
strcpy(middle, "Muddled");

// BETTER: strncpy — fault, may not fail
strncpy(middle, "Muddled", sizeof(middle));

// BEST: stringCopy (sensitive version) — GUARANTEED to fail
void stringCopy(char *target, const char *source, int howBig) {
    assert(strlen(source) < howBig);  // always catches the fault
    strcpy(target, source);
}
```

- `strcpy` → silent buffer overflow, undefined behavior
- `strncpy` → truncates silently, may mask the fault
- `stringCopy` → asserts the precondition, **always fails visibly** if violated

> Java's runtime array bounds checking is another example of sensitivity applied at the language level.

---

## 4. Redundancy

> Having **multiple overlapping checks** increases the chance of catching faults early.

### Examples

| Redundant Pair | Benefit |
|---|---|
| Static type checking + Dynamic type checking | Static catches mismatches earlier and cheaper |
| Validation of requirements + Validation of final software | Catches errors before costly implementation |
| Testing + Formal proof of properties | Used together to increase overall confidence |

> Redundancy is not waste — it is deliberate overlap to improve fault detection coverage.

---

## 5. Restriction 

> Reduce a hard (unsolvable) problem to a simpler (solvable) one by **restricting flexibility**.

### Examples

| Hard Problem | Restriction Applied | Result |
|---|---|---|
| Proving pointers used correctly (general) | Java requires pointers initialized before use | Statically enforceable |
| Proving no type errors at runtime (dynamic typing, e.g. Python) | Statically typed languages (C, Java) impose strict rules | Easily checkable at compile time |
| Proving lock/unlock match (arbitrary placement) | Java `synchronized` block — always locks on entry, unlocks on exit | Statically verifiable |

> Trade-off: you lose **flexibility**, but gain **verifiability**.

## 6. Partition 

> Break a complex problem into smaller, manageable sub-problems that can be reasoned about independently.

### 2 Types of Partitioning

** 1. Process Partitioning**
- Divide a complex activity into smaller, uniform sub-activities
- Example: unit testing -> integration testing -> system testing

** 2. Input Space Partitioning**
- Failures are **sparse** in the full input space, but **dense** in specific regions
- Strategy: identify partitions, then sample at least one test case from each partition
- This ensures the "dense failure regions" are covered

```
Full Input Space:
[ sparse ] [ sparse ] [ DENSE FAILURES ] [ sparse ] [ DENSE FAILURES ]
                            ↑ sample here                   ↑ sample here
```

> Both structural (code-based) and functional (spec-based) test criteria use partitioning to drive test case selection.

## 7. Visibility

> The ability to **measure progess or status** against goals at every stage of development.

### Key Concepts
- **Schedule visibility**: "Are we ahead or behind schedule?"
- **Quality visibility**: "Does quality meet our objectives?"
- Requires setting **assessable goals** at each development stage
- The biggest challenge: **early assessment** (evaluating specs and design before code exists)

### Proxy Measures
When true measures aren't available yet, use **proxies**:
- True meansure: reliability of final product
- Proxy: number of faults found in design inspections -> early indicator of potential quality problem

### Practical Example
> Choosing a **simple or standard internal data format** makes it easy to inspect data flow during unit testing — this is an act of designing for visibility.

---

## 8. Feedback 

> **Learn from experience** — each project provides information to improve the next.

### Examples
- **Checklists** are built from errors revealed in past projects
- **Error taxonomies** help build better test selection criteria
- **Design guidelines** help avoid recurring pitfalls
- **Software Analytics** — systematic collection and analysis of project data to drive improvement

> See: [Software Analytics](https://seanalytics.github.io/)

## 9. Test and Analysis Activities in the Software Process

### Core Insight: Quality is Not an Afterthought

- Qualities **cannot be added after development** — they result from inter-dependent activities throughout the lifecycle
- Testing is **not a phase** — it is a lifecycle activity from requirements through delivery

### Cost of Defects (Rule of Thumb)

| Phase Defect Found | Relative Cost to Fix |
|---|---|
| Design | **1x** |
| Implementation | **6.5x** |
| Testing | **15x** |
| Maintenance | **100x** |

> Fix bugs early. The later a defect is found, the exponentially more expensive it is to fix.

## 10. Properties of the Quality Process

| Property | Definition |
|---|---|
| **Completeness** | Appropriate activities are planned to detect each important class of faults |
| **Timeliness** | Faults are detected as early as possible (high leverage point) |
| **Cost-effectiveness** | Activities are chosen based on cost AND effectiveness over the whole lifecycle |

## 11. Quality Goals Taxonomy

```
Quality Software
├── Process Qualities (visibility, feedback, ...)
└── Product Qualities
    ├── Internal Qualities
    │   └── maintainability, reusability, traceability, ...
    └── External Qualities
        ├── Usefulness
        │   └── usability, performance, security, portability, interoperability
        └── Dependability
            └── correctness, reliability, safety, robustness
```

### Dependability Qualities Explained

| Quality | Definition | Example |
|---|---|---|
| **Correctness** | Program is consistent with its specification | Traffic light follows correct scheduling pattern |
| **Reliability** | Likelihood of correct function over time (statistical approximation of correctness) | Fails rarely under normal conditions |
| **Safety** | Prevents hazardous states | Traffic light never shows conflicting greens |
| **Robustness** | Acceptable (degraded) behavior under extreme conditions | Blinking red/yellow is better than no signal |

### Relationship Between Dependability Qualities

```
Correct ⊂ Reliable ⊂ Safe ⊂ Robust   (roughly, from strictest to most lenient)

- Correct but not safe/robust → specification is inadequate
- Reliable but not correct   → failures occur rarely (but spec may be wrong)
- Safe but not correct       → annoying (non-hazardous) failures can occur
- Robust but not safe        → catastrophic failures can still occur
```

## 12. A&T Techniques Overview

### Inspection (Manual)
- Applicable to **any document**: requirements, design, test plans, source code
- Secondary benefits: spreads good practices, instills shared standards
- Expensive: takes time, re-inspection after changes is costly
- Used where automated techniques are inapplicable or insufficient

### Automatic Static Analysis
- Applied to **formal representations** (not natural language)
- Substitutes machine cycles for human effort → very cost-effective when applicable
- More limited scope than inspection

### Testing
- Executed (and best **started early**) in the development cycle
- Early test generation advantages:
  - Tests written independently from code
  - Highlights inconsistencies in specs
  - Tests serve as a compendium of specs for developers

## 13. Organizational Factors

### Team Structure Options

| Structure | Pros | Cons |
|---|---|---|
| Separate dev & QA teams | Clear responsibility, independence | Potential adversarial dynamic |
| Unified team (e.g., XP) | Shared ownership of quality | Risk of skipping testing under pressure |

### Responsibility Allocation Anti-Patterns

**Case A** — Dev team measured by LOC/month, QA team responsible for quality
- Result: Dev maximizes code output without regard for quality → QA overwhelmed → project failure

**Case B** — Dev team responsible for both dev and quality
- Result: Testing delayed in favor of feature delivery → product shipped undertested → project failure

### Recommended Allocation
- **Unit testing** → dev team (needs code knowledge), QA monitors coverage
- **Integration/System/Acceptance testing** → QA team, dev provides scaffolding
- **Inspection/Walkthroughs** → mixed teams
- **Regression testing** → QA and maintenance teams
- **Process improvement** → external specialists

---

## 14. Summary

1. **6 principles** guide A&T: Sensitivity, Redundancy, Restriction, Partition, Visibility, Feedback
2. **Quality must be built in** from the start — not added at the end
3. A good quality process is **visible, timely, and cost-effective**
4. Goals include: reducing faults, assessing dependability before delivery, and continuously improving the process

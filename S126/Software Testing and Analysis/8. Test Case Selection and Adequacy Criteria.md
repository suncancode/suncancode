# Test Case Selection and Adequacy Criteria

---

## 1. The Core Problem — What is "Thorough Testing"?

When we say a software module has been "thoroughly tested", what does that actually mean?

Two key questions:
- **What do we mean by thorough testing?**
- **What criterion can we use to judge whether a test suite is adequate?**

This is not a trivial question. A suite of 10 well-chosen tests may be far more effective than 1000 random tests.

---

## 2. Test Adequacy — What We Want vs What We Can Get

### What we WANT
An adequacy criterion that guarantees:
> *"If the system passes an adequate suite of test cases, then it must be correct."*

### Why we CAN'T have it
This is **provably undecidable** — no algorithm can determine, for all programs and all test suites, whether passing the tests guarantees correctness.

### What we SETTLE FOR
Instead of proving adequacy, we use **proxies** — practical rules that help us **identify inadequacies** in a test suite.

> **Key mindset shift**: We do not prove a test suite is adequate. We use criteria to **expose when it is NOT adequate**.

---

## 3. Important Terminology

| Term | Definition |
|------|-----------|
| **Test case** | A set of inputs, execution conditions, and a pass/fail criterion |
| **Test case specification** | A requirement to be satisfied by one or more test cases |
| **Test obligation** | A pattern for test case specification — derived from an adequacy criterion — requiring some property deemed important to thorough testing |
| **Test suite** | A set of test cases |
| **Test execution** | The activity of running test cases and evaluating results |
| **Adequacy criterion** | A predicate that is either true (satisfied) or false (not satisfied) for a ⟨program, test suite⟩ pair |

### Example — Distinguishing the Terms

**Program**: sorts a sequence of words.

- **Test case specification**: "The input is two or more words"  
  → This is a *requirement* describing what kind of input to use.

- **Test case**: `input = ["banana", "apple", "cherry"]`, expected output = `["apple", "banana", "cherry"]`  
  → This is a *concrete* input + expected output + pass/fail rule.

- **Test obligation**: "Every sorting function must be tested with an input that contains duplicate words"  
  → This is a *property* we deem important to test, derived from an adequacy criterion.

---

## 4. Test Case Specification in Detail

A test case specification is **not just about inputs**. A complete specification includes:

- **Input specifications**: data names, ordering, values, states, timing
- **Output specifications**: expected data names, ordering, values, states, timing
- **Environmental needs**: hardware, software, other infrastructure
- **Special procedural requirements**: e.g., "remove system power at step 3"
- **Inter-case dependencies**: e.g., "this test must run after test #5"

### Example — Database System

The specification requires *robust failure recovery in case of power loss*.

A test case specification derived from this:
- **Input**: A database transaction in progress
- **Procedure**: Remove system power at a critical point mid-transaction
- **Pass/fail criterion**: System recovers to a consistent state on restart

This shows that test specifications can come from **behavior requirements**, not just input/output descriptions.

### IEEE 829 Template (Standard Format)

```
Test Case Specification Identifier: TC-001
Test Items: Sorting function — stability under mixed case
Input Specifications:
  - Values: ["Banana", "apple", "Cherry", "apple"]
  - Ordering: unsorted
Output Specifications:
  - Values: ["apple", "apple", "Banana", "Cherry"] (case-insensitive sort)
Environmental Needs:
  - JVM 17+
Special Procedural Requirements: None
Inter-Case Dependencies: None
```

## 5. Where Do Test Obligations Come From?

Test obligations can be derived from four sources:

### 5.1 Functional (Black-Box / Specification-Based)
Derived from **software specifications**, without looking at the code.

> Example: If the spec requires robust recovery from power failure → test obligation = simulate power failure during processing.

### 5.2 Structural (White-Box / Glass-Box)
Derived from **the source code itself**.

> Example: Traverse each program loop one or more times.  
> Example: Every `if` branch must be executed at least once.

### 5.3 Model-Based
Derived from **models** of the system (state machines, UML diagrams, etc.).

> Example: Exercise all transitions in a communication protocol state machine model.

### 5.4 Fault-Based
Derived from **known common bugs** — hypothesize what could go wrong and test for it.

> Example: Test with very large inputs to check for buffer overflow (a common vulnerability).

## 6. White-Box Test Obligations — Coverage Criteria

White-box testing uses the **structure of the code** to derive test obligations. Listed from weakest to strongest:

### 6.1 Statement Coverage
Every **executable statement** must be executed at least once.

```
Obligation: for each statement S in program P,
            at least one test case must execute S.
```

### 6.2 Decision (Branch) Coverage
Every **branch outcome** of every decision must be exercised at least once (both TRUE and FALSE for every `if`).

```
Obligation: for each decision point D,
            at least one test must make D evaluate to TRUE,
            and at least one must make D evaluate to FALSE.
```

### 6.3 Condition Coverage
Every **individual condition** within a compound decision must take both TRUE and FALSE values at least once.

```
Decision: if (age < 25 && gender == male && !married)
Conditions: (age < 25), (gender == male), (!married)
Obligation: each individual condition must be TRUE at least once
            AND FALSE at least once.
```

### 6.4 Path Coverage
Every **possible combination** of condition outcomes through the program must be exercised.

```
Obligation: all possible paths from entry to exit
            must be executed at least once.
```

> Note: Path coverage is the strongest but often **infeasible** for programs with loops (infinite paths).

## 7. Concrete Example — Insurance Premium Program

```java
premium = 500;
if (age < 25 && gender == male && !married) {
    premium += 500;
} else {
    if (married || gender == female)
        premium -= 200;
    if (age > 45 && age < 65)
        premium -= 100;
}
```

### Identifying Branches

```
Decision 1: (age < 25 && gender == male && !married)  → TRUE / FALSE
Decision 2: (married || gender == female)              → TRUE / FALSE
Decision 3: (age > 45 && age < 65)                    → TRUE / FALSE
```

### Test Suite for Statement Coverage (minimum)

We need every statement executed at least once — **2 test cases** suffice:

| Test | age | gender | married | Path | premium |
|------|-----|--------|---------|------|---------|
| T1 | 20 | male | false | TRUE branch (D1=T) | 500+500 = **1000** |
| T2 | 50 | female | true | FALSE branch (D1=F, D2=T, D3=T) | 500-200-100 = **200** |

T1 covers the `if` body. T2 covers the `else` body including both inner `if`s.

### Test Suite for Branch Coverage

We need every decision TRUE and FALSE at least once — **3 test cases**:

| Test | age | gender | married | D1 | D2 | D3 | premium |
|------|-----|--------|---------|----|----|-----|---------|
| T1 | 20 | male | false | T | — | — | **1000** |
| T2 | 50 | female | true | F | T | T | **200** |
| T3 | 30 | male | false | F | F | F | **500** |

T3 is needed to cover D2=FALSE and D3=FALSE (a 35-year-old single male doesn't trigger either inner `if`).

### Test Suite for Path Coverage

All combinations of D1, D2, D3 — note D2 and D3 only matter when D1=FALSE:

| Test | age | gender | married | D1 | D2 | D3 | premium |
|------|-----|--------|---------|----|----|-----|---------|
| T1 | 20 | male | false | T | — | — | 1000 |
| T2 | 50 | female | true | F | T | T | 200 |
| T3 | 30 | female | true | F | T | F | 300 |
| T4 | 55 | male | false | F | F | T | 400 |
| T5 | 30 | male | false | F | F | F | 500 |

## 8. Adequacy Criterion — Formal Definition

> **Adequacy criterion** = a set of test obligations.

A test suite **S** satisfies an adequacy criterion **C** for program **P** if and only if:
1. **All tests pass** (no failures), AND
2. **Every test obligation** in C is fulfilled by at least one test case in S.

### Example

**Question**: How does test suite S satisfy the *statement coverage* adequacy criterion for program P?

**Answer**: Every executable statement in P is executed by at least one test case in S, AND every test execution resulted in "pass".

---

## 9. Coping with Unsatisfiable Obligations

Sometimes **no test suite can satisfy a criterion** for a given program — for example, dead code (unreachable statements) can never be covered.

### Approach A — Exclude Unsatisfiable Obligations
Modify the criterion to only require execution of **reachable** statements.

> Problem: We cannot always determine which statements are reachable (this is undecidable in general).

### Approach B — Measure Partial Satisfaction (Coverage %)
Instead of pass/fail, measure the **fraction** of obligations satisfied.

> Example: If a test suite satisfies 85 out of 100 obligations → **85% coverage**.

### Important Distinction

| Term | Meaning |
|------|---------|
| **Adequacy criterion** | Binary: satisfied or not satisfied |
| **Coverage measure** | Fractional: percentage of satisfied obligations |

---

## 10. Is Coverage Useful or Dangerous?

### Coverage IS useful when:
- Tracking **progress** toward a thorough test suite
- Identifying **trouble spots** that need more test attention
- Comparing test suites systematically

### Coverage IS dangerous when:
- Treated as a **goal in itself** rather than a proxy for thoroughness
- It's easy to **inflate coverage without improving quality** — writing trivial tests that execute code without meaningfully checking behavior
- The only measure that truly matters is **cost-effectiveness** — finding real bugs efficiently

> **Warning**: 100% statement coverage does NOT mean the program is correct. It only means every line was executed — not that every line was executed correctly under all meaningful conditions.

## 11. Comparing Criteria — The Subsumes Relation

### Motivation
Different adequacy criteria have different strengths. We need a formal way to compare them.

### Definition

> **Test adequacy criterion A subsumes criterion B** if and only if, for every program P, every test suite that satisfies A with respect to P also satisfies B with respect to P.

Written as: **A ⊇ B** (A subsumes B, A is stronger than B)

### Intuition
If A subsumes B, then satisfying A **automatically** satisfies B. There is no need to check B separately once A is satisfied.

### Key Example

```
Branch coverage  SUBSUMES  Statement coverage
```

**Why?** If every branch (TRUE and FALSE) of every decision is exercised, then every statement has been reached — because every statement lies on some branch. The converse is NOT true: you can execute every statement without exercising both outcomes of every decision.

```
Path coverage  SUBSUMES  Branch coverage  SUBSUMES  Statement coverage
```

Strongest ←————————————————————→ Weakest

### Another Example — Specification-Based

A criterion requiring **all combinations** of values for attributes A and B subsumes a criterion requiring all values of A independently and all values of B independently.

```
All combinations of A×B  SUBSUMES  (All values of A  AND  All values of B separately)
```

### Practical Implication

If criterion C1 subsumes C2, and your test suite satisfies C1, then:
- You **automatically** satisfy C2 — no extra work needed.
- There is **no additional value** in measuring C2 separately.

---

## 12. Quiz — Subsumes and Fault Revelation

**Question**: Suppose test suite A satisfies adequacy criterion C1. Test suite B satisfies criterion C2, and C2 subsumes C1. Can we be certain that faults revealed by A will also be revealed by B?

**Answer**: **No, we cannot be certain.**

Here is why:

- C2 subsumes C1 means: every suite satisfying C2 also satisfies C1 (in terms of coverage obligations).
- But **many different test suites** can satisfy the same criterion.
- Suite B satisfies C2, but it may use **different concrete inputs** than suite A.
- Different inputs may hit different code paths, revealing different faults.

**Example**:
- Suite A (satisfies C1 = statement coverage): tests `x=5` and finds a bug triggered by `x=5`.
- Suite B (satisfies C2 = branch coverage, which subsumes C1): tests `x=10` and `x=-1` — covers all branches but never tests `x=5` → **misses the bug A found**.

> **Key lesson**: Subsumption is about **structural coverage guarantees**, not about which specific faults are found. A stronger criterion gives broader structural coverage but does not guarantee it finds every fault a weaker suite might accidentally catch.

## 13. Uses of Adequacy Criteria in Practice

### 13.1 Test Selection — Guiding Test Suite Design
Use criteria **proactively** while writing tests.

> Example: A specification-based criterion suggests test cases covering representative combinations of input values — guiding the tester toward meaningful coverage.

### 13.2 Revealing Missing Tests — Post-Hoc Analysis
Use criteria **retroactively** after writing tests to find gaps.

> Example: After writing tests from the specification, apply structural branch coverage to discover branches that no test exercises — revealing untested logic.

### 13.3 Combined Approach (Best Practice)

```
Step 1: Design test suite from specifications (functional/black-box)
Step 2: Measure structural coverage (e.g., branch coverage)
Step 3: Identify uncovered branches → design additional tests
Step 4: Repeat until coverage target is met
```

This combines the semantic richness of specification-based testing with the structural thoroughness of code-based criteria.

## 14. Summary

| Concept | One-line Definition |
|---------|-------------------|
| **Test adequacy** | A measure of how thoroughly a test suite exercises a program |
| **Test obligation** | A specific property that at least one test case must satisfy |
| **Adequacy criterion** | A set of test obligations; satisfied when all obligations are met and all tests pass |
| **Coverage measure** | Fraction of obligations satisfied (0%–100%) |
| **Statement coverage** | Every executable statement executed at least once |
| **Branch coverage** | Every decision outcome (T/F) exercised at least once |
| **Condition coverage** | Every individual condition in a decision exercised T and F |
| **Path coverage** | Every path through the program exercised |
| **Subsumption (A ⊇ B)** | Every suite satisfying A also satisfies B; A is strictly stronger |
| **Unsatisfiable obligation** | An obligation no test suite can fulfill (e.g., dead code) |

### Subsumption Hierarchy (Strongest → Weakest)

```
Path Coverage
      ⊇
Branch / Decision Coverage
      ⊇
Statement Coverage
```

### Three Core Takeaways

1. **We cannot prove test adequacy** — it is undecidable. We instead use criteria to identify *inadequacies* and guide improvement.

2. **Coverage is a proxy, not a goal** — high coverage is evidence of thoroughness but does not guarantee correctness. Trivial tests can inflate coverage without catching real bugs.

3. **Subsumption lets us compare criteria rigorously** — if A subsumes B, satisfying A makes B redundant. In practice, combine black-box (specification) criteria with white-box (structural) criteria for the best results.

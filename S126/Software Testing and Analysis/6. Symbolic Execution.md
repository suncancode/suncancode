# Symbolic Execution

---

## 1. Limitations of Traditional Testing

- Testing fits well with developer intuitions and is the most common form of bug detection in practice.
- However, **each test explores only one possible execution** of the system.
- The quality of testing depends entirely on the quality of test cases or inputs written by humans.
- Provides **little in terms of path coverage** — important paths and edge cases are easily missed.

**The key question**: Can we automatically generate test inputs that achieve high coverage?  
**Answer**: Yes — using **Symbolic Execution**.

---

## 2. What is Symbolic Execution?

### Core Idea
Instead of running a program with **concrete values** (e.g., `x = 5, y = 3`), symbolic execution runs the program with **symbolic variables** (e.g., `x = a, y = b`) that represent all possible values simultaneously.

### What it Tracks
- **Symbolic state**: a mapping from each variable to a symbolic expression
- **Path condition (path constraint)**: a quantifier-free formula over symbolic inputs that encodes all branch decisions taken so far along the current execution path

### What it Builds
- Predicates characterizing the **conditions for executing each path**
- Predicates characterizing the **effects of execution on program state**

### Applications
- Automatic **test data generation**
- **Program analysis** (e.g., reachability, dead code detection)
- **Formal verification** — proving program correctness for all inputs

### Implementations
- **KLEE** (C programs, LLVM-based)
- **Java PathFinder** (Java programs)
- **angr**, **SAGE**, **Triton** (binary-level)

## 3. How Symbolic Execution Works — Step by Step

### Example Program

```c
void func(int x, int y) {
    int z = 2 * y;
    if (z == x) {
        if (x > y + 10)
            ERROR;   // bug here
    }
}

int main() {
    int x = sym_input();   // x = a (symbolic)
    int y = sym_input();   // y = b (symbolic)
    func(x, y);
}
```

### Step 1 — Mark inputs as symbolic
```
x = a,  y = b      (a and b are symbolic — they represent ALL possible values)
```

### Step 2 — Execute statements symbolically

```
int z = 2 * y  →  z = 2b
```

Symbolic state:
```
x = a,  y = b,  z = 2b
```

### Step 3 — Hit a branch: FORK

At `if (z == x)` → `if (2b == a)`, the engine does **not** pick one branch.  
Instead it **forks** into two parallel branches:

```
                  [if 2b == a ?]
                 /               \
            2b ≠ a             2b == a
           (FALSE)              (TRUE)
               |                   |
           terminate          [if a > b+10 ?]
                              /              \
                       2b==a &&          2b==a &&
                       a ≤ b+10          a > b+10
                      (FALSE)             (TRUE)
                          |                  |
                      terminate           ERROR 🐛
```

### Step 4 — Collect path constraints

Each leaf of the execution tree has a **path constraint** — a logical formula describing exactly which inputs lead to that path:

| Path | Path Constraint | Description |
|------|----------------|-------------|
| 1 | `2b ≠ a` | z ≠ x, skips outer if |
| 2 | `2b == a AND a ≤ b+10` | z == x, inner if false |
| 3 | `2b == a AND a > b+10` | z == x, inner if true → **BUG** |

### Step 5 — Solve each constraint with an SMT Solver

| Constraint | Solver Output | Test Input |
|-----------|--------------|-----------|
| `2b ≠ a` | `b=1, a=0` | x=0, y=1 → OK |
| `2b==a AND a≤b+10` | `b=1, a=2` | x=2, y=1 → OK |
| `2b==a AND a>b+10` | `b=15, a=30` | x=30, y=15 → **BUG triggered** |

> **Key insight**: Symbolic execution found `x=30, y=15` automatically — no human guessing needed.

---

## 4. Path Constraints as Equivalence Classes

Each path constraint represents an **equivalence class** of inputs — all inputs satisfying the same constraint follow the exact same execution path and produce the same outcome.

**Example**: The constraint `2b==a AND a>b+10` is satisfied by infinitely many inputs:

```
x=30,  y=15   ✓   (2×15=30, 30 > 25)
x=40,  y=20   ✓   (2×20=40, 40 > 30)
x=48,  y=24   ✓
x=100, y=50   ✓
...
```

All of them trigger the same ERROR. We only need **one representative** per class.

> **Consequence**: Symbolic execution achieves the coverage of testing infinitely many inputs by only solving one constraint per path.

## 5. Symbolic State vs Concrete State

### Concrete execution (traditional)
```
Before:  low = 12,  high = 15,  mid = ?
Execute: mid = (high + low) / 2
After:   mid = 13
```

### Symbolic execution
```
Before:  low = L,   high = H,   mid = ?
Execute: mid = (high + low) / 2
After:   mid = (L + H) / 2
```

Values are **expressions over symbolic variables**. Each statement transforms expressions, not numbers.

## 7. Feasible vs Infeasible Paths

Not all paths in the execution tree are reachable. A path is:
- **Feasible**: its path constraint is **satisfiable** (SAT) → a real input exists that executes this path
- **Infeasible**: its path constraint is **unsatisfiable** (UNSAT) → no input can ever reach this path

### Example — Detecting Dead Code

```c
void hello(int x, int y) {
    int t = 0;
    if (x > y)  t = x;
    else        t = y;
    if (t < x)           // ← is this reachable?
        print("Hello");  // ← dead code?
}
```

**Symbolic analysis:**

- If `x > y` (branch 1 true): `t = x` → then `t < x` means `x < x` → **UNSAT** ❌
- If `x ≤ y` (branch 1 false): `t = y` → then `t < x` means `y < x` → contradicts `x ≤ y` → **UNSAT** ❌

Both branches lead to UNSAT for `t < x`.  
→ `print("Hello World")` is **dead code** — it can never be executed.

## 8. SMT Solver

### What is it?
**SMT = Satisfiability Modulo Theories**

An SMT Solver is an automated reasoning engine. Given a logical formula, it returns:
- **SAT** + a satisfying assignment (concrete values) if the formula can be satisfied
- **UNSAT** if no values can satisfy the formula

### SAT vs SMT

| Feature | SAT | SMT |
|---------|-----|-----|
| Input | Boolean formulas only | Boolean + arithmetic + arrays + strings + ... |
| Example | `(A ∨ B) ∧ (¬A ∨ C)` | `2b == a AND a > b + 10` |
| Power | Limited | Handles most real-world code constraints |

### Supported Theories in SMT

| Theory | Handles | Example |
|--------|---------|---------|
| Linear Integer Arithmetic (LIA) | Integer addition, subtraction, comparison | `a + b > 10` |
| Linear Real Arithmetic (LRA) | Same for real numbers | `x * 1.5 < y` |
| Arrays | Read/write array elements | `arr[i] == 5` |
| Strings | Length, concatenation | `len(s) > 3` |
| Bit-vectors | Bitwise operations | `x & 0xFF == 0` |

### Popular SMT Solvers

| Name | Author | Notable Use |
|------|--------|------------|
| **Z3** | Microsoft Research | Most widely used; used in KLEE, angr |
| **Yices** | SRI International | Very fast for arithmetic |
| **STP** | MIT | Specialized in bit-vectors |
| **CVC5** | Stanford/NYU/Iowa | Strong for strings and arrays |

### Simple Examples

**SAT example:**
```
Input:   a > 5  AND  a < 10
Output:  SAT  →  a = 7
```

**UNSAT example:**
```
Input:   a > 10  AND  a < 5
Output:  UNSAT  →  no solution exists
```

**From symbolic execution:**
```
Input:   2b == a  AND  a > b + 10
Step:    substitute a = 2b  →  2b > b + 10  →  b > 10
Output:  SAT  →  b = 15, a = 30   (i.e., x=30, y=15)
```

## 9. Finding Bugs with Symbolic Execution

Symbolic execution can be extended with **dedicated checkers** — one for each bug type.

### How it works
When the engine encounters a potentially dangerous operation, it **forks an extra branch** to check the error condition.

### Example — Divide by Zero

```c
int foo(int i) {
    int j = 2 * i;
    i = i + 1;
    i = i * j;
    if (i < 1) i = -i;
    i = j / i;     // ← potential divide by zero!
    return i;
}
```

At `j / i`, the engine forks:
- Branch A: `i == 0` → **divide by zero detected** → solver finds concrete input
- Branch B: `i != 0` → safe, continue normally

### Other checkable bug types
- **Buffer overflow**: check `index < 0 OR index >= array.length`
- **Integer overflow**: check arithmetic result exceeds type bounds
- **Null pointer dereference**: check pointer `== null` before use
- **Array out-of-bounds**: check `i < 0 OR i > a.length`

## 10. Summary Information — Simplifying Constraints

Path constraints can become **extremely complex** over time. We can replace a complex condition `P` with a **weaker condition** `W` such that `P ⟹ W`.

- `W` describes the path with **less precision** but is **sufficient** for the reasoning task.
- `W` is called a **summary** of `P`.

### Example — Binary Search (`mid = (high + low) / 2`)

**Full precise condition:**
```
low = L  AND  high = H  AND  mid = M  AND  M = (L+H)/2
```

**Weaker summary (sufficient for correctness reasoning):**
```
low = L  AND  high = H  AND  mid = M  AND  L ≤ M ≤ H
```

The weaker condition does not say exactly what `M` is, but it is enough to prove that `mid` always stays within bounds `[low, high]`.

## 11. Loops, Assertions, and Invariants

### The Problem with Loops
A loop can execute 0, 1, 2, 3, ... times → potentially **infinite execution paths**.  
Symbolic execution cannot enumerate them all.

### Solution: Loop Invariants

A **loop invariant** is an assertion (predicate) that must be true **every time execution reaches the loop entry point**, regardless of how many iterations have occurred.

### Pre-conditions and Post-conditions
- **Pre-condition**: a predicate that must hold **before** the function/program executes
- **Post-condition**: a predicate that must hold **after** the function/program terminates

**Key theorem**: If every loop has an invariant, and the program has pre/post-conditions, then every execution path is a finite sequence of segments from one assertion to the next — enabling tractable reasoning.

### Example — Binary Search

```c
// PRECONDITION: array is sorted
// ∀i,j: 0 ≤ i < j < size → dictKeys[i] ≤ dictKeys[j]

while (high >= low) {
    // LOOP INVARIANT: if key exists, it is in [low, high]
    // ∀k: 0 ≤ k < size: dictKeys[k] == key → low ≤ k ≤ high

    mid = (high + low) / 2;
    comparison = strcmp(dictKeys[mid], key);
    if (comparison < 0)      low = mid + 1;
    else if (comparison > 0) high = mid - 1;
    else return dictValues[mid];
}
return 0;
// POSTCONDITION: key not found → return null
```

### Verifying One Loop Iteration (Symbolic)

**State entering the loop:**
```
low = L,  high = H
∀i,j: dictKeys[i] ≤ dictKeys[j]          (precondition)
∀k: dictKeys[k]==key → L ≤ k ≤ H         (invariant holds)
H ≥ L                                     (while condition true)
```

**After `mid = (H+L)/2`:**
```
mid = M,  with L ≤ M ≤ H
```

**After taking branch `low = mid + 1` (when `dictKeys[M] < key`):**
```
low = M+1,  high = H
∀k: dictKeys[k]==key → M+1 ≤ k ≤ H      (invariant still holds ✅)
```

**When loop exits (`H < L`):**
```
∀k: dictKeys[k]==key → L ≤ k ≤ H
But L > H  →  no such k exists
→ key is not in the dictionary  →  return 0 is correct ✅
```

## 12. Pen-and-Paper Exercise — Full Walkthrough

```c
void hello(int x, int y) {
    int t = 0;
    if (x > y)  t = x;
    else        t = y;
    if (t < x)
        print("Hello World");
}
```

**Start:** `x = a`, `y = b`

**First branch — fork:**

| Branch | Condition | Result |
|--------|-----------|--------|
| Left | `a > b` | `t = a` |
| Right | `a ≤ b` | `t = b` |

**Second branch — fork again:**

| Previous | Second condition | Path constraint | Feasible? |
|----------|-----------------|----------------|-----------|
| `t=a` (a>b) | `t < x` → `a < a` | `a > b AND a < a` | **UNSAT** ❌ |
| `t=b` (a≤b) | `t < x` → `b < a` | `a ≤ b AND b < a` | **UNSAT** ❌ |

**Conclusion**: `print("Hello World")` is **dead code** — it can never be reached by any input.

## 13. Summary

| Concept | One-line Definition |
|---------|-------------------|
| **Symbolic Execution** | Run the program with symbolic variables instead of concrete values |
| **Symbolic State** | A mapping from variables to symbolic expressions |
| **Path Constraint** | A logical formula encoding all branch conditions taken so far |
| **Fork** | Split into two branches at every conditional statement |
| **SMT Solver** | Automated solver: given a constraint, finds satisfying values or proves UNSAT |
| **Feasible Path** | Path constraint is SAT → a real input exists for this path |
| **Infeasible Path** | Path constraint is UNSAT → no input can ever reach this path |
| **Equivalence Class** | All inputs satisfying the same path constraint behave identically |
| **Loop Invariant** | Predicate that holds at every loop entry, enabling finite reasoning about loops |
| **Pre/Post-condition** | Assertions before/after a function; together with invariants enable full verification |
| **Summary Information** | Replace a complex constraint P with a weaker W (P ⟹ W) to simplify reasoning |
| **Dedicated Checker** | Extra fork at dangerous operations (division, array access) to detect specific bugs |

### Three Core Takeaways

1. **Symbolic execution generalizes testing**: instead of one concrete input, a symbolic variable represents an entire equivalence class of inputs. One solve per path → covers the whole class.

2. **Path constraint + SMT solver = automatic test generation**: no human guessing required. If the constraint is SAT, the solver produces the test input. If UNSAT, the path is dead code.

3. **Loop invariants are the key to handling loops**: they replace the infinite set of loop paths with a finite, checkable predicate, enabling scalable verification.

## 14. Connection to Data Flow Analysis 

| Aspect | Data Flow Analysis | Symbolic Execution |
|--------|-------------------|-------------------|
| **Goal** | Statically compute facts about variable flow | Automatically generate test cases and find bugs |
| **Path handling** | Summarize over all paths (equations) | Explore each path individually (fork) |
| **Information tracked** | Reaching definitions, taint, liveness | Symbolic state, path constraints |
| **Solving engine** | Worklist algorithm (fixed-point) | SMT/SAT solver |
| **Output** | Program-wide facts | Concrete inputs per path |
| **Loop handling** | Fixed-point iteration | Loop invariants |
| **Taint analysis** | Static tracking of untrusted data | Generate inputs that exploit taint violations |

**Common ground**: Both reason about *how data flows through execution paths*. Data Flow Analysis does this **statically** across all paths at once. Symbolic Execution does this **path-by-path**, precisely but at higher cost.

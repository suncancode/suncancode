# Finite Models

---

## 1. Learning Objectives

- Understand goals and implications of finite state abstraction
- Learn how to model program control flow with **Control Flow Graphs (CFG)**
- Learn how to model system structure with **Call Graphs**
- Learn how to model finite state behavior with **Finite State Machines (FSM)**

---

## 2. Properties of a Good Model

A useful model must be:

| Property | Meaning |
|---|---|
| **Compact** | Representable and manipulable in a reasonably small form |
| **Predictive** | Represents salient characteristics well enough to distinguish good from bad outcomes |
| **Interpretable** | Analysis outcomes can be traced back to causes of failure |
| **Practically General** | Covers a broad range of characteristics for the intended domain |

> No single model represents all characteristics well enough for all kinds of analysis.

## 3. Graph Representations

### Directed Graph
A directed graph consists of:
- **N** = set of nodes
- **E** = set of directed edges (a relation on nodes)

```
Example:
Nodes: {a, b, c}
Edges: {(a,b), (a,c), (c,a)}

    a
   / \
  b   c
   ↖ /
```

### Labeled Nodes
Nodes can be labeled with code regions:
```
[ x = y + z; ] ──► [ a = f(x); ]
```

### Multidimensional Graphs
A single diagram can represent multiple relations over shared nodes.

```
Example (Java class relationships):
- extends relation: B extends A → edge (A, B)
- includes relation: B has field of type C → edge (B, C)

A ──(extends)──► B ──(includes)──► C
```

## 4. Finite Abstraction of Behavior

An **abstraction function** suppresses some details of program execution — it groups together execution states that differ only in suppressed details but are otherwise identical.

> This is how we go from the infinite space of all possible program states to a finite, manageable model.

## 5. Control Flow Graph (CFG)

### Definition
- **Nodes** = regions of source code called **basic blocks**
  - A basic block = maximal program region with a **single entry** and **single exit** point
- **Directed edges** = possibility that execution proceeds from the end of one region to the beginning of another

### Building Blocks

| Construct | CFG Shape |
|---|---|
| `if-else` | Node splits into True/False branches, merges after |
| `while` loop | Condition node loops back from loop body |
| `switch` | Condition node fans out to multiple case nodes |

### Example: `collapseNewlines`

```java
public static String collapseNewlines(String argStr) {
    char last = argStr.charAt(0);           // b1
    StringBuffer argBuf = new StringBuffer();
    for (int cIdx = 0;                      // b2: loop condition
         cIdx < argStr.length();
         cIdx++) {                          // b7: increment
        char ch = argStr.charAt(cIdx);      // b3
        if (ch != '\n' || last != '\n') {   // b4: if condition
            argBuf.append(ch);              // b5
            last = ch;                      // b6
        }
    }
    return argBuf.toString();               // b8
}
```

CFG path:
```
b1 → b2 → [exit loop: b8] or [enter loop: b3 → b4 → b5/b6 → b7 → b2]
```

### Linear Code Sequence and Jump (LCSJ)
LCSJs are **subpaths** of the CFG from one branch point to another. They are used to enumerate paths systematically for testing.

```
From     Sequence              To
Entry  → b1 b2 b3            → jX (exit loop immediately)
Entry  → b1 b2 b3 b4         → jT (if true)
jL     → b3 b4 b5 b6 b7      → jL (loop back)
jX     → b8                  → ret
```

## 6. Call Graphs (Interprocedural CFG)

### Definition
- **Nodes** = procedures / methods / functions
- **Edges** = calls relation between procedures

### Context-Insensitive Call Graph
Arguments and calling context are **not considered**. Multiple calls to the same function are collapsed into one edge.

```
main
├──► C.foo  ──► C.depends
└──► C.bar  ──► C.depends   (same node, context ignored)
```

Problem: cannot distinguish that `C.foo` calls `depends(array[3], 2)` (safe) while `C.bar` calls `depends(array[17], 16)` (out of bounds).

### Context-Sensitive Call Graph
Each call site is treated as a **distinct context**, including argument values.

```
main
├──► C.foo(3)  ──► C.depends(int[3], a, 2)   ✅ safe
└──► C.bar(17) ──► C.depends(int[17], a, 16) ❌ out of bounds
```

### Tradeoff: Exponential Growth
Context-sensitive graphs are more precise but grow **exponentially** with call depth:

```
1 context  → A
2 contexts → AB, AC
4 contexts → ABD, ABE, ACD, ACE
8 contexts → ...
```

### Overestimation in Static Call Graphs
Static analysis includes edges for calls through **dynamic bindings that never actually occur at runtime**.

```java
C myC = C.cFactory("S");  // always returns type S
myC.foo();                // only S.foo() runs, never C.foo()
```

But the static call graph includes an edge to both `C.foo()` and `S.foo()` — this is **pessimistic inaccuracy**.

## 7. Finite State Machines (FSM)

### Definition
An FSM consists of:
- A **finite set of states** (nodes)
- **Transitions** between states (edges), labelled with Event/Response pairs

### Notation
```
  Event
─────────  means: "upon receiving Event, respond with Response"
 Response
```

Example: `LF/emit` means "upon receiving a line feed character, emit the current buffer"

### Example: Line-processing FSM

**Abstract States:**
- `e` — Empty buffer
- `w` — Within line
- `l` — Looking for optional DOS LF
- `d` — Done

**Events:** LF (line feed), CR (carriage return), EOF (end of file), other

**Graph:**
```
(start) ──► [e: Empty buffer]
              │  LF/emit → e
              │  CR/emit → l
              │  EOF/– → d
              │  other/append → w
            [w: Within line]
              │  LF/emit → e
              │  CR/emit → l
              │  EOF/emit → d
              │  other/append → w
            [l: Looking for optional DOS LF]
              │  LF/– → e
              │  EOF/– → d
              │  other/append → w
            [d: Done]
```

### Tabular Representation

| State | LF | CR | EOF | other |
|---|---|---|---|---|
| e | e / emit | l / emit | d / – | w / append |
| w | e / emit | l / emit | d / emit | w / append |
| l | e / – | l / emit | d / – | w / append |

**Reading the table:** `Y / X` means "respond with X, then move to state Y"
- `e / emit` → emit the buffer, go to state e
- `–` → no response, just transition

> An FSM can be generated directly from a tabular representation.

## 8. Using Models to Reason About System Properties

A model is useful when it satisfies three conditions simultaneously:

```
Required Properties
       │
       ▼
  [FSM Model] ──── syntactically well-formed, consistent, complete
       │
       ▼
  [Program]  ──── model accurately represents the program
       │
       ▼
  Model satisfies the specification ✅
```

---

## 9. Model Abstraction: Building an FSM from a Program

The key step is defining the **abstraction function** — mapping concrete program states to abstract FSM states.

### Example: `transduce()` function

Concrete state variables: `lines`, `atCR`, `pos`

| Abstract State | lines | atCR | pos |
|---|---|---|---|
| e (Empty buffer) | 2–12 | 0 | 0 |
| w (Within line) | 12 | 0 | > 0 |
| l (Looking for LF) | 12 | 1 | 0 |
| d (Done) | 35 | – | – |

→ From this mapping, the FSM transition table is derived directly.

---

## 10. Summary

| Model | Built from | Used for |
|---|---|---|
| **CFG** | Source code | Modeling control flow within a procedure |
| **Call Graph** | Source code | Modeling calls between procedures |
| **FSM** | Specification or code | Modeling state-based behavior |

Three key takeaways:
1. Models must be **much simpler** than the artifact they describe to be analyzable
2. Models must still be **sufficiently detailed** to be useful
3. **CFG** is built from existing software; **FSM** can be built **before** software to document intended behavior

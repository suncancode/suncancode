# Dependence and Data Flow Models

---

## 1. Why Data Flow Models?

Previous models (CFG, Call Graph, FSM) focused on **control** — the order in which statements execute. Data flow models focus on **dependence** — where values come from and what they affect.

Key questions data flow answers:
- Where does this value of `x` come from?
- What would be affected if I change this variable?
- Is this user-supplied value being used unsafely?

Data flow information is often combined with control flow to build powerful analysis and test design techniques. Two practical examples:
- **"Taint" analysis** — tracks unvalidated user input to prevent SQL injection
- **Dataflow test criteria** — selects test cases based on def-use pairs

---

## 2. Def-Use Pairs

### Definitions and Uses

| Concept | Meaning | Examples |
|---|---|---|
| **Definition (def)** | A point where a variable gets a value | Declaration, initialization, assignment, parameter received |
| **Use** | A point where a variable's value is extracted | Expression, conditional, parameter passed, return value |

### Example

```java
if (...) {
    x = ...;   // DEFINITION: x gets a value here
}
y = ... + x + ...;  // USE: the value of x is extracted here
```

The path from the definition of `x` to its use forms a **def-use pair**.

---

## 3. Def-Use Pairs in GCD Example

```java
public int gcd(int x, int y) {
    int tmp;              // A: def x, def y, def tmp
    while (y != 0) {     // B: use y
        tmp = x % y;     // C: def tmp; use x, use y
        x = y;           // D: def x; use y
        y = tmp;         // E: def y; use tmp
    }
    return x;            // F: use x
}
```

### Key def-use pairs in this example:

| Definition | Use | Variable | Path |
|---|---|---|---|
| A | B | y | A→B |
| A | C | x | A→B→C |
| C | E | tmp | C→D→E |
| D | C | x | D→E→B→C (loop) |
| E | B | y | E→B |
| D | F | x | D→E→B(false)→F |

---

## 4. Definition-Clear Path and Killing

### Definition-clear path
A path from a **definition** to a **use** of the same variable, with **no other definition** of that variable in between.

### Killing
If another definition of the same variable appears on the path between a definition and its use, the **latter definition kills the former** — the earlier def-use pair is broken.

```
x = ...     // A: def x
q = ...
x = y;      // B: kills x from A, creates new def x
z = ...
y = f(x);   // C: use x
```

- Path A→C: **NOT definition-clear** (killed at B)
- Path B→C: **definition-clear** ✅

### Rule
> A def-use pair (d, u) exists **if and only if** there is a definition-clear path from d to u.

---

## 5. Data Dependence Graph

A **data dependence graph** is:
- **Nodes**: same as CFG nodes (individual statements, not combined)
- **Edges**: def-use pairs, labelled with the variable name

```
Example from GCD:
D (def x) ──x──► F (use x)
E (def y) ──y──► B (use y)
E (def y) ──y──► D (use y)   ← loop-back edge
C (def tmp) ──tmp──► E (use tmp)
```

This graph shows the **information flow** between statements — useful for understanding what depends on what, and for selecting test cases.

## 6. Control Dependence

### Concept
For each statement, **which other statement controls whether it is executed**?

- **Nodes**: same as CFG
- **Edges**: unlabelled, from branching points to the blocks they control

### Example

```
A → B → C → F → G
          ↘ E → F
```

Statement F is **control-dependent on B** because:
- B has at least two successors (C and E)
- B is not post-dominated by F (F doesn't always execute after B)
- B is the **last** decision point that controlled whether F executes

---

## 7. Dominators

### Pre-dominators
Node **M dominates node N** if every path from the root to N passes through M.

- Every node has many dominators
- There is a unique **immediate dominator** of N — the closest one on any path from root

### Post-dominators
Calculated on the **reverse** of the CFG, using a special "exit" node as root.

Node M **post-dominates** N if every path from N to the exit passes through M.

### Example (graph A→B→{C,E}→F→G, B→E→F)

| Node | Pre-dominated by | Post-dominated by |
|---|---|---|
| All nodes | A | G |
| G | B (immediate) | G |
| E and F | G | G, F |

Key rules:
- A pre-dominates all nodes; G post-dominates all nodes
- G is the immediate post-dominator of B (not C)
- B and all its post-dominators form a subtree

---

## 8. Control Dependence — Formal Definition

Node N is **control-dependent** on node C if:
1. C has at least two successors in the CFG (it's a branching point)
2. C is **not post-dominated** by N
3. C is the **last** such branching point before N

> Intuitively: C was the last decision that determined whether N would execute.

## 9. Computing Def-Use Pairs: Reaching Definitions

### The problem with paths
Even without loops, the number of paths in a CFG can be **exponentially large** — too expensive to trace individually.

```
A → B → C → D → E → ... → V
 ↘   ↗  ↘  ↗
   (2 paths to B, 4 to C, 8 to D, 128 to V)
```

### Efficient solution: Reaching Definitions
Instead of tracing each path, we **summarize** which definitions reach each node across all paths.

> Definition **vd** (variable v defined at node d) is a **reaching definition** at node u if there exists a definition-clear path from d to u.

---

## 10. Data Flow Equations

Two data structures per node n:

| Structure | Meaning |
|---|---|
| **Reach(n)** | All definitions that reach the **entry** of n |
| **ReachOut(n)** | All definitions that reach the **exit** of n |

### Equations

```
Reach(n)    = ∪ ReachOut(m)   for all predecessors m of n

ReachOut(n) = (Reach(n) \ kill(n)) ∪ gen(n)
```

Where:
- **gen(n)** = { vn | v is defined at n } — new definitions created at n
- **kill(n)** = { vx | v is defined at any other node x ≠ n } — definitions killed at n

### Example: Node E (`y = tmp`) in GCD

```
Reach(E)    = ReachOut(D)
ReachOut(E) = (Reach(E) \ {yA}) ∪ {yE}
```

→ The old definition of y from A is killed; the new definition yE takes over.

### Example: Node B (`while (y != 0)`) — two predecessors

```
Reach(B)     = ReachOut(A) ∪ ReachOut(E)   ← joins from before loop AND end of loop
ReachOut(A)  = gen(A) = {xA, yA, tmpA}
ReachOut(E)  = (Reach(E) \ {yA}) ∪ {yE}
```

---

## 11. Worklist Algorithm

An iterative algorithm that computes reaching definitions by repeatedly applying flow equations until the solution **stabilizes** (reaches a fixed point).

### General idea

```
1. Initialize: all nodes on worklist, default values
2. While worklist is not empty:
   a. Pick any node n, remove from worklist
   b. Apply data flow equations → compute new Reach(n), ReachOut(n)
   c. If values changed:
      - Add successors to worklist (forward analysis)
      - OR add predecessors (backward analysis)
3. Stop when worklist is empty (no more changes)
```

> The algorithm always terminates because the sets of reaching definitions can only grow (monotone), and are bounded by the finite set of all definitions in the program.

---

## 12. Customizing Data Flow Analysis: Taint Analysis

The Worklist Algorithm generalizes to any property, not just reaching definitions.

| Component | General meaning | Taint analysis example |
|---|---|---|
| **gen(n)** | Facts that become true at n | Value received from untrusted source (web form) |
| **kill(n)** | Facts that stop being true at n | Value was validated/sanitized |
| **Flow equations** | How facts propagate | Taint spreads through assignments |

### Why this matters
If a tainted value flows into a SQL query without being killed (validated), that is a potential SQL injection vulnerability — detectable purely by data flow analysis without running the program.

---

## 13. Summary

| Model | Focus | Built from | Key construct |
|---|---|---|---|
| CFG | Control flow | Source code | Basic blocks + edges |
| Data Dependence Graph | Data flow | CFG + def-use pairs | Def-use edges |
| Control Dependence | Which branch controls what | CFG + post-dominators | Post-dominator tree |

Three key ideas:
1. **Def-use pairs** connect where values are created to where they are consumed
2. **Definition-clear paths** determine whether a def-use pair is valid (not killed)
3. **Reaching definitions** can be computed efficiently with the Worklist Algorithm — no need to enumerate all exponentially many paths

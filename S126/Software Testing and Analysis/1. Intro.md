# Software Testing and Analysis in a Nutshell

## Chapter 1 - Introduction
Based on slides by Mauro Pezze & Michal Young

### 1. Engineering Processes
- Sophisticated tools amplify capabilities but **do not remove human error**.
- All engineering disciplines pairs **construction** activities with **verification** activities.
- Software engineering is no exception: high-quality software requires both **construction** and **verification**.

### 2. Verification and Design Activities
Verification activities take different forms depending on:
1. Engineering discipline
2. Contrucstion process
3. Final product
4. Quality requirements

### 3. How Software is Different?
- Software is soft and intangible
- No physical laws govern its behavior.
- Software never wears out -> traditional reliability measures do not apply.
- Specifications continuously change.

### 4. Peculiarities of Software
- Many different quality requirements.
- Evolving (and sometimes deteriorating) structure.
- Inherent non-linearity.
- Uneven distribution of faults.

### 5. Impact of New Technologies
- Advanced technologies reduce some errors but **do not eliminate** them.
- New approches introdue **new kind of faults**:
  - Deadlock/race conditions in distributed systems.
  - Hallucination of GenAI affecting software validity.

### 6. Variety of Approaches
- No fixed recipes.
- Test designers must choose and schedule the **right blend** of techniques to: 
  - Reach required quality level within cost constraints.
  - Suit the specific problem, requirements, and development environment.

### 7. 5 Basic Questions of Software Testing and Analysis
1. When do verification and validation start and when are they complete?
- Testing is **not** a late phase.
- V&V starts as soon as we dicide to build the product (or even in feasibility study).
- V&V continues **beyond delivery** (maintainance, evolution, new environments).

*Early start (feasibility study):* risk analysis, quality planning, cost/schedule impact.
*Long lasting (maintainance):* change analysis, new test suites, regression testing, fault tracking.

2. What particular techniques should be applied during development?
No single technique serves all purposes. Combine technique because of:
- Effectiveness for different fault classes.
- Applicability at different project stages.
- Different purposes.
- Cost and assurance trade-offs.

**Staging A&T techniques** (across development lifecycle):
- Requiments -> Plan acceptance & system tests.
- Design -> Inspect & analyze architecture/detailed design.
- Coding -> Unit & integration test generation + execution.
- Integration & Delivery -> System & acceptance testing.
- Maintainance -> Regresson testing + process improvement.

3. How can we assess the readiness of a product?
- A&T reveals faults but cannot remove **all** faults.
- We must define the required **level of dependability** and stop when it is reached.

**Dependability measures:**
- Availability (running v down time)
- MTBF (mean time between failures)
- Reliability (fraction of successful operations)

**Practical ways to assess:**
- Random tests following operational profile.
- Alpha testing (controlled envirionment).
- Beta testing (real users, real environment).

4. How can we control the quality of successive releases?
- Quality work never stops at first release.
- Products live for years and change constantly.
- Post-delivery tasks:
  - Test & analyze new/modified code.
  - Re-execute system & regression tests.
  - Extensive record-keeping (bug/issue trackers).

5. How can the development process itself be improved?
Same defects repeat project after project. Long-term goal: fault-driven process improvement.
**Four steps:**
  1. Define data to collect and implement collection procedures.
  2. Analyze data -> identify important fault classes.
  3. Analyze root causes & weaknesses in current techniques.
  4. Adjust development & quality process.
